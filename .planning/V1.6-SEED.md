**Created:** 2026-06-15 (post-v1.5.0 close). Scope locked by Eric. Build in a fresh window.

## The one feature (locked scope)

Extend the Do-It-Here reply loop so the operator can **reply-in-place to resume a STUCK agent** —
not just a human-wait. Today (Phase 14) the reply affordance + `situation.replyAndResume` fire only
on `AWAITING_HUMAN` rows. v1.6 surfaces the same affordance on **`AWAITING_AGENT_STUCK`** rows.

**This is the ONLY thing in v1.6.** Explicitly DROPPED by Eric ("not even needed"):
- Per-user opt-in toggle UI / multi-user.
- Phase-4.2 deferred power features (archive full-view, paused-agent banner, ref-chip peek cards,
  storage-pin, cold-task-from-global).
- `R3-self-assign-one-assignee`.

## Why it's wiring, not invention

- **Handler exists:** `src/worker/handlers/situation-reply-and-resume.ts` (`situation.replyAndResume`,
  Plan 14-01). Header: works "on the dominant AWAITING_HUMAN+status=blocked real path."
- **Resume recipe proven:** the Phase-10 spike (`.planning/phases/10-unblock-resume-spike/`) showed a
  plain comment resumes an agent in BOTH the awaiting-answer AND `status='blocked'` cases — no special
  transition. So a STUCK agent resumes the same way.
- **Engine unchanged:** `src/shared/blocker-chain.ts` already classifies `AWAITING_AGENT_STUCK`
  (kinds: AWAITING_HUMAN / AWAITING_AGENT_STUCK / AWAITING_AGENT_WORKING / HUMAN_ACTION_ON). Stuck rows
  land in the Phase-15 **Watch** tier. blocker-chain.ts stays AI-free and untouched.
