<objective>
Close the two Phase-1 safety-CLI defects that the Plan 02-01 Task 2 smoke spike surfaced (defect #1 — `mode-detect.mjs` schema drift — was fixed inline in commit e26f874 during the spike). Specifically:

- **Defect #2 (pg_dump not on PATH on Windows + bundled embedded-pg is server-only on Windows):** add a pg_dump locator that tries Paperclip's bundled binary first, falls back to system PATH, fails fast with a platform-specific install hint when neither works. Auto-derive `dbUrl` for `embedded-postgres` mode (port + hardcoded creds) so the operator doesn't have to pass `--db-url` for a dev clone.

- **Defect #3 (server/client major-version mismatch — embedded-pg 18.1-beta vs stable pg_dump 17.x):** pre-check version compatibility BEFORE invoking pg_dump and emit a clean, actionable error message naming both versions + the restore-by-deletion fallback for throwaway dev clones. We CANNOT silently fix the version mismatch (pg_dump's strict policy is correct by design) — but we CAN turn a confusing error into a documented decision point.

Plus runbook hygiene: capture every operator gotcha surfaced across Phase 1 + Phase 2 spike into a single append-only `runbook/operator-gotchas.md` so future operators (and future re-spikes) don't have to re-discover them.

- Closing the BEAAA install path. BEAAA is Hostinger Linux + hosted Postgres; Plan 01-04's PASS already proves that path works. This plan is about the EMBEDDED-POSTGRES path that Windows dev work needs.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| safety CLI ← Paperclip-bundled binary | The locator may execute a binary inside Paperclip's `node_modules/@embedded-postgres/...`. This is the same binary the dev-runner spawns; trust is inherited from `pnpm install` integrity (lockfile pinning). |
| safety CLI ← system PATH | The locator may execute whatever `pg_dump` is first on PATH. Operator-controlled; standard CLI-tooling trust posture. |
| pg-dump-locator ← operator override | `--pg-bin <path>` is operator-trusted; the locator does NOT validate the binary is genuinely pg_dump (the operator can shoot themselves in the foot — that's the contract). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05-01 | Spoofing | Bundled `@embedded-postgres/.../pg_dump` could be a malicious binary if the operator's `node_modules` was tampered | accept | Out of scope: lockfile integrity + npm registry trust handles this for all node_modules content, not just pg_dump. Same posture as Plan 01-01. |
| T-01-05-02 | Tampering | `--pg-bin` operator override could point to a wrapper that exfiltrates the dump | accept | Documented contract: `--pg-bin` is operator-trusted. The locator's role is to RESOLVE a path, not validate it. Snapshot output is local-only by default (see Plan 01-01's path-traversal mitigations). |
| T-01-05-03 | Information disclosure | dbUrl with embedded `paperclip:paperclip` creds appears in logs / error messages | mitigate | Redact `:paperclip@` in any logged/printed dbUrl. `assertVersionMatch` does NOT print the full dbUrl on failure — only `{host, port, db}` per existing redaction helpers from Plan 01-01. |
| T-01-05-04 | Denial of service | `psql` round-trip in `assertVersionMatch` hangs (server unreachable, network partition) | mitigate | Wrap the `psql -c "SELECT current_setting('server_version_num')"` call in a 5-second timeout via `AbortSignal.timeout(5000)`. On timeout, throw a clean error naming the dbUrl host/port and suggest verifying the dev server is running. |
</threat_model>
