<threat_model>
Plan 02-09 is presentation-layer + bridge-identity. No new attack surface beyond what the existing opt-in-guard threat model covers.

The one new concern: get-viewer is EXEMPT from opt-in-guard. The exempt list is sensitive тАФ anything on it can be called by opted-out users. get-viewer is safe to expose because:
- It returns the caller's OWN identity, which the host has already authenticated
- The host bridge auto-identifies the caller; we're just surfacing what the host already knows
- It does not read or write plugin-namespace data, only echoes session identity
- The opted-out caller can't do anything with the returned userId other than thread it back into other handlers, which the guard correctly denies (since the caller is opted-out)

Add a test that asserts get-viewer for an opted-out caller still returns their userId, AND that a subsequent issue.reader call with that userId is correctly denied.
</threat_model>
