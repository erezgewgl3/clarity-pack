<objective>
Close the LIVE output-channel gap that the 2026-05-16 Countermoves Plan 03-06 re-drill surfaced. Plan 03-06's scoped-issue task-delivery architecture is PROVEN — the Editor-Agent ran scoped to the operation issue (`PAPERCLIP_TASK_ID` confirmed), read the compile prompt, and produced a flawless `BulletinDraft`. **That architecture is NOT re-opened by this plan.** The remaining defect is purely how the result bytes travel back: the agent filed the `BulletinDraft` JSON as a Paperclip issue *document* and posted prose as the comment; `deliverAgentTask`'s readback polls `listComments` for a JSON comment, finds only prose, times out, and nothing publishes.

This plan adopts **Option C** from `03-RESULT-READBACK-RESEARCH.md` — the canonical `plugin-llm-wiki` pattern: the Editor-Agent delivers its result by **calling a declared plugin tool** (`submit-compile-result`), and the tool handler — running synchronously inside the worker — receives the structured payload directly and resolves the in-flight `deliverAgentTask` promise. No comment-poll race, no JSON-from-prose extraction on the designed path. **Option A (comment scan) + Option B (document scan) are retained as a slow belt-and-suspenders fallback poll** so the readback degrades gracefully if the agent ever ignores the tool.

Folded in per Eric's decision: the **DURABLE-BREAKER-STALE-HISTORY** operability bug. `isCircuitOpenDurable` currently counts ALL `editor_agent_failures` rows including pre-fix noise, so a fresh install is silently DOA until the rows are hand-cleared (the re-drill had to `DELETE` 518+482 stale rows). The fix: an additive `plugin_version` column on the failures table; `recordFailure` stamps the current version; `isCircuitOpenDurable` counts only current-version rows.

**Authoritative source:** `.planning/phases/03-daily-bulletin/03-RESULT-READBACK-RESEARCH.md` — "Concrete Build Delta (for the Plan 03-07 planner)", items 1-5, and "Open Questions for the Planner".

Scope — exactly five deltas, nothing else:
1. Manifest — `agent.tools.register` capability + a one-entry `tools[]` declaring `submit-compile-result` + the Editor-Agent `agents[].permissions` plugin-tool allow block (copied verbatim from `plugin-llm-wiki`).
2. Tool handler — `compile-result-tool.ts`: `ctx.tools.register('submit-compile-result', …)` resolving the in-flight delivery via a shared `Map`.
3. `deliverAgentTask` readback — register a pending promise before `requestWakeup`, `Promise.race` it against the timeout, keep a slow comment+document fallback poll.
4. Editor-Agent instructions — rewrite the delivery clause: deliver by calling `submit-compile-result`; do NOT comment, do NOT file a document.
5. DURABLE-BREAKER-STALE-HISTORY — version-scoped `editor_agent_failures` count via an additive migration.

NO change to `computeFactsTable` / `computeStandingNumbers`, `verifyDraft`, `publishBulletin`, `validateDraftSchema`, schema validation, or the operation-issue CREATION path (`deliverAgentTask` steps 1-3). The verified-numerics contract is preserved — `verifyDraft` (pass-2) still re-runs every SQL on the returned draft.

Purpose: this is the last gap before Phase 3 closes. After this plan and a passing re-drill, Phase 3 closes.
Output: a result-readback channel that is a typed tool boundary instead of a polling race, plus a circuit breaker that no longer false-trips a fresh install on stale failure history. Net delta per research: ~120-180 LOC.
</objective>

| agent document/comment → fallback poll | If the agent files a document or comment instead of calling the tool, the slow fallback poll reads it; same untrusted-text boundary. |
| failure-row `plugin_version` → breaker | The version stamp scopes which rows the durable breaker counts. |
