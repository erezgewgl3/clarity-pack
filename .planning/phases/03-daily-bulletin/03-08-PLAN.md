<threat_model>
The worker now reads an agent-authored issue document and parses JSON out of it. Surface and the existing/added guards:

1. **Malformed or non-JSON document body.** The agent stores JSON-in-a-markdown document (`DocumentFormat` is typed `"markdown"`, research Q1). `isResultDocument` MUST run the body through `extractJsonObject` → `JSON.parse` → `validateDraftSchema` inside try/catch; a body that is not a schema-valid `BulletinDraft` (prose, a progress note, an empty rev) is NOT a result — the poll continues. Never resolve on a bare `{` or a non-validating object (the W-4 rule already in `isResultComment`).
2. **Oversized document body.** A runaway document body is bounded the same way the comment path is: `tldr-compile` results are gated `length <= 8000`; a `bulletin-compile` document that is not parseable JSON within the existing `extractJsonObject` brace-scan is simply rejected. No unbounded read is introduced — `documents.get` returns one keyed document, not a stream.
3. **Wrong document — the agent files a document that is not the BulletinDraft.** The readback keys on the EXACT document key `compile-result` first (`documents.get(issueId, 'compile-result', companyId)`); only if that misses does it fall back to scanning `documents.list`. A document at a different key that happens to schema-validate would still be accepted by the list scan — acceptable because `validateDraftSchema` + downstream `verifyDraft` are the real gate.
4. **The readback channel does NOT bypass verification.** Whatever bytes arrive — document, comment, or (inert) tool — `verifyDraft` pass-2 in `compile-bulletin.ts` re-executes every standing-number SQL query against `ctx.db.query` and rejects the draft on any numeric mismatch BEFORE `publishBulletin`. This plan does not touch `verifyDraft`; the publish gate is unchanged. A malicious or hallucinated document cannot publish unverified numbers.
5. **Document-poll on the wrong operation issue.** `deliverAgentTask` polls the SAME `operationIssueId` it created/reused in steps 1-3 (the variable `operationIssueId = issue.id`). The 03-07 fallback-poll-miss candidate cause "keys to the wrong operation issue" is closed structurally because the poll and the create share one id. Task 1's diagnosis step confirms no second id is introduced.
</threat_model>
