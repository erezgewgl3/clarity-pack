**SQL-injection surface — invariant T-03-10 (must be preserved by this plan):**
- Every `sql` field is a **static module-level constant string** — no template literals, no string concatenation, no interpolation.
- The ONLY bound parameter is `$1` (companyId), supplied via the `params: ['<companyId>']` placeholder that `computeStandingNumbers` and `verifyDraft` substitute at call time.
- The standing-numbers source-grep test asserts `/\$\{[^}]*\}/` never matches any slot SQL — this assertion MUST continue to pass after the rewrite. The new SQL strings in Task 1 are plain quoted constants; verify no `${` survives.

**Why the new SQL is safe:** all 5 queries are `SELECT ... WHERE company_id = $1` (or `WHERE id = $1`) — `company_id` is the host-scoped value, and `$1` is a bound param, never concatenated. No user-supplied free text reaches the query. The fix is *correct columns*, not laxer verification — `verifyDraft`'s strict reject-on-`query_failed` behaviour is correct per BULL-05/06 and is NOT weakened.

**Out-of-scope guard:** this plan adds NO database migration (worker SQL strings only — no DDL), touches NO UI, does NOT modify `facts-table.ts` (verified pure — no SQL), and does NOT reopen the Plan 03-08 Option B document-handoff or the Plan 03-09 `validateDraftStructure` readback (both proven live).
