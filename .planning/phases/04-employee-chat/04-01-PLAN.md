The probe is a Node ESM script that, given `PAPERCLIP_API_URL` + a bearer token + a company id, performs the following sequence against live Countermoves using the same `paperclip-api` REST client pattern as `scripts/safety/` (do NOT use the plugin SDK ctx — this runs outside the worker; mirror the REST calls scripts/safety/paperclip-api uses, or invoke via the MCP server if simpler):

1. PROBE-D01/OQ-4 (agent wake + reply form): create a parent issue titled `Chat — <employee> [SPIKE]`, then a child issue titled `Spike topic [SPIKE]` with `parentId` = the parent, `assigneeAgentId` = a real employee-agent id resolved from the agent list (NOT the Editor-Agent), `description` containing the literal instruction "Reply to comments on this issue by posting a COMMENT on this issue (not a document).", `originKind: 'plugin:clarity-pack'`, `originId: 'chat-spike'`. Post a comment on the child issue with body "What is 2 + 2? Reply with a comment." Then poll `listComments` on the child issue every 30s for up to 8 minutes. Record: did a new comment appear from the agent? Did the agent ALSO file a document (check the documents list)? Capture the exact author field set on the reply comment row (`author_agent_id` vs `author_user_id`).

2. PROBE-OQ2 (event payload): the probe cannot subscribe to `ctx.events` from outside the worker — instead, after the reply lands, fetch the new comment via `listComments` and record its full JSON shape (id, body, created_at, author fields). Document in findings that the worker stream bridge in 04-03 must derive the comment id from `event.entityId` (issue id) + a `listComments` re-fetch IF the live event payload turns out opaque; note this as a 04-03 design input.

3. PROBE-OQ3 (auto-reopen re-wake): set the child issue status to `done`. Post another comment "Re-wake test — reply with a comment." Wait 60s. If no agent reply within 4 minutes, flip the issue status to `in_progress` and DO NOT call requestWakeup; poll 4 more minutes. If still no reply, call `requestWakeup` and poll 4 more minutes. Record which step produced the wake: (a) status flip alone, (b) status flip + requestWakeup, or (c) neither.

4. PROBE-OQ1 (attachment path): attempt, in order, and record the exact error/success for each: (a) `ctx.issues.documents.upsert`-equivalent REST call carrying a small base64 text body keyed e.g. `chat-attachment-test`; (b) any host multipart upload route discoverable in the REST surface; (c) confirm there is no `assets` write route. Record whether ANY path stored a retrievable file.

Make every step wrapped in try/catch that records the outcome rather than crashing the whole probe — partial findings are still findings. Print a structured JSON summary at the end.
