<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| UI → chat.send / chat.edit (bridge) | An opted-out or non-owning user could call these via the same-origin bridge. |
| host → issue.comment.created event | Untrusted: the event entityId could be any issue, not a chat topic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Elevation of Privilege | opted-out user calls chat.send/chat.edit via the bridge | mitigate | Both handlers register via `wrapActionHandler` (opt-in-guard) — the wrapper returns OPT_IN_REQUIRED before the inner handler runs (OPTIN-04, server-side enforcement; UI gating alone is insufficient under same-origin trust). |
| T-04-09 | Tampering | a user edits another user's message | mitigate | chat-edit re-verifies the prior message's sender_kind + ownership server-side before appending — returns NOT_OWNED otherwise (the bulletin-action-approve NOT_OWNED pattern, ASVS V4). |
| T-04-10 | Spoofing / Tampering | duplicate / replayed sends double-post comments | mitigate | message_uuid dedup via getChatMessageByUuid + chat_messages PRIMARY KEY (message_uuid) ON CONFLICT DO NOTHING — a replay returns the original comment_id (CHAT-06 / D-10). |
| T-04-11 | Information Disclosure | stream bridge relays comments on non-chat issues | mitigate | The bridge emits only when getChatTopicByIssueId returns a row; the channel is scoped `chat:<companyId>` so a tab only ever subscribes to its own company's channel. |
| T-04-12 | Denial of Service | a throwing event handler crashes the worker | mitigate | The stream-bridge event body is wrapped in try/catch + ctx.logger?.warn?. (the worker.ts event-handler discipline). |
</threat_model>
