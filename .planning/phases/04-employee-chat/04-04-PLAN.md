<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| UI → chat data/action handlers (bridge) | An opted-out user, or a user querying another company, crosses here. |
| user search term → ctx.db.query | Untrusted free text; ILIKE wildcards and SQL injection are the risks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-13 | Tampering | SQL / LIKE injection in chat.search | mitigate | The ILIKE term is a `$N` bound parameter (parameterized ctx.db.query — SELECT-only on core tables); `escapeLike` additionally escapes `%`, `_`, `\` so a user term cannot become a wildcard. Static module-constant SQL — mirrors the T-03-10 invariant. |
| T-04-14 | Information Disclosure | cross-company data leak in search / roster | mitigate | chat.search JOINs through chat_topics with `t.company_id = $1`; chat.roster passes companyId to ctx.agents.list; companyId is the resolved host-context value, never a free param trusted blindly. |
| T-04-15 | Elevation of Privilege | opted-out user calls a chat handler | mitigate | All six handlers register via wrapDataHandler / wrapActionHandler — none added to EXEMPT_HANDLER_KEYS (OPTIN-04). |
| T-04-16 | Tampering | promote/pin on someone else's message | mitigate | chat.promote re-verifies the caller can see the source message server-side before creating the issue; chat.pin operates on the caller-scoped chat_messages row. |
| T-04-17 | Information Disclosure | search returns comments from non-chat issues | mitigate | The CHAT-08 query JOINs ONLY through chat_topics — non-chat issue comments are structurally excluded from the result set. |
</threat_model>
