## PROBE-OQ2-FILTER — D-08 active-tasks query path

**VERDICT: WEAK (REST-LIMIT).**

### Question

D-08 — does `ctx.issues.list({companyId, originKind, originId})` filter
precisely enough to support the active-tasks-per-topic query by an
`originId` prefix (`chat-task:<topicIssueId>:...`), or do we need a
`0007_chat_topic_tasks.sql` side table?

### Evidence

The probe created two probe true-task issues, distinct `originId`s under
`originKind: 'plugin:clarity-pack:chat-task'`, then ran three filter calls
against the REST `issues.list` surface:

| Filter | HTTP status | rows returned | both probe tasks in result? | precision |
|--------|-------------|---------------|-----------------------------|-----------|
| `originKind` only | 200 | 500 (cap) | **no** (neither task A nor B in the 500-row result) | useless — 500-cap, probe tasks not even present |
| `originKind` + `originId` (exact match) | 200 | 0 | **no** | filter ignored — exact-match returns 0 even though the row exists |
| `originKind` + `originIdPrefix` | 200 | 500 (cap) | **no** | filter ignored — prefix returns the unfiltered 500-cap |

`originKind`, `originId`, and `originIdPrefix` are all silently ignored at the
REST layer. The 500-row cap hits before the filter would prune. Even when a
genuinely-matching row exists, exact-match returns zero rows — the filter is
not just imprecise, it is broken at the REST surface.

The probe was unable to test SDK-level `ctx.issues.list({originKindPrefix})`
because the probe runs OUTSIDE the worker (no `ctx`). That branch remains
unobserved.

### Locked finding

**REST surface does NOT honor origin-based filters.** The Plan 04.1-05 executor
SHOULD ship the side-table-as-defensive-default — the cost is low and the REST
finding makes any non-side-table path strictly speculative on this host.

**Recommended path: ship `migrations/0007_chat_topic_tasks.sql`** (additive
plugin-namespace side table: `chat_topic_tasks(topic_issue_id, task_issue_id,
created_at)`). The active-tasks query (D-08) joins this side table to the
chat-topic id for O(1) lookup; no scan of `issues.list` involved.

**Defensive client-side filter** stays as a belt-and-braces fallback: if the
side table is missing (e.g. the plugin is upgraded mid-flight), the handler
can still fall back to `issues.list` and client-side-filter by
`originId.startsWith('chat-task:<topicIssueId>:')`. But the steady-state path
is the side table.

### Design input for downstream plans

- **Plan 04.1-05** — ships `migrations/0007_chat_topic_tasks.sql` (additive, plugin-namespace, two-column side table).
- **Plan 04.1-05** — `createTrueTask` (Plan 04.1-02) MUST insert into `chat_topic_tasks` immediately after `ctx.issues.create`. Race-safe (ON CONFLICT DO NOTHING) per the Phase 4 D-05 chat_employee_parents pattern.
- **Plan 04.1-05** — `chat.taskOwned` data handler queries via the side table (`SELECT task_issue_id FROM chat_topic_tasks WHERE topic_issue_id = $1`) and then `ctx.issues.get` per row for status/title. NO `issues.list({originKind})` scan in the steady-state path.
- **Plan 04.1-05** — defensive fallback: if the side table is empty for a given topic AND the chat-topic issue is older than the migration timestamp, the handler MAY fall back to `issues.list` + client-side `originId.startsWith(...)` filter. Document this as a one-time backfill path; the steady state remains the side table.
- **Plan 04.1-02** — `createTrueTask` originId convention is `chat-task:<topicIssueId>:<commentId>` per RESEARCH.md. Confirmed by the probe (both task A and B used this shape).

## FLAG-1 reconciliation — can a chat topic enter recovery while held non-terminal?

**VERDICT: MOOT (native wake works).**

### Evidence

The OQ3 attempt-2 evidence captured a real disposition-recovery notice on
the OQ3 probe issue (see PROBE-D14-DISCRIM raw JSON above). This shows the
host's recovery service CAN engage post-run on a chat-topic issue even when
status starts at `in_progress` — the recovery service flipped the issue
toward terminal disposition machinery from a successful run that ended
without a clear next-step disposition.

However, because **native multi-turn wake works** (PROBE-OQ3 PASS-NATIVE),
this is NOT a host-blocked branch. The watchdog (Plan 04.1-04) catches the
flip on its next poll and flips status back to `in_progress`. The agent then
re-wakes natively on the next operator comment. The bounded recovery loop is
contained inside D-11's defensive sweep.

### Locked finding

**Reconciliation choice: NEITHER `host-blocked-residual` NOR `prevented-by-D09-D11` — instead `caught-by-watchdog`.** The recovery service does engage, but D-11's defensive sweep catches it before the operator sees the topic as stranded. CTT-05 ships as the routine backstop, NOT as exceptional path.

**CTT-06 "⚠ TOPIC STUCK HOST-SIDE" banner ships as the LAST-resort defensive
fallback** — used only if the D-11 watchdog fails twice in a row to flip the
topic back to non-terminal (e.g. the disposition-recovery service races the
watchdog and re-applies its flip immediately). On this probe run no such race
was observed; CTT-06 may end up rarely-rendered in practice but its presence
is non-negotiable per the "never silent" clause.

### Design input for downstream plans

- **Plan 04.1-04** — `topicStuck` field returns TRUE only if the watchdog has attempted N≥2 flips in the last M poll cycles without success. Not on the first watchdog flip — that's expected steady state.
- **Plan 04.1-06** — CTT-06 banner rendered when `topicStuck === true`. Operator instruction text per UI-SPEC.
