<objective>
Wave 2 — the worker-only core of the operator-side true-task path AND the upgrade of the existing agent-message Promote onto the same mechanism. ONE shared `createTrueTask()` helper, TWO entry-points (operator composer + agent-message Promote), one consistent result: a top-level, assigned, findable Paperclip issue with a marker-comment back-link on the chat topic.

Purpose: D-04's load-bearing rule is "one consistent mechanism." Today's `chat-promote.ts` parents the new issue under the topic (the exact bug the Phase 4 followup identifies — orphan task nested under plugin plumbing) AND leaves it unassigned (`status: 'todo'`, no `assigneeAgentId` — nobody acts on it; the operator observed only the Editor-Agent react, not the chatted employee). This plan deletes that behaviour and replaces it with the helper, so the bug is impossible by construction.
