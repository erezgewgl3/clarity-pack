## Decisions Made

- **userId is gated by opt-in-guard, not reqStr.** The plan invariant says missing userId THROWS, but the opt-in-guard wrapper fires first (`extractUserId` returns null → `isOptedIn` returns false → `{ error: 'OPT_IN_REQUIRED' }`). The handler's defensive `reqStr(params, 'userId')` never runs on this path. The test suite was split: Test 3 iterates 6 required strings that DO throw; Test 3b pins the OPT_IN_REQUIRED return for missing userId. This matches existing chat.send/chat.promote behaviour and is documented inline.
- **Anti-regression comments reworded to satisfy the grep contract.** The plan's `<acceptance_criteria>` includes `grep -c "parentId" <file> == 0`. The originally-written comment "// D-05 — TOP-LEVEL: NO parentId" contained the literal property name and would have failed the grep even though it carried no semantic risk. The comment was reworded to "not nested under the topic" + "we deliberately do NOT pass a parent-pointer field on the create payload" — same warning to future readers, zero `parentId` string occurrences.
- **Marker-comment failure = warn + continue.** The helper does NOT re-throw on `ctx.issues.createComment` failure. The handler returns `{ ok: true, issueId }` even when the marker did not land. The originId is the authoritative back-link; the marker is in-thread UX. This was explicit in the plan invariants but is the load-bearing design call worth surfacing — it trades chat-thread completeness for operator-trust in the create result.
