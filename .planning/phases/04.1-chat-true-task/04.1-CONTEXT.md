<domain>
## Phase Boundary

From the **Employee Chat** surface (shipped in Phase 4), the operator can turn an intention into a **TRUE task** — a real Paperclip issue that is **assigned** to an employee-agent, **tracked** with a status lifecycle, and **findable** in the Issues list — and a chat topic can host sustained multi-turn back-and-forth without the agent-task lifecycle corrupting it (no runtime bookkeeping leaking into the thread, no stranded topic issue, the agent reliably re-wakes on *every* operator message, not just the first).

**Origin:** Inserted 2026-05-19 from Phase 4 live drilling on Countermoves. Full problem statement, evidence, and open design questions: `.planning/phases/04-employee-chat/04-FOLLOWUP-chat-true-task.md`.

**In scope:**
- A first-class operator-initiated "create a true task" action from the chat surface.
- The conversation/runtime-noise separation in `chat.messages`.
- A chat-topic lifecycle that does not strand the topic issue and re-wakes the agent reliably on every message.
- Upgrading the existing agent-message `Promote to task` (D-13) onto the same true-task mechanism.

**Out of scope (Phase 4's delivered Employee Chat stands):** the conversation surface, topics, send/edit, pin, polling, the four-region shell. This phase is purely the true-task capability + the conversation/lifecycle separation. Group threads (`CHAT-G-01`) remain v2.

</domain>

<decisions>
## Implementation Decisions

### True-task trigger UX

- **D-01: The operator triggers a true task via a composer mode toggle ("Send as task").** A toggle in the chat composer flips the next message from "chat message" to "create task" — the operator types the task and hits Send-as-task. The message body *is* the task. Keeps creation in the natural typing flow; no separate form.
- **D-02: Creation goes through a lightweight confirm dialog.** Before the task commits, the operator sees a pre-filled **title** + **assignee** and can tweak them. Not a full task form, not one-click-fire — just enough to prevent a malformed one-line task or a wrong assignee.
- **D-03: After creation, the chat thread shows an inline task card.** A distinct in-thread card ("Task created — BEAAA-NNN, assigned to {employee}") with a link, visually unlike a chat bubble. Keeps the conversation aware of what was spun off and stays greppable.
- **D-04: The existing agent-message `↗ Promote to task` (D-13) is upgraded onto the same true-task path.** Agent-message Promote produces the **same** assigned, findable, top-level task as the operator-side path — one consistent mechanism. Today's Promote (`chat-promote.ts`: unassigned `todo`, parented under the topic issue) is replaced, not kept alongside.

### Task structure & assignment

- **D-05: The true task is a standalone top-level issue with an originId back-link.** Created with **no chat parent** (`parentId` is not the chat-topic issue and not the per-employee `Chat —` parent), so it appears in the normal Paperclip Issues list like any other task — satisfying Success Criterion #1. It carries `originId`/origin metadata back-referencing the chat topic + source comment for provenance. *(This overrides today's `chat-promote.ts` behavior, which parents the created issue under the topic issue and is therefore filtered out of the Issues view — the exact bug the followup identifies.)*
- **D-06: The true task is assigned to the chatted employee-agent by default, reassignable in the confirm dialog.** The confirm dialog (D-02) pre-fills the employee-agent of the current chat topic — immediate ownership + wake — but the operator may pick a different employee-agent before committing. A true task is **never** left unassigned (unlike today's Promote).
- **D-07: The true task links back to its chat origin via origin metadata AND a marker comment on the topic issue.** The task issue carries an `originId` back-reference (provenance/grep), and a "task created" marker comment is posted on the chat-topic issue so the inline task card (D-03) has a durable host-side artifact that survives plugin disable. **This marker comment is a first-class plugin message** — it must be classified as conversational content, not filtered out as runtime noise (see D-12).
- **D-08: The chat surface tracks the spun-off task's status in the context rail.** The new task appears in the right context rail's "active tasks owned" list (the section already present in the Phase 4 chat sketch) with a live status pill, so the operator watches progress without leaving chat.

### Chat-topic lifecycle model

- **D-09: The chat-topic issue is a non-completable conversation container.** The chat-topic *host* issue is held in a non-terminal status and is **never marked `done`** — so Paperclip's post-completion disposition / recovery-owner machinery never engages on it. All directed work spins off as separate true-task issues (D-05). Rationale: the chat-topic issue is conversation infrastructure (closer to an email thread than a work item); org-level "what's open / what's done / what's achieved" is answered by the spun-off true-task issues, which complete with a fully normal lifecycle. Non-completable applies only to the conversation container, not to work.
- **D-10: "Archive a topic" is a plugin-side concept, decoupled from host issue status.** The operator gets an "Archive topic" affordance that sets `chat_topics.archived` (the column already exists) and drops the topic from the chat UI — **without** marking the host issue `done` (which would re-trigger the disposition machinery). This gives the operator conversation-tidying control while keeping D-09 intact. *(This supersedes Phase 4 D-06's "classic close → `done` + auto-reopen on send" model for chat-topic issues.)*
- **D-11: Non-completability is enforced by topic-description instruction + a worker watchdog.** The topic issue description instructs the assigned agent to converse only and never complete the topic issue; **and** a worker watchdog flips the topic issue back to a non-terminal status if it is ever observed in a `done`/disposition-blocked state. Instruction alone relies on agent compliance; the watchdog is the deterministic plugin-side backstop (no host change required).
- **D-12: Multi-turn re-wake uses an explicit per-message wake nudge.** On every operator send, the worker explicitly ensures the topic issue is in a wakeable state (non-terminal + `resume` flag) as part of posting the comment — deterministic re-wake rather than relying on the issue happening to be in the right state. **This must be falsified-first on Countermoves** (post a follow-up message, observe the agent re-wake) before UI work depends on it — the observed bug is "agent replies once then goes silent on follow-ups," and the exact cause is not yet proven.
- **D-13: Residual host-side stranding surfaces a notice rather than auto-recovering.** If a topic issue still ends up stranded "blocked on a recovery owner" after D-09/D-11 (i.e. a genuinely host-side disposition state the plugin cannot cleanly reverse), the chat surface shows the operator a clear "this topic is stuck host-side" notice rather than the worker silently mutating host state. Consistent with the project's "never silent" posture. **See FLAG-1** — this may require reconciling Success Criterion #4.

### Runtime-noise filtering

- **D-14: Runtime/system comments are filtered out of the chat thread; strategy is host-field-first with a body-pattern fallback.** Research first determines whether Paperclip runtime comments (`disposition`, `recovery owner`, `finish_successful_run_handoff`, etc.) carry a distinguishing field (`originKind`, a system flag, a non-agent author). **If a discriminator exists →** filter on it (robust). **If not →** fall back to a body-pattern blocklist of the known runtime phrases. This is a falsify-first spike: the followup explicitly flags that runtime noise may be posted under the *agent's own* `author_agent_id`, which would defeat a naive whitelist-by-participant approach. *(`chat-messages.ts` today returns every comment on the topic issue with no filter.)*
- **D-15: The runtime-noise filter runs worker-side in the `chat.messages` handler.** The UI never receives runtime noise; the filter logic is unit-testable in the `node --test` suite, consistent with the bridge-only model.
- **D-16: Filtered-out runtime comments are hidden by default but remain diagnosable.** Runtime noise is hidden from the conversation, but a debug/diagnostics path (a toggle, or simply the classic Paperclip issue view) can still reveal it so a stuck topic stays diagnosable. Comments are **never destroyed** — they remain ordinary host `issue_comments` rows (coexistence intact).

### Claude's Discretion

The planner / executor may choose pragmatic values within these constraints:
- Exact composer toggle affordance and label (D-01) — must match the Phase 4 chat sketch's composer tools row idiom.
- Confirm-dialog layout/styling (D-02) — scoped under `[data-clarity-surface="chat"]`.
- Inline task-card visual treatment (D-03) — distinct from `eric-bubble`/`agent-bubble`; may reuse the `.decision-msg` centered-typeform idiom from the sketch.
- The non-terminal status value the topic issue is held in (D-09/D-11) — pick whatever Paperclip status is non-terminal and reads least like "active work"; verify available statuses during research.
- Watchdog cadence / trigger (D-11) — on-send check, on-poll check, or a job; planner picks.
- Marker-comment wording (D-07) and its classification rule so D-14's filter never strips it.
- `chat_topics`/`chat_messages` schema additions for the task back-reference and archive — additive-only migration `migrations/0007_*.sql`, never touching prior files or `public.*`.

<specifics>
## Specific Ideas

- **The chat-topic issue is conversation infrastructure, not a work unit** — this reframing (an email thread, not a task) is the load-bearing idea behind D-09. Org-level work status is answered entirely by the spun-off standalone true-task issues.
- **One consistent true-task mechanism** — operator-side (D-01) and agent-message Promote (D-04) must produce identical results; do not ship two divergent task-creation paths.
- **FLAG-1 — Success Criterion #4 reconciliation:** ROADMAP SC#4 says a chat topic issue is "never stranded in a stuck task-completion state." D-09 + D-11 are what *prevent* stranding; D-13 handles the residual case by surfacing a notice rather than auto-recovering. If research finds the residual host-side disposition state is genuinely unrecoverable plugin-only, SC#4 should be reconciled as "host-blocked" (CHAT-04 precedent) — the verifier must not treat the surface-a-notice path as an SC#4 failure.
- **FLAG-2 — Situation Room cross-effect:** non-completable chat-topic issues assigned to employee-agents (D-09) could inflate an agent's "active tasks owned" count or "now-doing" narration in the Situation Room (Phase 2). The planner should check whether the Situation Room needs to exclude `originKind: plugin:clarity-pack` chat-topic issues from work-status aggregation.
- **Falsify-first is mandatory here** — D-12 (re-wake) and D-14 (discriminator) are the two genuine unknowns. The first plan should include a Countermoves spike for both before UI/handler work depends on the outcome.

</specifics>
