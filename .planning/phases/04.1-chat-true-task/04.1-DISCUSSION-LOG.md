## Chat-topic lifecycle model

### Lifecycle model for the chat-topic issue
| Option | Description | Selected |
|--------|-------------|----------|
| Non-completable + plugin archive | Host issue never done; plugin-side archive flag tidies the UI | ✓ (after clarification) |
| Non-completable, no archive | Never terminal; no tidy-up affordance | |
| Completes on explicit archive (host done) | Archive marks host issue done; accepts disposition-trigger risk | |
| Keep D-06 close + auto-reopen | Today's model (produced the observed stranding) | |

**Notes:** The operator did not initially pick an option — instead asked how a non-completable container would affect the org-level view of "what's open / what's actually closed / what's been achieved." Claude answered: the chat-topic issue is conversation infrastructure (an email thread, not a work item); org-level work status is answered by the spun-off standalone true-task issues, which complete with a fully normal lifecycle; chat-topic containers are a bounded, plugin-origin-tagged, tree-grouped set. Claude proposed adding a plugin-side archive (decoupled from host issue status) to keep the conversation list tidy. On the re-ask, the operator chose "Non-completable + plugin archive."
