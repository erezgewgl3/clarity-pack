## User Constraints (from CONTEXT.md)

### Locked Decisions (D-01..D-16)

**True-task trigger UX:**
- **D-01:** Operator triggers a true task via a composer mode toggle ("Send as task"). A toggle in the chat composer flips the next message from "chat message" to "create task". The message body *is* the task.
- **D-02:** Creation goes through a lightweight confirm dialog — pre-filled **title** + **assignee**, operator can tweak. Not a full task form, not one-click-fire.
- **D-03:** After creation, the chat thread shows an inline task card ("Task created — BEAAA-NNN, assigned to {employee}") with a link, visually unlike a chat bubble.
- **D-04:** The existing agent-message `↗ Promote to task` (Phase 4 D-13) is upgraded onto the SAME true-task path. One consistent mechanism — today's `chat-promote.ts` behavior is replaced, not kept alongside.

**Task structure & assignment:**
- **D-05:** The true task is a standalone TOP-LEVEL issue (no chat `parentId` — not the topic issue, not the per-employee `Chat —` parent) with an `originId` back-link to the chat topic + source comment. Appears in the normal Issues list.
- **D-06:** The true task is assigned to the chatted employee-agent by default, reassignable in the confirm dialog. Never left unassigned.
- **D-07:** Links back via origin metadata AND a marker comment posted on the topic issue. **The marker comment is a first-class plugin message — must be classified as conversational content, NOT filtered out as runtime noise.**
- **D-08:** The chat surface tracks the spun-off task's status in the right context rail's "active tasks owned" list with a live status pill.

**Chat-topic lifecycle model:**
- **D-09:** The chat-topic issue is a NON-COMPLETABLE conversation container — held non-terminal, never marked `done`. All directed work spins off as separate true-task issues.
- **D-10:** "Archive a topic" is a plugin-side concept — sets `chat_topics.archived` (column already exists), drops topic from the chat UI, WITHOUT marking the host issue `done`. Supersedes Phase 4 D-06's "classic close → `done` + auto-reopen" model.
- **D-11:** Non-completability enforced by topic-description instruction + a worker watchdog that flips the topic issue back to a non-terminal status if observed `done`/disposition-blocked.
- **D-12:** Multi-turn re-wake uses an explicit per-message wake nudge — on every operator send the worker ensures the topic issue is wakeable. **Must be falsified-first on Countermoves before UI work depends on it.**

**Runtime-noise filtering:**
- **D-13:** Residual host-side stranding surfaces a NOTICE rather than auto-recovering. Consistent with "never silent". **See FLAG-1.**
- **D-14:** Runtime/system comments filtered out of the chat thread; strategy host-field-first with body-pattern fallback. **Falsify-first spike.**
- **D-15:** The runtime-noise filter runs worker-side in the `chat.messages` handler. Unit-testable.
- **D-16:** Filtered-out runtime comments hidden by default but remain diagnosable (debug toggle or classic Paperclip view). Comments are NEVER destroyed.

## Summary

This phase is small in surface area but turns on two host-mechanics unknowns. The good news from this research: **both unknowns resolve in the plugin's favor with HIGH confidence, and the SDK exposes every API the locked decisions require.** No host change is needed.

The **D-14 runtime-comment discriminator EXISTS and is robust.** `IssueComment` carries a real `authorType: "user" | "agent" | "system"` discriminator AND a `presentation` field whose `kind` is `"message" | "system_notice"`. Paperclip's recovery service authors disposition / recovery-owner / `finish_successful_run_handoff` notices with `authorType: "system"` (verified in `server/src/services/recovery/service.ts` — `addComment(id, body, {}, { authorType: "system" })`). The followup's worst-case fear — runtime noise posted under the *agent's own* `author_agent_id` — is NOT how Paperclip works. D-14 ships as a clean host-field filter; the body-pattern blocklist is a defense-in-depth fallback, not the primary path.

The **D-12 multi-turn re-wake gap has a known root cause.** Paperclip's heartbeat runs a `finish_successful_run_handoff` after an agent run "succeeds". If the agent marks the topic issue `done` (terminal), `TERMINAL_ISSUE_STATUSES = {done, cancelled}` permanently skips it from comment-wake. Even if the issue stays non-terminal, after a *successful run with no disposition* the host enters the missing-disposition recovery loop and the issue can be escalated to `blocked`. **D-09 (never let the topic go terminal) + D-11 (watchdog flips it back off `done`/`blocked`) is the correct architecture.** D-12's explicit per-message nudge is `ctx.issues.requestWakeup(issueId, companyId, {reason, contextSource})` — a real SDK method requiring the `issues.wakeup` capability the manifest already declares. This still must be falsified live on Countermoves.

**Primary recommendation:** Build a shared `createTrueTask` worker helper (top-level issue, `assigneeAgentId` set, `originId` back-link, `originKind: "plugin:clarity-pack"`) consumed by both the operator path (D-01) and the upgraded Promote (D-04). Add a `classifyComment` pure function (filters `authorType === "system"` and `presentation.kind === "system_notice"`) used by `chat.messages`. Replace the topic issue's never-`done` enforcement with a watchdog on the existing `chat.messages` poll path. Add `ctx.issues.requestWakeup` to `chat.send`. Make Plan 1 a Countermoves falsify-first spike for D-12 + D-14 before any UI/handler build.

### Pattern 1: Shared true-task creation helper (D-04 / D-05 / D-06)
**What:** A single `createTrueTask()` function consumed by both the operator composer path and the upgraded agent-message Promote, so there are never two divergent task-creation paths.
**When to use:** Any "turn this into a real task" action.
**Example:**
```typescript
// Source: synthesized from PluginIssuesClient.create signature
// (node_modules/@paperclipai/plugin-sdk/dist/types.d.ts:1023-1048)
// and verified-good chat-topics.ts ctx.issues.create usage.
export async function createTrueTask(ctx: TrueTaskCtx, input: {
  companyId: string;
  title: string;
  description: string;
  assigneeAgentId: string;        // D-06 — NEVER omitted; never an unassigned task
  topicIssueId: string;           // for the D-07 back-link + marker comment
  sourceCommentId: string | null; // chat message the task came from
}): Promise<{ issueId: string }> {
  // D-05 — TOP-LEVEL issue: NO parentId. This is the bug fix vs. today's
  // chat-promote.ts which passes parentId: topicIssueId and gets filtered out
  // of the Issues list.
  const issue = await ctx.issues.create({
    companyId: input.companyId,
    // parentId: OMITTED — top-level, appears in the normal Issues list.
    title: input.title,
    description: input.description,
    status: 'todo',                       // a real work item with a real lifecycle
    assigneeAgentId: input.assigneeAgentId, // D-06 — assignment is the wake contract
    originKind: 'plugin:clarity-pack',      // provenance tag (PLUGIN_SPEC §14.1)
    originId: `chat-task:${input.topicIssueId}:${input.sourceCommentId ?? 'composer'}`,
  });
  return { issueId: issue.id };
}
```
Note: `originKind` is typed `PluginIssueOriginKind` — keep it `"plugin:clarity-pack"` (the value Phase 4 already uses), NOT the `"plugin:clarity-pack:operation:*"` namespace, which the Editor-Agent heartbeat self-loop filter deliberately skips (`src/manifest.ts:188`).

### Pattern 2: Host-field comment classification (D-14 / D-15)
**What:** A pure function that decides whether a comment is conversational or runtime noise, run worker-side in `chat.messages`.
**When to use:** Every `chat.messages` thread assembly.
**Example:**
```typescript
// Source: IssueComment shape from packages/shared/src/types/issue.ts +
// packages/shared/src/constants.ts (ISSUE_COMMENT_AUTHOR_TYPES,
// ISSUE_COMMENT_PRESENTATION_KINDS) — verified 2026-05-20 against master.
type CommentClass = 'conversation' | 'runtime-noise';

export function classifyComment(c: {
  authorType?: string | null;
  presentation?: { kind?: string | null } | null;
  body?: string;
}): CommentClass {
  // PRIMARY (HIGH confidence) — host discriminator.
  if (c.authorType === 'system') return 'runtime-noise';
  if (c.presentation?.kind === 'system_notice') return 'runtime-noise';
  // FALLBACK (defense-in-depth) — body-pattern blocklist for any host build
  // that posts a notice without the presentation field. Kept narrow.
  const RUNTIME_PHRASES = [
    'needs a disposition', 'blocked on a recovery owner',
    'finish_successful_run_handoff', 'exhausted the bounded corrective handoff',
  ];
  const body = (c.body ?? '').toLowerCase();
  if (RUNTIME_PHRASES.some((p) => body.includes(p.toLowerCase()))) return 'runtime-noise';
  return 'conversation';
}
```
**The D-07 marker comment** is posted by the plugin via `ctx.issues.createComment` — it comes back with `authorType` NOT `"system"` (the plugin posts as a worker/agent author) and no `system_notice` presentation, so `classifyComment` returns `'conversation'` for it automatically. No special-casing of the marker is needed; D-14's filter never strips it. The planner should still cite this explicitly so a future body-pattern addition never accidentally matches the marker wording.

### Pattern 3: Topic-wakeable guarantee (D-09 / D-11 / D-12)
**What:** A worker helper that, on every operator send AND on the watchdog path, ensures the topic issue is in a wakeable, non-terminal status and explicitly nudges the agent.
**When to use:** `chat.send` (per-message, D-12) and a watchdog trigger (D-11).
**Example:**
```typescript
// Source: PluginIssuesClient.get / .update / .requestWakeup
// (types.d.ts:1022, 1049-1053, 1065-1069). requestWakeup needs the
// `issues.wakeup` capability — already declared (manifest.ts:297).
const NON_TERMINAL_CONVERSATION_STATUS = 'in_progress'; // see Open Question 1

export async function ensureTopicWakeable(ctx: TopicCtx, topicIssueId: string, companyId: string) {
  const issue = await ctx.issues.get(topicIssueId, companyId);
  if (!issue) return;
  // D-09/D-11 — a chat-topic issue must NEVER be terminal and must not be
  // parked in the blocked recovery state. Flip it back if observed wrong.
  if (issue.status === 'done' || issue.status === 'cancelled' || issue.status === 'blocked') {
    await ctx.issues.update(topicIssueId, { status: NON_TERMINAL_CONVERSATION_STATUS }, companyId);
  }
  // D-12 — explicit per-message nudge. A bare comment SHOULD wake the assignee
  // (Phase 4 D-01), but the followup proves re-wake is unreliable after a run
  // completes. requestWakeup is the deterministic backstop.
  await ctx.issues.requestWakeup(topicIssueId, companyId, {
    reason: 'issue_commented',
    contextSource: 'clarity-pack:chat',
  });
}
```
**FALSIFY-FIRST:** the exact `reason` / `contextSource` strings that the host's `isVerifiedIssueTreeControlInteractionWake()` gate accepts must be confirmed live (see Common Pitfalls — wake-gate rejection). `ISSUE_TREE_CONTROL_INTERACTION_WAKE_REASONS` are `"issue_commented" | "issue_reopened_via_comment" | "issue_comment_mentioned"`.

### Anti-Patterns to Avoid
- **Parenting the true task under the topic issue.** This is the exact Phase 4 bug. `chat-promote.ts` today passes `parentId: topicIssueId`; the task is then nested under plugin plumbing and filtered out of the Issues list. D-05 forbids it — pass NO `parentId`.
- **Leaving the true task unassigned (`status: 'todo'`, no `assigneeAgentId`).** Today's Promote does this; the followup observed only the *Editor-Agent* react (its TL;DR-compile job), never the chatted employee. D-06: always set `assigneeAgentId`.
- **Letting the topic issue reach `done`.** Terminal → permanently un-wakeable; recovery machinery can also engage. D-09 forbids it.
- **A whitelist-by-participant comment filter.** Tempting but wrong: it would also strip the D-07 marker comment and any cross-posted system message that *is* relevant. Use the `authorType`/`presentation` discriminator (D-14).
- **A continuous worker loop for the watchdog.** Violates governance parity. Run the watchdog on the existing `chat.messages` poll path or a `jobs.schedule` cron — not `setInterval`.
- **Adding an `issues.update` capability string.** Not a real `PLUGIN_CAPABILITIES` member; risks the install validator. `ctx.issues.update` already works under the declared `issues` family.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Detecting runtime/system comments | A bespoke regex parser over comment bodies as the *primary* path | `IssueComment.authorType === 'system'` + `presentation.kind === 'system_notice'` | The host already classifies. Body patterns are brittle across host versions; use them only as fallback. |
| Waking the agent on a new message | A custom heartbeat / `setInterval` poke | `ctx.issues.requestWakeup(issueId, companyId, {...})` | Real SDK method; goes through host budget/lock/heartbeat policy. Custom loops break governance parity. |
| Reading the true-task status for D-08 | A new plugin table mirroring host status | `ctx.issues.get(taskIssueId, companyId)` → `.status` (or `ctx.issues.list({originKind, originId})`) | Host is the source of truth for issue status; a mirror table drifts. |
| Finding all tasks spun off a topic | Walking the issue tree | `ctx.issues.list({ companyId, originKind: 'plugin:clarity-pack', originId: 'chat-task:<topicIssueId>:*' })` — or store the task id list in `chat_topics`/a side table | `originId` carries the back-link by design (D-05/D-07). |
| Reversing a host disposition/recovery state | A worker that mutates host recovery rows | Surface a notice to the operator (D-13) | Recovery state is host-internal and **cannot be reversed by a plugin** (verified in `recovery/service.ts`). Only a status flip to `done` or manual reassignment exits it. |

**Key insight:** Almost everything this phase needs is already a typed SDK method. The temptation is to hand-roll filtering and waking because the *symptoms* (silent agent, leaked noise) look like plugin bugs — but they are host-lifecycle interactions, and the host exposes the exact primitives (`authorType`, `requestWakeup`, status `update`) to handle them correctly.

## Common Pitfalls

### Pitfall 1: Agent goes silent after its first reply (the core observed bug)
**What goes wrong:** The CMO agent replied once on CHT-1116, then never re-woke on follow-ups.
**Why it happens:** When an agent run "succeeds", Paperclip's heartbeat runs `decideSuccessfulRunHandoff()` / a `finish_successful_run_handoff`. Two failure modes follow: (a) if the agent marked the topic issue `done`, it is now terminal — `TERMINAL_ISSUE_STATUSES = {done, cancelled}` permanently skips it from comment-wake; (b) even non-terminal, after a *successful run with no valid disposition* the host can enter the bounded missing-disposition handoff and, when exhausted, `escalateStrandedAssignedIssue()` moves the issue to `blocked`.
**How to avoid:** D-09 (topic issue never goes `done`) + D-11 watchdog (flip it off `done`/`blocked` back to a non-terminal status) + D-12 explicit `requestWakeup` on every send.
**Warning signs:** `chat.messages` thread shows a long gap after the first agent reply; a topic issue's status reads `done` or `blocked` in classic Paperclip; runtime notices appear in the thread.

### Pitfall 2: `requestWakeup` silently no-ops due to the interaction-wake gate
**What goes wrong:** You call `ctx.issues.requestWakeup` but the agent still doesn't run.
**Why it happens:** `isVerifiedIssueTreeControlInteractionWake()` only honors wake reasons in `ISSUE_TREE_CONTROL_INTERACTION_WAKE_REASONS` (`issue_commented`, `issue_reopened_via_comment`, `issue_comment_mentioned`) AND requires the `contextSource` to match the reason's expected source AND the comment author to match a verified wakeup record. A made-up `reason` is rejected.
**How to avoid:** Use a recognized reason string. **Falsify-first** on Countermoves — post a follow-up, call `requestWakeup`, confirm a run is queued (`PluginIssueWakeupResult.queued === true`, `runId` non-null).
**Warning signs:** `requestWakeup` returns `{ queued: false, runId: null }`.

### Pitfall 3: The 04-01 spike concluded "STATUS-FLIP-NOT-NEEDED" — do not over-trust it
**What goes wrong:** Phase 4 D-06 / OQ-3 found a bare comment wakes the assignee and `chat-send.ts` therefore does NOT call `requestWakeup`. The followup proves that conclusion holds only for the *first* message.
**Why it happens:** The spike tested a single comment on a fresh agent-assigned issue. It did not test re-wake after a run completes.
**How to avoid:** Phase 4.1's D-12 explicitly adds the nudge `chat-send.ts` omitted. The planner must not treat the 04-01 finding as a reason to skip D-12.
**Warning signs:** N/A — this is a planning-time trap, not a runtime one.

### Pitfall 4: Stripping the D-07 marker comment as noise
**What goes wrong:** The "Task created — BEAAA-NNN" marker comment disappears from the thread.
**Why it happens:** An over-broad body-pattern blocklist, or a participant-whitelist filter, catches it.
**How to avoid:** Rely on the `authorType` discriminator (the marker is plugin-authored, not `system`). Keep the body-pattern fallback narrow and unit-test that the exact marker wording classifies as `conversation`.
**Warning signs:** Inline task card has no durable host-side artifact after a thread reload.

### Pitfall 5: D-08 task-status read causes an N+1 fan-out
**What goes wrong:** The context rail reads each spun-off task's status with one `ctx.issues.get` per task on every poll.
**Why it happens:** Naive per-card fetch.
**How to avoid:** Use a single `ctx.issues.list({ originKind: 'plugin:clarity-pack', originKindPrefix/originId filter })` call, or batch in one worker handler. Honor the existing `usePoll` visibility-pause + low cadence.
**Warning signs:** Chat poll thundering-herd; host RPC volume spikes when many tasks are tracked.

### Pitfall 6: Watchdog fights the agent (status thrash)
**What goes wrong:** The watchdog flips the topic off `done`, the agent re-marks it `done`, ping-pong.
**Why it happens:** D-11's instruction ("converse only, never complete") relies on agent compliance; the watchdog is the backstop, but if the agent ignores the instruction the two can race.
**How to avoid:** The topic-description instruction must be strong and unambiguous (it propagates — Phase 4 D-14 confirmed issue *description* reaches the agent). Watchdog flips should be idempotent and logged; if thrash is observed, escalate to D-13's operator notice rather than looping. Consider holding the topic in `in_progress` (not `todo`) so the agent reads it as "ongoing conversation" not "fresh task to complete".
**Warning signs:** Repeated status-flip log lines on one topic issue within minutes.

## Open Questions

1. **Which non-terminal status holds the topic issue (D-09 Claude's discretion)?**
   - What we know: `ISSUE_STATUSES = backlog | todo | in_progress | in_review | done | blocked | cancelled`. Terminal = `{done, cancelled}`. `blocked` is non-terminal but reads as stuck and is where recovery escalation lands. `backlog`/`todo` read as "not started". `in_review` implies awaiting a human gate.
   - What's unclear: which non-terminal status (a) keeps the agent wakeable, (b) reads least like "active work" in classic Paperclip / Situation Room, (c) does not itself invite the disposition machinery.
   - Recommendation: hold the topic in **`in_progress`** — it is unambiguously non-terminal, keeps the agent wakeable, and is the status `chat-send.ts` auto-reopen already flips to. Avoid `blocked` (recovery target) and `todo` (reads "unstarted"). The D-12 spike should confirm an `in_progress` topic with a fresh comment wakes the agent. **Verify on Countermoves which status the agent itself tends to set after a successful run** — that is the value the watchdog must override.

2. **Does this phase need a `0007_*.sql` migration at all?**
   - What we know: `chat_topics` already has `archived` (D-10 needs no new column). `chat_messages` maps `message_uuid → comment_id`. The true-task back-link can live entirely in the host issue's `originId` (D-05) — no plugin table required.
   - What's unclear: whether D-08's "active tasks owned" list is better served by an `originId` query (`ctx.issues.list`) or a small `chat_topic_tasks(topic_issue_id, task_issue_id, created_at)` side table for O(1) lookup + ordering.
   - Recommendation: prefer **no migration** — query by `originId`. If the planner finds the `originId` filter shape (`originId` exact vs prefix) too weak for "all tasks for this topic", add a minimal additive `0007_chat_topic_tasks.sql`. Decide during the Plan 1 spike by testing `ctx.issues.list({ originKind, originId })` filter behavior live.

3. **Exact `requestWakeup` `reason`/`contextSource` strings the host gate accepts (D-12).**
   - What we know: `ISSUE_TREE_CONTROL_INTERACTION_WAKE_REASONS = {issue_commented, issue_reopened_via_comment, issue_comment_mentioned}`; the gate also checks `contextSource` and author.
   - What's unclear: whether a plugin-originated `requestWakeup` (actorType `plugin`) satisfies the author-match check, and which `contextSource` value pairs with `issue_commented`.
   - Recommendation: **Plan 1 falsify-first spike** — call `requestWakeup` with `reason: 'issue_commented'` and observe `{queued, runId}`. If rejected, try `issue_reopened_via_comment` paired with a status flip. Fall back path: if `requestWakeup` cannot be made to fire, D-09 + D-11 alone (topic always non-terminal) may be sufficient because a bare comment on a non-terminal assigned issue natively wakes (Phase 4 D-01) — the spike must distinguish "non-terminal is enough" from "explicit nudge required".

4. **FLAG-1 — is the D-13 residual stranding genuinely host-blocked?**
   - What we know: `recovery/service.ts` confirms recovery state **cannot be reversed by a plugin** — only a status update to `done` or manual reassignment exits it. But D-09 *prevents* the topic ever reaching the state that triggers recovery (recovery needs a `succeeded` run that exhausts handoff attempts on a non-terminal issue). If D-09/D-11 hold, the topic never enters recovery.
   - What's unclear: whether an agent can complete a *run* "successfully" on the topic issue without the issue going `done` — in which case the missing-disposition loop could still engage even though the watchdog keeps status non-terminal.
   - Recommendation: **The planner should reconcile Success Criterion #4 as host-blocked-residual (CHAT-04 precedent).** D-09 + D-11 are the prevention; D-13's operator notice is the honest fallback for the genuinely unrecoverable residual. The verifier must NOT treat "surface a notice" as an SC#4 failure. The Plan 1 spike should attempt to provoke the recovery state to size how often the residual actually occurs.

5. **FLAG-2 — do non-completable topic issues inflate Situation Room agent work counts?**
   - What we know: `src/worker/jobs/situation-snapshot.ts` builds `now_doing` from a per-employee `current_task_summary` field and walks blockers via `ctx.issues.relations.get` — it does **not** appear to count assigned non-terminal issues as "active tasks". Risk is therefore bounded.
   - What's unclear: whether any Situation Room surface (a card's task list, a velocity metric) does an `issues.list` by `assigneeAgentId` that would now include every chat-topic issue.
   - Recommendation: the planner should grep the Situation Room handlers for an `issues.list({ assigneeAgentId })` call. If one exists, exclude `originKind === 'plugin:clarity-pack'` chat-topic issues from work-status aggregation. `ctx.issues.list` supports an `originKind` filter and an `includePluginOperations` flag — both usable to scope the query. Low-risk, but a one-line check.
