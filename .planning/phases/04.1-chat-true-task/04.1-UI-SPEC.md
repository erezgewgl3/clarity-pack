# Phase 4.1 вҖ” UI Design Contract

> Visual and interaction contract for the **new affordances** Phase 4.1 adds to the existing Phase 4 Employee Chat surface. Generated by gsd-ui-researcher in `--auto` mode against CONTEXT.md D-01..D-16 (locked), RESEARCH.md (host APIs verified), and the sketch-findings-clarity-pack skill (LOCKED visual ground truth).
>
> **What this contract covers:** the seven new visual affordances scoped in Phase 4.1 (composer mode toggle, confirm dialog, inline task card, active-tasks-owned live status in the context rail, archive-topic affordance, diagnostics toggle, host-stuck notice). The existing chat surface idiom вҖ” three-column shell, roster, topic strip, bubbles, day dividers, reasoning panel, composer base, attachment chip, `.decision-msg` centered typeform, `.auto-refresh` live indicator вҖ” is already locked by Phase 4 and `sketches/paperclip-fix-employee-chat.html`. **Re-stating it would invite drift.** Where the contract below references existing classes / tokens / patterns, they are cited verbatim and reused, never redefined.

> **Source of all visual values:** `sketches/paperclip-fix-employee-chat.html` (ll. 11-33 palette, ll. 100-307 thread region, ll. 47-99 roster, ll. 307-340 composer, ll. 342-392 context rail). All new affordances scope every CSS rule under `[data-clarity-surface="chat"]` per SCAF-06 / T-04-20.

## New Component Patterns

### Pattern A вҖ” Composer mode toggle (D-01) `<TrueTaskToggle>`

**Where it sits:** Inside the existing `.composer-tools` row at `composer.tsx:180`, between `рҹ“Һ Attach` and the existing `вҶ— New task` button (which Phase 4.1 supersedes вҖ” the existing `вҶ— New task` button at `composer.tsx`-equivalent / sketch l. 679 is currently a stub; this phase REPLACES that stub with the live toggle, NOT adding a parallel control).

**Visual states:**

| State | Border | Background | Text | Pre-icon |
|-------|--------|------------|------|----------|
| OFF (default) | `1px solid transparent` | transparent | `--ink-3` | `вҶ—` glyph, `--ink-3` |
| HOVER (off) | `1px solid var(--line)` | transparent | `--ink-2` | `вҶ—` glyph, `--ink-2` |
| ARMED | `1px solid var(--you)` | `var(--you-soft)` | `--you` | `вҶ—` glyph, `--you`, with a 6pxГ—6px solid `--you` dot to the left of the glyph |
| DISABLED (no employee assigned to topic вҖ” defense-in-depth) | `1px solid var(--line)` | transparent | `--ink-4` | `вҶ—` glyph, `--ink-4` |

**Interaction contract:**
- Click toggles armed/off.
- When ARMED, the composer placeholder updates (see copywriting) and the Send button label changes to `Open task form`.
- `aria-pressed="true"` when armed.
- Keyboard: focusable with Tab; Enter / Space toggles. Activating Send (Enter in the textarea, or clicking Send) when ARMED opens the confirm dialog instead of calling `chat.send`.
- After a successful task create, the toggle disarms automatically. After a Cancel from the dialog, the toggle STAYS armed (the operator was just adjusting вҖ” they shouldn't lose their typed body).

**Why a toggle, not a slash-command or split-button:** D-01 locks the affordance as a toggle. Slash-commands are out of v1 chat scope (Phase 4 D-14 / Claude's discretion). A split-button would imply two equally-weighted send paths; the toggle correctly models "the next send is a different verb."

### Pattern B вҖ” Confirm dialog (D-02) `<TrueTaskDialog>`

Click on backdrop does NOT close (small modal, deliberate friction вҖ” the only ways out are Create or Cancel).

### Pattern C вҖ” Inline task card (D-03) `<InlineTaskCard>`

**Where it sits:** Inside `.messages` at the point the marker comment (D-07) lands, ordered by server timestamp like any other message. **NOT a `.bubble`** вҖ” distinct visual idiom so a sighted operator never confuses a spun-off task with a sent message.

**Closest existing idiom:** the `.decision-msg` centered typeform (sketch ll. 294-306). The inline task card extends that idiom with task-specific affordances. It is **NOT styled as `eric-bubble` or `agent-bubble`**.

## Accessibility

- The composer toggle uses `<button type="button" aria-pressed="вҖҰ">` with the ARMED state surfacing `aria-label="Sending as task вҖ” turn off to send a normal message"`.
- The confirm dialog uses the native `<dialog>` element which provides focus-trap and ARIA semantics for free. `aria-labelledby` points at the dialog heading; `aria-describedby` points at the helper text.
- The inline task card is rendered inside `<article class="msg">` with `role="group"` and `aria-label="Task created вҖ” {title}, assigned to {employee}, status {status}"`.
- The host-stuck banner uses `role="alert"` so it announces on render (the operator must know the conversation has paused).
- The diagnostics toggle uses `aria-pressed`.
- All status pills carry `aria-label="Status: {label}"` so a screen reader announces the status word, not just the color class.
- Focus visibility on every new button: `outline: 1px solid var(--ink-2); outline-offset: 2px` on `:focus-visible` (matches the rest of the chat surface).

**Operator creates a true task from chat:**

```
1. Operator clicks "Send as task" toggle in composer вҶ’ toggle visually arms (gold border+dot, text "Sending as task").

   - Title field pre-filled with first ~80 chars of body.
   - Assignee pre-filled with CFO.
   - FROM MESSAGE shows full body in a quote block.
5. Operator tweaks title (optional) and clicks "Create task".
   - Create button: "CreatingвҖҰ"
6. chat.createTrueTask succeeds вҶ’ dialog closes.
   - Toggle disarms.
   - Composer clears.
   - Optimistic <InlineTaskCard> appears in thread with status="CreatingвҖҰ".

8. As CFO works the task, the status pill updates on each poll (Todo вҶ’ In progress вҶ’ In review вҶ’ Done).
```

