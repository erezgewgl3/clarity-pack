# Phase 4.1 — UI Design Contract

> Visual and interaction contract for the **new affordances** Phase 4.1 adds to the existing Phase 4 Employee Chat surface. Generated by gsd-ui-researcher in `--auto` mode against CONTEXT.md D-01..D-16 (locked), RESEARCH.md (host APIs verified), and the sketch-findings-clarity-pack skill (LOCKED visual ground truth).
>
> **What this contract covers:** the seven new visual affordances scoped in Phase 4.1 (composer mode toggle, confirm dialog, inline task card, active-tasks-owned live status in the context rail, archive-topic affordance, diagnostics toggle, host-stuck notice). The existing chat surface idiom — three-column shell, roster, topic strip, bubbles, day dividers, reasoning panel, composer base, attachment chip, `.decision-msg` centered typeform, `.auto-refresh` live indicator — is already locked by Phase 4 and `sketches/paperclip-fix-employee-chat.html`. **Re-stating it would invite drift.** Where the contract below references existing classes / tokens / patterns, they are cited verbatim and reused, never redefined.
>
> **Source of all visual values:** `sketches/paperclip-fix-employee-chat.html` (ll. 11-33 palette, ll. 100-307 thread region, ll. 47-99 roster, ll. 307-340 composer, ll. 342-392 context rail). All new affordances scope every CSS rule under `[data-clarity-surface="chat"]` per SCAF-06 / T-04-20.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (CSS-only — Clarity Pack does NOT ship a Tailwind, inherits host CSS per STACK pin) |

## Spacing Scale

Multiples of 4 only. Phase 4.1 introduces no new spacing tokens; every new-component padding picks from this table.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap inside toggle, task-card bullet inset |
| sm | 8px | Tool button gap, composer foot gap |
| md | 12px | Inline-task-card vertical padding; tight component padding |
| lg | 16px | Confirm-dialog vertical padding (top + bottom); inline-task-card horizontal padding; general card padding |
| xl | 20px | Confirm-dialog horizontal padding |
| 2xl | 24px | Day-divider top margin; section-to-section vertical rhythm |

**Exceptions — inherited-verbatim from the existing Phase 4 chat surface CSS (NOT available to new Phase 4.1 components):** The existing `src/ui/styles/chat.css` uses occasional non-multiple-of-4 values inside tightly-packed components shipped in Phase 4 (badges, dots, status pills, bubble interiors, composer chrome, rail rhythm). These include the micro-values **5px / 6px / 7px / 9px / 10px / 11px / 13px / 15px**, plus the larger inherited values **14px** (the `.bubble` padding is `13px 16px`, and the composer-box padding top is `14px`) and **18px** (the `.task-row` vertical rhythm in the context rail), and **22px** (the thread + composer horizontal gutter at sketch l. 105 / l. 309). These values are preserved verbatim where they already exist in Phase 4 CSS — this phase neither re-targets them nor extends them to new components. Every new Phase 4.1 component (Patterns A–G below) uses only the multiple-of-4 tokens in the main table above.

Inherited from the existing chat surface idiom (sketch ll. 25-27, ll. 56-58 Geist, ll. 124, 144, 167, 188, 313 Geist Mono uppercase tracked, ll. 56, 115, 263, 296, 357 Instrument Serif italic). This phase introduces NO new type roles.

## New Component Patterns

### Pattern A — Composer mode toggle (D-01) `<TrueTaskToggle>`

**Where it sits:** Inside the existing `.composer-tools` row at `composer.tsx:180`, between `📎 Attach` and the existing `↗ New task` button (which Phase 4.1 supersedes — the existing `↗ New task` button at `composer.tsx`-equivalent / sketch l. 679 is currently a stub; this phase REPLACES that stub with the live toggle, NOT adding a parallel control).

**Visual states:**

| State | Border | Background | Text | Pre-icon |
|-------|--------|------------|------|----------|
| OFF (default) | `1px solid transparent` | transparent | `--ink-3` | `↗` glyph, `--ink-3` |
| HOVER (off) | `1px solid var(--line)` | transparent | `--ink-2` | `↗` glyph, `--ink-2` |
| ARMED | `1px solid var(--you)` | `var(--you-soft)` | `--you` | `↗` glyph, `--you`, with a 6px×6px solid `--you` dot to the left of the glyph |
| DISABLED (no employee assigned to topic — defense-in-depth) | `1px solid var(--line)` | transparent | `--ink-4` | `↗` glyph, `--ink-4` |

**Interaction contract:**
- Click toggles armed/off.
- When ARMED, the composer placeholder updates (see copywriting) and the Send button label changes to `Open task form`.
- `aria-pressed="true"` when armed.
- Keyboard: focusable with Tab; Enter / Space toggles. Activating Send (Enter in the textarea, or clicking Send) when ARMED opens the confirm dialog instead of calling `chat.send`.
- After a successful task create, the toggle disarms automatically. After a Keep-editing from the dialog, the toggle STAYS armed (the operator was just adjusting — they shouldn't lose their typed body).

**Why a toggle, not a slash-command or split-button:** D-01 locks the affordance as a toggle. Slash-commands are out of v1 chat scope (Phase 4 D-14 / Claude's discretion). A split-button would imply two equally-weighted send paths; the toggle correctly models "the next send is a different verb."

### Pattern B — Confirm dialog (D-02) `<TrueTaskDialog>`

**Construction:** Native `<dialog>` element (React 19 supports `<dialog>` props; opens via the `dialog.showModal()` imperative API which we wrap in a `useEffect`). Backdrop is the browser default `::backdrop` styled to `rgba(14, 13, 10, 0.78)` (the `--bg` token at 78% alpha). Focus is auto-trapped by the `<dialog>` API. Escape closes (= Keep editing). Click on backdrop does NOT close (small modal, deliberate friction — the only ways out are Create or Keep editing).

**Layout:** A single column, ~480px wide centered on viewport, `--bg-2` surface, `1px solid var(--line-bright)` border, `padding: 16px 20px 16px` (vertical = lg token 16px, horizontal = xl token 20px — both multiples of 4).

```
┌──────────────────────────────────────────────┐
│ FROM CHAT · Capital sensitivity · CHT-31     │ ← Meta-label, --ink-3
│                                               │
│ Create task for CFO                           │ ← Heading, Geist 18px weight 400
│                                               │
│ ┌─────────────────────────────────────────┐  │
│ │ TITLE                                    │  │ ← Meta-label
│ │ [ Lock sec.4 at 12% in v1.5            ] │  │ ← Geist input, --bg, 1px --line
│ │                                          │  │
│ │ ASSIGNEE                                 │  │
│ │ [ CFO · Capital · Economics      ▼ ]    │  │ ← native <select>, same idiom
│ │ The task will appear in the Issues list, │  │ ← helper text Geist Mono 10px
│ │ assigned to this employee.               │  │
│ │                                          │  │
│ │ FROM MESSAGE                             │  │
│ │ ┃ Lock the broker comm % at 12% per…   │  │ ← quote, --ink-2, border-left 2px --line
│ │ ┃ Bedrock-Re negotiated rate.           │  │
│ └─────────────────────────────────────────┘  │
│                                               │
│         Keep editing       [ Create task ]   │ ← .btn.ghost + .btn (existing)
└──────────────────────────────────────────────┘
```

**Field validation:**
- Title required, non-empty after trim, max 200 chars. Live counter rendered as `{N}/200` Geist Mono 10px `--ink-3` to the right of the TITLE label when N > 160.
- Assignee required — defaults to the chatted employee. Select options come from `chat.roster` (the same handler that already feeds the roster rail at `roster-rail.tsx`). The Editor-Agent is excluded from the dropdown (D-03 from Phase 4 — reuse).
- The FROM MESSAGE block is read-only — the operator cannot edit the source body here. If they want to change wording they Keep editing, edit the composer, and reopen.

**Action wiring:**
- `Create task` calls `usePluginAction('chat.createTrueTask')` (new handler per RESEARCH.md Pattern 1) with `{ topicIssueId, sourceMessageUuid: null, title, body, assigneeAgentId, companyId, userId }`. On `{ ok, issueId }`, dialog closes, toggle disarms, composer body clears, and the new inline task card appears in the thread on the next poll (15s) — meanwhile a transient optimistic inline card with status `Creating…` renders immediately.
- `Keep editing` closes the dialog. The composer body is preserved (operator can keep tweaking) and the toggle stays ARMED.
- Loading state: Create button label → `Creating…`, disabled; Keep-editing stays clickable.
- Failure: inline error under the Create button, dialog stays open.

### Pattern C — Inline task card (D-03) `<InlineTaskCard>`

**Where it sits:** Inside `.messages` at the point the marker comment (D-07) lands, ordered by server timestamp like any other message. **NOT a `.bubble`** — distinct visual idiom so a sighted operator never confuses a spun-off task with a sent message.

**Closest existing idiom:** the `.decision-msg` centered typeform (sketch ll. 294-306). The inline task card extends that idiom with task-specific affordances. It is **NOT styled as `eric-bubble` or `agent-bubble`**.

**Layout:**

```
                ┌──────────────────────────────────────┐
                │  ↗  TASK CREATED                      │ ← Geist Mono uppercase, --you
                │                                       │
                │  Lock sec.4 at 12% in v1.5            │ ← Geist 14.5px weight 500
                │                                       │
                │  Assigned to CFO · Capital · Economics│ ← Geist 13px, --ink-2
                │  BEAAA-202  ·  [In progress]          │ ← Geist Mono + status pill
                │                                       │
                │                          10:13        │ ← timestamp, Geist Mono 10px --ink-3
                └──────────────────────────────────────┘
```

**Visual details:**
- `align-self: center` (matches `.decision-msg`).
- `max-width: 520px`.
- `padding: 12px 16px` (vertical = md token 12px, horizontal = lg token 16px — both multiples of 4).
- `border: 1px solid var(--line-bright)` (a brighter line than the default `--line` — assertive but not alarming; the gold accent is reserved for the eyebrow only).
- `background: var(--bg-3)`.
- `border-radius: 2px` (matches existing `.bubble` `border-radius: 2px` and `.decision-msg` lack of radius — settle on 2px for visual consistency).
- The `BEAAA-NNN` identifier renders as a chip identical in shape to the existing `.ref-inline` chip at sketch ll. 199-211 — uses the EXISTING ref-chip primitive `src/ui/primitives/ref-chip.tsx`. Clicking the chip opens the inline ref peek (Phase 2 primitive — no navigation, per core value).
- The status pill renders via the new `<ChatTaskStatusPill>` (a thin wrapper over the existing chat status-pill classes `task-row .st.done / .review / .todo / .blocked` at sketch ll. 375-379). Live-updates on poll (D-08).

**State pill mapping (D-08 live status):**

| Host issue status | Pill class | Label |
|-------------------|------------|-------|
| `backlog` | `.st.todo` | `Backlog` |
| `todo` | `.st.todo` | `Todo` |
| `in_progress` | `.st.todo` (using the `--ink-3` neutral track to signal "moving but not yet noteworthy" — matches today's `.task-row` idiom) | `In progress` |
| `in_review` | `.st.review` (warn) | `In review` |
| `done` | `.st.done` (live green) | `Done` |
| `blocked` | `.st.blocked` (alert) | `Blocked` |
| `cancelled` | `.st.todo` with `text-decoration: line-through` | `Cancelled` |
| Loading / undefined | `.st.todo` with `opacity: 0.5` | `· — ·` |

**Why not a hover-affordance on the source message:** D-03 explicitly locks "an inline task card in the thread (not just a pill on the source message, not silent)." The card is a first-class participant in the conversation timeline.

### Pattern D — Active tasks owned, live status (D-08, context rail extension)

**Where it sits:** Replaces the existing stub in `context-rail.tsx:51-54`. Renders inside the existing `<aside class="ctx">` under the existing `<h3>Active tasks owned</h3>` heading (sketch l. 708).

**Layout:** Reuses the existing `.task-row` row idiom verbatim (sketch ll. 368-379). One row per spun-off task for the current topic. Each row is:

```
┌───────────────────────────────────────────────┐
│ BEAAA-202   CSO deck v3 — broker-comm…   In progress │
└───────────────────────────────────────────────┘
```

Three columns via `grid-template-columns: auto 1fr auto` (existing).

**Ordering:**
1. Open tasks first (anything not `done` / `cancelled`), ordered by `created_at` descending — newest at top.
2. Then `done` tasks created in the last 7 days, also descending.
3. Older `done` / all `cancelled` collapse under `Show {N} more` (rendered as a `--ink-3` Geist Mono Quick-actions-style row at the bottom of the section). Clicking expands inline.

**Data source:** A new `chat.taskOwned` getData handler (worker-side; not in scope for this UI-SPEC) returns `[{ issueId, identifier, title, status, createdAt }]` for the current topic via `ctx.issues.list({ originKind: 'plugin:clarity-pack', originId: 'chat-task:<topicIssueId>:*' })` (RESEARCH.md Open Question 2 path). Polls on the same cadence as `chat.messages` (15s) — re-uses `usePoll` visibility-pause.

**Empty state:** When the list is empty for the current topic, render the empty-state copy from the copywriting contract above, NOT a `<p class="ctx-empty">` placeholder.

**Click behavior:** Each row's `BEAAA-NNN` chip opens the task via `useHostNavigation().linkProps()` — same idiom as the inline task card and the Reader-view ref chip.

### Pattern E — Archive topic affordance (D-10)

**Placement:** A new `.qa` row in the existing `.quick` block at sketch ll. 388-392 (already rendered in `context-rail.tsx:70`). Inserted as the FIRST `.qa` row above the existing `⌕ Search` and `⏸ Pause heartbeat` rows.

**Two-step confirmation:**
1. Click `⌗ Archive this topic` → button text mutates in place to `Archive {topic.title}? · YES · CANCEL` (the row grows ~8px in height to accommodate). YES is a `.btn` (gold-accent existing button), CANCEL is a `.btn.ghost`.
2. YES → calls `usePluginAction('chat.topic.archive')` (new handler, sets `chat_topics.archived = true`). On success the entire `.qa` row collapses for 2s, then re-renders as `✓ Topic archived`, then the topic strip refreshes and the archived topic disappears.

**Why two-step in-row and not a dialog:** Archive is reversible (the row still exists in the DB; the operator can re-open from the `+N archived` strip toggle). A modal would over-weight it. The two-step in-row pattern is also smaller than introducing a second dialog primitive.

**`+N archived` pill on the topic strip:**
- Renders at the right end of the topic strip when `archived_count > 0`.
- Styled as `.topic` but with `border: 1px dashed var(--line)`, `color: var(--ink-4)`, `font-style: italic`.
- Clicking toggles archived topics into the strip at `opacity: 0.6` with an inline hover-action `Unarchive`.

### Pattern F — Diagnostics toggle (D-16)

**Placement:** A `.btn.ghost` rendered in the existing `.head-actions` group at sketch l. 503-506, to the LEFT of `Open profile`.

**States:**
- OFF (default): `⏿ Diagnostics` — visually identical to `Open profile`.
- ON: `⏿ Diagnostics ON` — the same button with the existing `.btn` (non-ghost) styling, indicating active state.

**Behavior:** Toggling ON adds an inline-revealed group beneath each runtime-noise comment in the thread (worker filter receives a `includeDiagnostics: true` flag — the same toggle the existing `chat.messages` handler will read per D-15). Revealed comments render in the runtime-noise styling (Geist Mono, `--ink-4`, dashed border, NOT bubble) defined in the copywriting contract above.

**Persistence:** The toggle state is local React state — does NOT persist across reloads. (D-16 emphasizes "diagnosability" not "default-on for power users." If an operator reaches for it twice, that's fine; if they need it always-on, they file an issue.)

### Pattern G — Host-stuck notice (D-13)

**Placement:** Inside the `.messages` container, `position: sticky; top: {auto-refresh-height}px`, so it sits BELOW the existing `.auto-refresh` indicator but ABOVE the thread, ALWAYS visible while the topic is stuck.

**Visual:**

```
┌──────────────────────────────────────────────────────────────┐
│ ⚠ TOPIC STUCK HOST-SIDE                                       │ ← Geist Mono uppercase tracked
│                                                                │
│ Paperclip's recovery owner has this topic and the plugin       │
│ cannot release it.                                             │
│                                                                │
│ Conversation paused here. The topic in classic Paperclip       │
│ shows the host-side status.                                    │
│                                                                │
│ [ Open in classic Paperclip ]                                  │ ← .btn.ghost
└──────────────────────────────────────────────────────────────┘
   ▲ 3px solid --alert left rule, --warn-soft fill
```

**Trigger:** The new `chat.messages` handler (per D-15) returns a `{ topicStuck: true, recoveryOwner: string | null }` field when the topic issue's `activeRecoveryAction` is set or `successfulRunHandoff` is exhausted (RESEARCH.md `recovery/service.ts` reference). The UI reads that field and renders the banner unconditionally when `topicStuck` is true.

**Auto-dismiss:** On the next 15s poll, if `topicStuck` returns false (e.g. an operator-side intervention released the topic), the banner unmounts silently. NO toast, NO "stuck cleared" confirmation — the absence of the banner is the confirmation.

**Why a banner, not a toast:** D-13's "never silent" + the banner's persistent visibility are load-bearing for operator awareness. A toast would dismiss and the operator would forget. A banner sticks until the underlying state changes.

## Accessibility

- The composer toggle uses `<button type="button" aria-pressed="…">` with the ARMED state surfacing `aria-label="Sending as task — turn off to send a normal message"`.
- The confirm dialog uses the native `<dialog>` element which provides focus-trap and ARIA semantics for free. `aria-labelledby` points at the dialog heading; `aria-describedby` points at the helper text.
- The inline task card is rendered inside `<article class="msg">` with `role="group"` and `aria-label="Task created — {title}, assigned to {employee}, status {status}"`.
- The host-stuck banner uses `role="alert"` so it announces on render (the operator must know the conversation has paused).
- The diagnostics toggle uses `aria-pressed`.
- All status pills carry `aria-label="Status: {label}"` so a screen reader announces the status word, not just the color class.
- Focus visibility on every new button: `outline: 1px solid var(--ink-2); outline-offset: 2px` on `:focus-visible` (matches the rest of the chat surface).

## Interaction Flow (end-to-end, for executor verification)

**Operator creates a true task from chat:**

```
1. Operator clicks "Send as task" toggle in composer → toggle visually arms (gold border+dot, text "Sending as task").
2. Operator types body in composer textarea. Placeholder reads "Task for CFO… Enter to open the task form".
3. Operator hits Enter (or clicks "Open task form").
4. Confirm dialog opens.
   - Title field pre-filled with first ~80 chars of body.
   - Assignee pre-filled with CFO.
   - FROM MESSAGE shows full body in a quote block.
5. Operator tweaks title (optional) and clicks "Create task".
   - Create button: "Creating…"
6. chat.createTrueTask succeeds → dialog closes.
   - Toggle disarms.
   - Composer clears.
   - Optimistic <InlineTaskCard> appears in thread with status="Creating…".
7. On next 15s poll, the marker comment (D-07) arrives in chat.messages, the inline card reconciles to its persisted form (now showing BEAAA-NNN), and a corresponding row appears in "Active tasks owned" with the live status pill.
8. As CFO works the task, the status pill updates on each poll (Todo → In progress → In review → Done).
```

**Operator archives a topic:**

```
1. Operator clicks "⌗ Archive this topic" in the rail's Quick actions block.
2. Row mutates in place to "Archive Capital sensitivity? · YES · CANCEL".
3. Operator clicks YES.
4. chat.topic.archive succeeds → row collapses for 2s, shows "✓ Topic archived", then refreshes.
5. The topic disappears from the topic strip; a "+1 archived" muted pill appears at the right end of the strip.
6. (Optional) Operator clicks "+1 archived" → strip expands to include the archived topic at opacity 0.6 with an "Unarchive" hover action.
```

**Operator diagnoses runtime noise:**

```
1. Operator notices the agent has gone silent and clicks "⏿ Diagnostics" in the thread head.
2. Toggle arms ("⏿ Diagnostics ON").
3. chat.messages re-fetches with includeDiagnostics: true.
4. Previously-filtered runtime comments now render inline in the thread, dashed-bordered Geist-Mono dim style — clearly NOT bubbles.
5. Operator sees the "finish_successful_run_handoff" / "needs a disposition" / "blocked on a recovery owner" comments and understands the topic is host-stuck.
6. The host-stuck banner is (probably) also showing because the worker classified the topic as stuck — the two affordances corroborate.
```
