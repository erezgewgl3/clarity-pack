<objective>
Phase 4.2 single-plan opener — the **Reader↔Chat Bridge**. Phase 4.1 (CTT-01..08) shipped the Chat → Task direction: operators turn chat into real assigned issues. This plan ships the symmetric inverse — from any Paperclip issue's Reader view the operator has a deterministic `Continue in chat` path, and the issue↔conversation graph becomes navigable in **both** directions via bidirectional backlinks.

Purpose:
- **RCB-01/RCB-02 — deterministic Continue-in-chat.** A `Continue in chat` primitive in the Reader header whose behaviour is fully determined by the issue's lineage (chat-spawned task → jump to source topic + comment; cold/regular assigned task → pre-seeded New Topic dialog; chat-topic issue → button hidden; unassigned → disabled with guidance). No navigation guessing — this is the "zero rabbit-holes" core value (CLAUDE.md) applied to the inverse direction.
- **RCB-03 — chat deep links.** The chat surface learns to honour URL params so the Reader can hand it an exact destination (topic + comment, or a seeded New Topic dialog).
- **RCB-04/05/06 — bidirectional backlinks.** Topics created from a Reader persist `origin_issue_id`; the chat thread shows an `About <COU-NNNN> ↗` chip; the Reader header surfaces `<N> conversations about this issue ↗`.
- **RCB-07 — additive everywhere.** Migration 0009 is `IF NOT EXISTS`-idempotent and plugin-namespace only; pre-0009 topics and button-less Readers keep working.

<design_source>
The locked design lives in project memory `phase-4.2-deferred-from-4.1` §"Phase 4.2 OPENING SCOPE". Recall it before Task 1. The two load-bearing tables it contains are reproduced here so this plan is self-contained:

**Issue-lineage → Continue-in-chat routing (RCB-01):**

| Issue lineage | Detected by | `chat.openForIssue` route | UI behaviour |
|---|---|---|---|
| Chat-spawned task | `originId` matches `chat-task:<topicIssueId>:<commentId-or-composer>` | `existing-topic` (carries `topicIssueId`, `sourceCommentId`) | Navigate chat → source topic → scroll to + 1.5s flash-highlight source comment |
| Cold/standalone task | `originId` matches `cold-task:<userId>:<unix-ms>` | `new-topic-needed` (carries `assigneeAgentId`, `seedTitle`, `seedBody`) | Navigate chat → assignee employee → open New Topic dialog pre-seeded |
| Regular assigned task (assignee, no chat origin) | has `assigneeAgentId`, `originId` matches neither prefix | `new-topic-needed` | Same as cold-task |
| Chat-topic issue itself | `originKind === 'plugin:clarity-pack:chat-topic'` | `topic-itself` | Render NOTHING — already a chat surface |
| No assignee | `assigneeAgentId` is null/empty | (returns `error: 'NO_ASSIGNEE'`) | Disabled button + tooltip "Assign this issue to an employee before opening chat." |

**`chat.openForIssue` contract:**
- Input: `{ companyId, userId, issueId }`
- Output: `{ kind: 'chatOpenForIssue', route: 'existing-topic' | 'new-topic-needed' | 'topic-itself', topicIssueId?, sourceCommentId?, assigneeAgentId?, seedTitle?, seedBody?, error? }`
- `seedTitle` = `issue.title`; `seedBody` = `Continuing from <COU-NNNN>: <issue.title>` (the `<COU-NNNN>` is the issue's human `identifier`).
</design_source>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Reader UI → chat.openForIssue → ctx.issues.get | Untrusted UI-supplied `{companyId, userId, issueId}` drives a host issue read. Opt-in gate + company-scope is the enforcement layer; the handler is read-only (no mutation). |
| Reader-built URL → chat surface params | `seedTitle`/`seedBody`/`originIssueId`/`topic`/`comment` are operator-controlled strings carried through the URL into the chat surface and (for originIssueId) into a DB write. |
| chat.topic.create → chat_topics.origin_issue_id | A UI-supplied `originIssueId` is written to a plugin-namespace column. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-04.2-01-01 | T (Tampering) | A crafted `issueId` to `chat.openForIssue` reads an issue in another company | mitigate | `ctx.issues.get(issueId, companyId)` is company-scoped by the host; the opt-in-guard additionally requires a `clarity_user_prefs` row for the caller in this company. |
| T-04.2-01-02 | E (Elevation) | `chat.openForIssue` induced to mutate the host issue | mitigate | By construction — the handler is read-only; Task 2 acceptance pins `grep -c "ctx.issues.update"` = 0. |
