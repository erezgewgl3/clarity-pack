**GAP-RCB-01-STYLING — refined root cause (the VERIFICATION report is PARTIALLY WRONG here):**
The VERIFICATION report says `.clarity-continue-in-chat` "has no CSS rule at all" and the fix belongs in `chat.css`. That is incorrect. A `.clarity-continue-in-chat` rule **already exists** in `src/ui/primitives/theme.css` at lines 771-786, scoped `[data-clarity-surface='reader']`, and `theme.css` IS the Reader surface's stylesheet (injected at runtime by `src/ui/index.tsx` — `injectClarityStyles(themeCss, 'data-clarity-pack-styles')`).

The ACTUAL defect: that rule's declared values consume CSS custom properties — `var(--clarity-you)`, `var(--clarity-you-soft)`, `var(--clarity-ink-3)`, `var(--clarity-line)` — and those tokens are defined in `theme.css` **only under the `[data-clarity-surface='situation-room']` selector block** (around line 371), NOT under the base `[data-clarity-surface]` block (lines 13-34) and NOT under any `[data-clarity-surface='reader']` block. On the Reader surface `var(--clarity-you)` resolves to nothing → `background` falls through to transparent → the button renders unstyled / near-invisible. The same undefined-token defect affects `.clarity-reverse-topics-*` (RCB-06) and the topic-strip About-chip on `[data-clarity-surface='reader']` if they share those tokens. The fix is to make the tokens resolve at Reader scope (promote them to the base `[data-clarity-surface]` block, or add a `[data-clarity-surface='reader']` token block) — NOT to add an unscoped rule (that would violate SCAF-06 / `check-css-scope.mjs`).

**GAP-RCB-03-DEEPLINK — root cause (confirmed against source):**
`continue-in-chat-button.tsx` builds a full path-with-query string (`/<prefix>/chat?topic=…&comment=…` or `…?newTopic=1&seedTitle=…&seedBody=…&originIssueId=…`) and calls `useHostNavigation().navigate(href)`. The chat surface (`chat/index.tsx`, the RCB-03 effect around lines 290-355) reads `useHostLocation().search` and parses it with `new URLSearchParams(search)`.

The SDK type contract (`@paperclipai/plugin-sdk/dist/ui/types.d.ts`): `HostNavigation.navigate(to: string, options?: HostNavigationOptions)` takes a single `to` string; `HostLocation.search` is documented as "Current search string, e.g. `?tab=config` (includes the leading `?`)". The open question — and the load-bearing one — is whether a query string embedded in the `to` argument of `navigate()` is actually carried into `useHostLocation().search`. The drill says it is NOT: the chat surface opens to its empty state, so `search` arrives empty (or without the params). The two hooks do not agree on the contract. The executor MUST verify empirically how the host carries a query string (test the SDK's host-router fake / dev-server, or inspect the SDK runtime) and make `navigate()`-emit and `search`-read agree — whatever the verified mechanism is (a query string in `to`; a `?`-prefixed value; `linkProps`; the `state` option; whatever the SDK actually honours).
</gap_root_cause_findings>
