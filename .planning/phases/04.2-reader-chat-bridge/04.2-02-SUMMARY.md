1. **`?query` tail** stripped by `resolveHref()` when applying the company prefix — confirmed by the bare URL bar.

Plan 04.2-02 swapped one stripped carrier (`?query`, the 04.2-01 carrier) for another stripped carrier (`{ state }`). The contract test D1–D6 passed because the **router fake** honored `state` — a faithful model of the SDK's type docstring, but **not** of the live host's actual behavior. **Same class of test-vs-production gap as 04.2-01, one structural layer deeper.**

## Lesson — Router fakes derived from SDK docstrings can pass D-tier contract tests while production fails

The 04.2-01 drill failure was diagnosed (correctly) as "two independently-mocked hooks let the cross-hook contract drift." 04.2-02's fix replaced two independent mocks with ONE shared router fake — a structurally better test design. But the fake's behavior was modeled from the SDK's `dist/ui/types.d.ts` docstrings, NOT from the live host's actual implementation. The fake honored `state` because the docstring says `state` is "forwarded by the host router." The live host strips it.

This is **fake-vs-production drift** at one structural layer deeper than 04.2-01. The structural mechanism that stops this from recurring at a third layer:

> **A contract test against a host capability MUST exercise either the live host runtime (via `paperclip-plugin-dev-server`) OR a fake that was empirically derived from the host's actual implementation (e.g. by reading `node_modules/@paperclipai/plugin-sdk/dist/ui/runtime.js` until you can write the fake from observed behavior). A fake derived from type docstrings is a re-statement of assumptions, not a test.**

Filed as MemPalace drawer `router-fake-vs-production-host` (`clarity_pack/runbook`) so the 04.2-03 planner sees it before designing the next contract test.
