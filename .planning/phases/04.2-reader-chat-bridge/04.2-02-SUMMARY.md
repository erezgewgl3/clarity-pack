## Key Decisions

1. **Carrier channel: `navigate()`'s `state` option + `?query` fallback.** The empirical SDK finding (documented in `deep-link.mjs`'s header against `node_modules/@paperclipai/plugin-sdk/dist/ui/{types.d.ts,hooks.js,runtime.js}`): the host's `resolveHref` is a path-only company-prefixer that strips/garbles a `?query` tail in `navigate()`'s `to`; the SDK type contract guarantees `state` is forwarded verbatim. We carry the params on `state` (load-bearing) AND append `?query` to `to` (best-effort fallback so a reloaded chat URL with no React `state` can still recover from `search`). One contract, two channels, both verified.
2. **Pure-function emit + read in `deep-link.mjs`.** The codebase has no React render harness under `node --test`; the only way to write a REAL cross-hook contract test (D3: NO-INDEPENDENT-MOCKS) was to extract the contract into pure exported functions and round-trip them through ONE shared router-state object. Same `.mjs` pattern as `reasoning-block-parser.mjs`.
3. **GAP-RCB-01 fix at the token level, not the rule level.** The VERIFICATION report said the rule "had no CSS" — incorrect; the rule existed, the tokens it consumed didn't resolve at Reader scope. Promoting the shared `--clarity-*` tokens into the base block is one structural fix that cascades to every Reader-mounted clarity-pack affordance (button, reverse-topics, About-chip) — a single source of truth, no per-rule patches.
4. **Keep the `?query` tail.** Even though `state` is the load-bearing channel, the URL keeps human-readable params so a copied chat link can be re-opened and the chat surface can recover them via `search`. The two-channel design is defense-in-depth — if either the host router or the React `state` forwarding ever changes, the other half degrades gracefully.
