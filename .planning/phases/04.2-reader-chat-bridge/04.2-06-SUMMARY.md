## What shipped

Two atomic-commit fixes for the visible UUID-leak defects D9 + D10 surfaced during the 2026-05-24 Countermoves drill of rc.3. Both share the same root-cause family as D7 (missing display-name resolution → fallback to raw UUID) but in different code paths the D7 fix did not cover.

## Defect disposition

| ID | Defect (rc.3 behaviour) | Fix | Commit |
|----|-------------------------|-----|--------|
| **D9** | Reader-header Continue-in-chat button rendered `Continue in chat with 618ebd0d-4d39-45f4-8380-3b30b205d02d →` because the UI fell back to `result.assigneeAgentId` when no display name was available. | **chat-open-for-issue.ts** resolves the assignee name via `ctx.agents.get(assigneeAgentId, companyId)` and threads `assigneeName: string \| null` into both result branches (existing-topic + new-topic-needed). Degrades silently to null on lookup failure (warn-logged). **continue-in-chat-button.tsx** consumes `assigneeName ?? 'this employee'` — never the UUID. | `21008f6` |
| **D10** | Topic-strip "About `<UUID>` ↗" chip leaked the origin issue's UUID into visible text + navigated to `/COU/issues/<UUID>` which 404s per runbook `paperclip-issue-url-pattern`. | **chat-topics.ts** new `resolveOriginIdentifiers` helper resolves each distinct `originIssueId` UUID to its BEAAA-NNN identifier via `ctx.issues.get` (one host roundtrip per distinct id; typical strip = 1-5 calls). TopicEntry exposes `originIssueIdentifier: string \| null`. **topic-strip.tsx** derives `aboutIssueId` from the resolved identifier and **hides the chip entirely** when resolution degraded — no broken `/issues/<UUID>` link can render. | `f9f9f6a` |
