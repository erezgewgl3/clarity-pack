<decisions>
## Implementation Decisions

### D-7 routing — resolution algorithm (`chat.openForIssue` rewrite)

- **D-01 Resolution algorithm.** `chat.openForIssue` adds a new resolution step between the existing chat-task lineage check (today's `existing-topic`) and the cold-task fallback (today's `new-topic-needed`):
  1. (Unchanged) Issue's `originId` matches `chat-task:<topic>:<source>` → return `existing-topic` for that exact topic + source comment.
  2. **(New)** Query `chat_topics WHERE origin_issue_id = $issueId AND employee_user_id = $assigneeAgentId` ordered by activity DESC (includes archived rows; see D-04). Branch on cardinality:
     - 0 matches → fall through to step 5.
     - 1 match → return `existing-topic` shape with that topic's id (silent resume; no `sourceCommentId` because we are resuming the thread, not a specific comment).
     - ≥2 matches → return a **new** route shape `existing-topics-ambiguous` carrying the candidate topic list (id + last activity + last message preview + archived flag). UI auto-opens the RCB-06 popover pre-filtered (see D-02).
  3. (Unchanged) `topic-itself` — the issue IS a chat-topic; render nothing.
  4. (Unchanged) `NO_ASSIGNEE` — disabled button + tooltip.
  5. (Unchanged) `new-topic-needed` — seeded New Topic dialog.

- **D-02 Picker UX = reuse RCB-06 popover.** The disambiguation surface for the ≥2-match case is the existing reverse-topics popover at [src/ui/surfaces/reader/reverse-topics-link.tsx](src/ui/surfaces/reader/reverse-topics-link.tsx). Continue-in-chat auto-opens it pre-filtered to same-assignee matches (an entry-point prop tells the popover whether to filter and whether to suppress the manual-click toggle). Topics with other employees stay accessible by toggling the filter off. **Zero new picker component.** This implicitly resolves the deferred D-7 Q4 (RCB-06 popover relationship): the two affordances share one component, two invocation sources.

- **D-03 Cross-employee fall-through.** When 0 same-assignee matches but N>0 other-employee topics exist (e.g. issue reassigned from CTO to CMO), `chat.openForIssue` returns `new-topic-needed` (today's seeded dialog) — treats the issue as cold for this assignee. Other-employee threads remain reachable via the RCB-06 manual-click path (which lists ALL topics about this issue, unfiltered). Two clean rules: button = current-assignee only; popover = full history.

- **D-04 Archived topics: auto-unarchive on resume.** The new step-2 lookup **includes** archived rows (no `archived_at IS NULL` filter). On silent resume of a single archived match, the resolver also flips `archived_at = NULL` via the existing `setChatTopicArchived(topicId, false)` repo helper (Plan 04.1-05). Silent — no toast, no confirmation prompt. Worker logs the unarchive at INFO level for audit. CTT-07 archive is plugin-side (host issue untouched), so this is trivially reversible by the operator if needed.

### D-7 routing — Claude's discretion (lock in plan)

- **D-05 Sort order / "most recent" tiebreaker.** Most-recent activity DESC = `MAX(chat_topics.last_activity_at, max(chat_messages.created_at) WHERE chat_messages.topic_id = chat_topics.id)`. Same definition for 1-match selection (when collapsing N=1 silently) and picker default ordering.
- **D-06 Reader button label unchanged.** Keep `Continue in chat with <employeeName> →` for both existing-topic flavors (originId lineage and origin_issue_id reverse-lookup) and for `existing-topics-ambiguous`. One consistent label; tooltip differentiates the destination ("Open source topic <id> →" for originId lineage; "Resume conversation about <issueIdentifier> →" for reverse-lookup; "Pick from N conversations about <issueIdentifier> →" for ambiguous).
- **D-07 Deep-link contract unchanged.** Picker-resolved navigation MUST emit the SAME `existing-topic` deep-link shape (URL_HASH carrier, Plan 04.2-03) AFTER the operator picks a topic in the popover. The `existing-topics-ambiguous` route is consumed by the Reader-side button only; the chat surface still only knows `existing-topic` / `new-topic-needed`. No new chat-surface parse path; do not extend `parseChatDeepLink`.
- **D-08 D9/D10 hygiene pattern.** The new ambiguous-route shape MUST carry server-resolved display names (employee name, source-issue BEAAA-NNN identifier) in the same pattern Plan 04.2-06 established — NEVER fall back to a raw UUID in any text the operator sees.

### Polish fold-ins

- **D-09 D-5 mitigation (operator runbook).** Update `runbook/operator-gotchas.md` (the catalog established in Plan 01-05) with a new entry: **the next operator drill's Step 9 MUST capture either (a) a DevTools Network HAR or screenshot showing the rc.X contract firing OR (b) a test issue with cached TL;DR + ≥2 AC sources whose manual toggle materially changes the auto-status derivation**. Clarity-pack does not modify any host agent's output. Plan 05-03's `reader-ac-autostatus.ts` scanner is unchanged.

### Implicit decisions (no work, just keep)

- **D-11 Empty-state behavior unchanged.** Cold task + 0 same-assignee matches + 0 other-employee matches → today's `new-topic-needed` seeded New Topic dialog. Identical to current rc.6 behaviour for genuinely cold issues.
- **D-12 Agent-side context handoff unchanged.** Resume itself does not post a system comment. The agent is woken natively on the operator's first `chat.send` after resume per Plan 04.1-03 `ensureTopicWakeable`. No new agent prompt, no system pre-amble.

<specifics>
## Specific Ideas

- Eric's verbatim quote (260524-sm8 drill, 2026-05-25): *"when I'm in the reader and I'm looking at something, at one of the items, and I want to continue to chat with the agent on this topic, there's no real way to do it. As soon as I click the golden button, it takes me to the chat, but it tries to open a new task."* — operator's mental model is "continue with the agent assigned to this", which is exactly what D-01 + D-03 encode.
- Two clean affordances by construction: **Continue-in-chat = current-assignee, optimistic resume**; **RCB-06 popover = full conversation history about this issue, manual choice**. Do not let scope creep blur the line.
- The unarchive-on-resume design (D-04) intentionally privileges operator-friction-zero over preserving the "archived" signal — Eric is the only operator on Countermoves, and CTT-07 archive is reversible. If a multi-operator model ever emerges, revisit.
