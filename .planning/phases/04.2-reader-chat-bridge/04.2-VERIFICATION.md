### GAP-RCB-03-CARRIER — NEW (refines GAP-RCB-03-DEEPLINK with conclusive root cause from 04.2-02 drill)

- **status:** failed
- **requirement:** RCB-03 (functional — load-bearing); blocks RCB-05 and RCB-06 navigation
- **severity:** high
- **supersedes:** GAP-RCB-03-DEEPLINK (Plan 04.2-01's drill-time hypothesis)
- **found:** Plan 04.2-02 / v0.9.1 operator drill on live Countermoves, 2026-05-23 — confirmed on drill path 1 (assigned task `COU-2215`).
- **symptom:** clicking `ContinueInChatButton` navigates to `/COU/chat` but the chat surface opens to its generic "Select an employee / No topics yet" empty state. Same visible symptom as 0.9.0; 0.9.1 swapped one stripped carrier for another stripped carrier.
- **root cause (CONCLUSIVE, empirical):** the Paperclip host wrapper around `useHostNavigation().navigate()` strips BOTH carriers for cross-route navigation:
  - **`?query` tail** is stripped by `resolveHref()` when applying the company prefix. Evidence: URL bar after click reads `countermoves.gl3group.com/COU/chat` (no `?...`).
  - **`{ state }` option** is dropped somewhere in the host's `navigate` wrapper before reaching react-router's `useNavigate`. Evidence: `JSON.stringify({ state: history.state })` after click shows `{ "state": { "usr": null, "key": "vtw64gqp" } }`. The `usr` field is react-router's slot for the user-provided `state` argument — it's `null`, meaning the host didn't forward our `clarityDeepLink` payload.
- **why 0.9.1 unit tests missed it:** Plan 04.2-02's contract test (`test/ui/continue-in-chat-deeplink-contract.test.mjs` D1-D6) uses ONE shared router fake (a structural improvement over 04.2-01's two-independent-mocks) — but the fake's behavior is modeled from the SDK's `dist/ui/types.d.ts` docstrings, NOT from the live host's actual implementation. The fake honored `state` because the docstring says `state` is "forwarded by the host router." The live host strips it. **Fake-vs-production drift at one structural layer deeper than 04.2-01.**
- **fix target:** pick a carrier the host does NOT strip. Candidates in priority order:
  1. **URL hash** (`/COU/chat#h=<encoded>`) — `HostLocation.hash` is a first-class SDK type field; if hash is also stripped, every plugin's navigation contract is broken (implausible). Cheapest fix candidate.
  2. **sessionStorage handoff** — plugin UI is same-origin trusted JS (CLAUDE.md trust model); sessionStorage is available. Write on click, read+delete on chat surface mount.
  3. **Worker-side handoff row** (new `chat_handoffs` table or in-memory worker singleton) — biggest scope (migration 0010 + new handler); most robust (audit trail + refresh-safe).
- **fix files:** the existing `src/ui/surfaces/chat/deep-link.mjs` module is reusable — the contract single-source-of-truth design is correct; only the *carrier* inside it needs swapping. Plus `continue-in-chat-button.tsx` + `chat/index.tsx` + `reverse-topics-link.tsx` (the three consumers) for any wiring changes the new carrier requires.
- **NEW REQUIRED test discipline:** Plan 04.2-03's contract test MUST exercise the actual host runtime (via `paperclip-plugin-dev-server`) OR a fake empirically derived from inspecting `node_modules/@paperclipai/plugin-sdk/dist/ui/runtime.js` until you can write the fake from observed behavior. A fake derived from type docstrings is a re-statement of assumptions, not a test. (See Lessons.)

## Lessons

### Lesson 1 — Router fakes derived from SDK docstrings can pass D-tier contract tests while production fails

The 04.2-01 drill failure was correctly diagnosed as "two independently-mocked hooks let the cross-hook contract drift." 04.2-02's fix replaced two independent mocks with ONE shared router fake — structurally better. But the fake's behavior was modeled from the SDK's `dist/ui/types.d.ts` docstrings, NOT from the live host's actual implementation. The fake honored `state` because the docstring says `state` is "forwarded by the host router." The live host strips it.

**Fake-vs-production drift at one structural layer deeper than 04.2-01.** The structural mechanism that stops this from recurring at a third layer:

> **A contract test against a host capability MUST exercise either the live host runtime (via `paperclip-plugin-dev-server`) OR a fake that was empirically derived from the host's actual implementation (by reading `node_modules/@paperclipai/plugin-sdk/dist/ui/runtime.js` until you can write the fake from observed behavior). A fake derived from type docstrings is a re-statement of assumptions, not a test.**

Filed as MemPalace drawer `router-fake-vs-production-host` (`clarity_pack/runbook`). Plan 04.2-03's planner must read it before designing the next contract test.

### Lesson 2 — Drill faithfully even when the source unit suite is green

The 04.2-02 unit suite was 1323/1321 GREEN with zero failures — and the deep-link still broke on the first live click. This is a recurring pattern in this project (Plan 02-03b CTM-01 missed `companyId === null`; Plan 04.2-01 missed the cross-hook contract; Plan 04.2-02 missed the docstring-vs-runtime drift). The live operator drill is the only test that runs against the same code path the operator does. **Always drill load-bearing UI changes on Countermoves — never close a plan on green unit tests alone.**
