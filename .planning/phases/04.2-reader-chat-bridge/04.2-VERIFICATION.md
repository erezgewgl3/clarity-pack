## Gaps

The following gaps are consumable by `/gsd:plan-phase 4.2 --gaps`.

### GAP-RCB-01-STYLING

- **status:** failed
- **requirement:** RCB-01 (visual)
- **severity:** medium (visual; the button is functionally present but near-invisible)
- **found:** Task 8 operator drill on live Countermoves, 2026-05-22, confirmed on drill paths 1 and 2.
- **symptom:** `ContinueInChatButton` renders unstyled / near-invisible on the Reader surface.
- **root cause:** the button uses `className="btn primary clarity-continue-in-chat"`, but the `.btn` and `.btn.primary` rules in `src/ui/styles/chat.css` (lines 328 and 1702) are scoped under the `[data-clarity-surface="chat"]` selector and never match on the Reader surface; `.clarity-continue-in-chat` has no CSS rule at all.
- **why unit tests missed it:** `node --test` does not evaluate the CSS cascade — the component renders correct DOM, just unstyled.
- **fix target:** add a surface-correct CSS rule for the Reader-mounted button — e.g. a `.clarity-continue-in-chat` rule in `src/ui/styles/chat.css` NOT gated under `[data-clarity-surface="chat"]`, carrying the primary-button visual weight.
- **fix files:** `src/ui/styles/chat.css` (primary); optionally `src/ui/surfaces/reader/continue-in-chat-button.tsx` if a class rename is cleaner than an un-scoped rule.

### GAP-RCB-03-DEEPLINK

- **status:** failed
- **requirement:** RCB-03 (functional — load-bearing)
- **severity:** high (the whole Reader→Chat deep-link mechanism is non-functional)
- **found:** Task 8 operator drill on live Countermoves, 2026-05-22, confirmed on BOTH drill path 1 and drill path 2 — not branch-specific.
- **symptom:** clicking `ContinueInChatButton` navigates to `/COU/chat`, but the chat surface opens to its generic "Select an employee / No topics yet" empty state. Neither `?newTopic=1&employee=&seedTitle=&seedBody=&originIssueId=` (path 1) nor `?topic=&comment=` (path 2) is consumed.
- **root cause:** the `useHostNavigation().navigate()` (button) → `useHostLocation().search` (chat surface) handoff. Plan-execution deviations #3 and #4 swapped both hooks off react-router (`useNavigate` / `useSearchParams` do not exist in this codebase); the two hooks do not agree on how the query string is carried, so the params never reach the chat surface's URL-param effect.
- **why unit tests missed it:** the UI tests mock the host hooks independently, so the navigate-emit and the location-read were each tested in isolation against compatible mocks — the real cross-hook contract was never exercised.
- **fix target:** make the `useHostNavigation().navigate()` emit and the `useHostLocation().search` read agree on the query-string contract — verify how the host carries a query string through `navigate()` and read it back the same way.
- **fix files:** `src/ui/surfaces/reader/continue-in-chat-button.tsx` (the navigate call) + `src/ui/surfaces/chat/index.tsx` (the URL-param effect).
