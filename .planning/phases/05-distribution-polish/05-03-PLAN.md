<objective>
**Phase 5 Plan 03 — DIST-03: AC auto-status event-derived.**

Phase 2 shipped a manual acceptance-criteria checklist on the Reader view (`src/ui/surfaces/reader/ac-checklist.tsx`). Operators check items by hand. DIST-03 promotes this to event-derived auto-status while preserving the Phase 2 manual UX — both modes coexist.

**This plan is PLAN-READY-AWAITING-DESIGN-REVIEW.** It captures the implementation shape but explicitly defers three product questions that need operator alignment before code ships. Once those answers land, the plan becomes execute-ready (the executor takes the captured shape + the resolved answers, ships a TDD pass).
</objective>

<design_questions>
**Q1 — What events derive an AC item to "complete"?**

Candidates:
- (a) **Comment-marker event** — `issue.comments.created` where comment body matches `AC: <item-id-or-text>: ✓` or a similar marker. Cheapest; mirrors the chat-true-task marker-comment pattern from Plan 04.1-02. Cons: requires operator (or agent) discipline to write the marker.
- (b) **Issue-status event** — when the issue moves to `done` or `in_review`, ALL ACs auto-flip to checked. Coarse; doesn't handle partial completion.
- (c) **Attachment-upload event** — when an attachment is uploaded whose filename matches an AC item's deliverable spec, that AC flips. Requires a deliverable-spec field on the AC item that operators specify when creating it. New data shape.
- (d) **Agent-emitted event** — the assigned employee-agent fires a `clarity.ac.completed` plugin event when its heartbeat reasoning concludes the AC is satisfied. Most principled; deepest integration with Paperclip's agent loop. Highest infrastructure cost.

**Recommended default (executor picks if no other guidance): (a) Comment-marker event.** It mirrors the chat-true-task marker-comment pattern Eric already trusts, scales linearly with operator + agent activity, and needs no new schema.

**Q2 — How does auto-status surface in the UI alongside the manual checkbox?**

Candidates:
- (a) **Side-by-side**: each AC row shows BOTH the manual checkbox AND a small "auto: ✓ via @agent-name 12 min ago" indicator. Operator's manual state is the source of truth; auto is informational.
- (b) **Auto-fills manual**: when the auto-status fires, it sets the manual checkbox state. Operator can then uncheck (which "overrides" the auto-status). Conflict: re-firing event might re-check.
- (c) **Two columns**: rename "Done" to "Manually done" + add an "Auto-detected" column. Clearer semantics; more visual real estate.

**Recommended default: (a) Side-by-side, manual is source of truth.** Honors "without breaking Phase 2's manual UX" verbatim — the manual UX is structurally untouched.

**Q3 — What's the conflict resolution model?**

- The manual checkbox is the operator's intent.
- The auto-status is the system's observation.
- They can disagree (the operator un-checks something the system reported as done; OR the operator checks something the system thinks is incomplete).

The recommended default ((a)) keeps these on separate axes — no conflict to resolve. Operator-state is what gets persisted to the AC store; auto-state is render-time only.

Alternative answers to Q3 (if Q2 went option (b) or (c)) would need to specify which side wins on disagreement.

**Until Eric answers Q1/Q2/Q3, this plan does not ship code.**
</design_questions>
