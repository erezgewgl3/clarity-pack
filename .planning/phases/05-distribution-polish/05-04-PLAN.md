<objective>
**Phase 5 Plan 04 — DIST-04: Full-fidelity deliverable previewers + Plan 05-02's deferred visual-regression baseline.**

Phase 2 shipped a placeholder previewer for unrecognized deliverable file types. DIST-04 promotes this to a registry of full-fidelity previewers: **xlsx → grid, pdf → embed, md → rendered, png → img.** This plan is PLAN-READY-AWAITING-DESIGN-REVIEW because the library choices (especially xlsx + md) have meaningful trade-offs that need operator alignment before code ships.

Also captures the Plan 05-02 deferred sub-requirement: **visual-regression baseline in CI** (the third bullet of DIST-05 that was deferred when lockfile-audit + a11y-check shipped).
</objective>

<design_questions>

### Previewers — library + bundle-size choices

**Q1 — xlsx → grid renderer.**

Candidates:
- (a) **SheetJS (`xlsx` npm package)** + a custom React grid (`<table>` with virtualization for large sheets). Pros: industry-standard parsing, handles weird Excel formats, ~500 kB minified. Cons: nearly 2× the current UI bundle (currently 288 kB → would be ~800 kB+); same-origin trust model means it loads on every chat-surface-or-reader render.
- (b) **`@e965/xlsx`** (community fork of SheetJS without the security-advisory baggage). Similar size; same trade-offs.
- (c) **Server-side xlsx → HTML via a worker handler.** Worker uses SheetJS to convert; UI receives parsed HTML. UI bundle stays small. Cons: latency on every preview, worker memory pressure on large files.
- (d) **Defer xlsx → grid; keep placeholder.** Ship pdf/md/png previewers in this plan; address xlsx in a follow-up.

**Recommended default if no other guidance: (c) Server-side conversion** — protects the UI bundle, keeps the trust model unchanged, lets the worker enforce file-size guards centrally. Worker handler: `deliverable.preview` returning `{ kind: 'xlsx-grid', sheets: [{name, rows: string[][]}] }`.

**Q2 — pdf → embed.**

Candidates:
- (a) **Native `<embed type="application/pdf">`** — zero JS deps; browser-native PDF viewer. Works in Chrome/Firefox/Safari/Edge. Cons: no JS-side hooks (can't search, can't sync scroll with comments).
- (b) **`pdfjs-dist`** — Mozilla's pdf.js. ~2 MB of WebAssembly + JS; supports search/annotations/text-layer. Cons: huge bundle; a worker-side pdf.js could mitigate.
- (c) **`<iframe src="/path/to.pdf">`** — same as (a) effectively, with a different DOM container.

**Recommended default: (a) Native `<embed>`.** Smallest blast radius, no new deps, browsers do the right thing. If we later need pdf.js features (search, annotation overlays), we can add it scoped to a "Pro preview" mode.

**Q3 — md → rendered.**

Candidates:
- (a) **`marked`** + **`DOMPurify`** — `marked` parses md to HTML; DOMPurify strips unsafe HTML; result rendered via `dangerouslySetInnerHTML`. But check-a11y R3 currently forbids that primitive. We'd need to add the previewer file to the R3 allowlist, with DOMPurify as the load-bearing safety.
- (b) **`react-markdown`** — parses md and emits React elements directly (no HTML, no `dangerouslySetInnerHTML`). Recommended pattern. Bundle: ~50 kB.
- (c) **`markdown-it`** + custom React renderer.

**Recommended default: (b) `react-markdown`** — keeps the no-dangerouslySetInnerHTML invariant. ~50 kB bundle cost.

**Q4 — png → img.**

Trivial: `<img src={deliverableUrl} alt={filename} loading="lazy" />`. No design questions. The `alt` attribute satisfies check-a11y R1.

### Visual-regression baseline (Plan 05-02 deferred)

**Q5 — Visual-regression approach.**

Plan 05-02 deferred this because no clean approach was obvious without picking infrastructure. Three options carry forward:

- (a) **Static-sketch regression**: Playwright + image snapshot against `sketches/*.html`. Catches design-system drift (token changes, theme drift), NOT runtime UI drift. Cheap (~30 min setup); zero new external deps if we use playwright (already widely available in CI runners).
- (b) **Live-host regression**: Playwright + Paperclip dev-server bootstrap + plugin install + screenshot. Catches real drift. Significant infra (CI needs Postgres + the host workspace).
- (c) **Storybook + Chromatic**: third-party visual-regression-as-a-service. Lowest dev effort; cost + external dependency.

**Recommended default: (a) Static-sketch regression** — pins the design contract, scopes the blast radius, doesn't require a Paperclip host in CI. Adds Playwright as a devDep + a single workflow.

### Previewers (when Q1-Q4 land)

1. **Worker tier — `deliverable.preview` data handler** (NEW, only if Q1=(c) server-side xlsx). Returns kind+payload union: `{kind: 'xlsx-grid', sheets} | {kind: 'pdf-embed', url} | {kind: 'md', body} | {kind: 'img', url} | {kind: 'placeholder', reason}`.

2. **UI tier — `src/ui/primitives/deliverable-preview.tsx`** (NEW). Dispatches on `kind`:
   - `xlsx-grid` → a virtualized `<table>` with sticky headers; renders `sheets[0].rows` initially with a sheet-tabs selector.
   - `pdf-embed` → `<embed type="application/pdf" src={url}>`.
   - `md` → `<ReactMarkdown>{body}</ReactMarkdown>` (per Q3=(b)).
   - `img` → `<img src={url} alt={filename} loading="lazy">`.
   - `placeholder` → existing Phase 2 placeholder, unchanged.

3. **Reader integration**: extend `src/ui/surfaces/reader/index.tsx` to consume the new data hook + render `<DeliverablePreview>` for the deliverable section.

4. **Tests**:
   - Per-kind unit test for `<DeliverablePreview>` render output.
   - Worker handler test for `deliverable.preview` if Q1=(c).
   - check-a11y allowlist update if any kind uses `dangerouslySetInnerHTML` (per Q3=(a)).

### Visual-regression baseline (when Q5 lands)

For Q5=(a) static-sketch regression:
- Install `@playwright/test` as devDep.
- New `test/visual/sketch-regression.test.mjs` — loops over `sketches/*.html`, renders in headless Chromium, takes a snapshot, diffs against `test/visual/baselines/<sketch>.png`.
- New `.github/workflows/visual-regression.yml` — runs the test on PR; on failure, uploads the diff image as an artifact for review.
- Add a `pnpm visual:update` script to regenerate baselines.
