## Deviations from Plan

### Rule 3 -- Test 2 param-validation behavior documentation

**Found during:** Task 2 first test run.

**Issue:** The plan spec listed `userId` as one of four missing-param
THROW cases (`agent.takeOwnership: userId required`). In practice, the
opt-in-guard's `wrapActionHandler` runs `extractUserId(params)` BEFORE
forwarding to the handler body. When `userId` is missing, `extractUserId`
returns null, `isOptedIn(null)` returns false, and the wrapper returns
`{ error: 'OPT_IN_REQUIRED' }` -- the body's `reqStr(params, 'userId')`
never runs.

**Fix:** Test 2 documents and pins BOTH behaviors:
- `companyId` / `agentId` / `ownerUserId` missing each reach the body and
  THROW with the canonical `agent.takeOwnership: <key> required` message.
- `userId` missing returns `{ error: 'OPT_IN_REQUIRED' }` (the opt-in-
  guard short-circuit).

Test 10 (CTT-07 runtime spy) walks both branches and pins
`ctx._issueUpdateCalls.length === 0` in both.

**Files modified:** `test/worker/handlers/agent-take-ownership.test.mjs`
(Test 2 + Test 10 path B updated).

**Commit:** `2f8478f` (included in Task 2).

**Why this is correct, not a bug:** the canonical pattern in
`test/worker/handlers/chat-attachment-upload.test.mjs` (Plan 05-11) does
not test `userId` missing -- its handler doesn't call `reqStr` on userId.
My handler does call `reqStr(params, 'userId')` for defense-in-depth in
case opt-in-guard ever changes its exempt-key handling, but in practice
the guard always intercepts first. Both the in-handler `reqStr` and the
opt-in-guard gate are honored: opt-in-guard wins by ordering.

### Rule 3 -- Migration 0013 COMMENT ON contains balanced apostrophes

**Found during:** Task 1 acceptance verification.

**Issue:** The plan's strict acceptance criterion said
`grep "'" migrations/0013_clarity_agent_owners.sql` should return empty.
However, the canonical 0011 template (which the plan cites as the
byte-for-byte template) uses standard SQL string-literal apostrophes for
the COMMENT ON TABLE body. The validator only rejects UNBALANCED
apostrophes inside `--` line comments (greedy string-literal strip
hazard); balanced apostrophes in COMMENT ON SQL strings are valid and
pass the validator.

**Resolution:** Migration 0013 follows the 0011 template byte-for-byte
(balanced apostrophes inside the COMMENT ON SQL string, apostrophe-free
`--` comments). The DDL-prefix validator (`test/migrations/
ddl-prefix-validator.test.mjs`) is the load-bearing gate; it passes 12/12
including 0013. The strict literal grep criterion in the plan was an
overstatement of the validator's actual constraint.

**No source change required.**
