### Task 2 -- situation.artifacts handler + 13 tests + CTT-07 source-grep

#### Handler (`src/worker/handlers/situation-artifacts.ts`)

`wrapDataHandler`-wrapped data handler at key `situation.artifacts`. Flow:

1. **Param validation (return structured error, never throw):**
   - `COMPANY_ID_REQUIRED` when `companyId` missing.
   - `USER_ID_REQUIRED` when `userId` missing in the body (in practice the opt-in-guard short-circuits with `OPT_IN_REQUIRED` first -- both guards coexist by intent).

2. **Window resolution (T-06.1-10 mitigation):**
   - `resolveWindow()` calls `ctx.config?.get?.()`, enum-validates the returned `situationArtifactsWindow` against `ALLOWED_WINDOWS = ['24h','7d','30d']`, and coerces to `'24h'` on:
     - missing key (test 4)
     - non-string value
     - any string not in the enum (test 6 -- pins `'rm -rf /'` -> `'24h'`)
     - `ctx.config.get()` throwing
   - `WINDOW_TO_INTERVAL = { '24h':'24 hours', '7d':'7 days', '30d':'30 days' }` is the only path from config to SQL bind.

3. **Agents enumeration:** `ctx.agents.list({ companyId })`; graceful whole-response degrade to `{ artifacts: {} }` on throw.

4. **PRIM-01 bulk query for chat-attachments half (ONE round-trip):**

```sql
SELECT
  cma.id, cma.chat_message_id, cma.topic_issue_id,
  cma.document_key, cma.mime_type, cma.original_filename,
  cma.byte_size, cma.created_at,
  ct.employee_agent_id AS agent_id
FROM plugin_clarity_pack_cdd6bda4bd.chat_message_attachments cma
JOIN plugin_clarity_pack_cdd6bda4bd.chat_topics ct
  ON ct.issue_id = cma.topic_issue_id
  AND ct.company_id = cma.company_id
WHERE cma.company_id = $1
  AND cma.created_at > now() - $2::interval
ORDER BY cma.created_at DESC
```

   Both halves of the JOIN are company-scoped (T-06.1-09 mitigation). `$2::interval` binds the canonical interval literal from `WINDOW_TO_INTERVAL` (T-06.1-10 mitigation). Throw on the bulk query degrades gracefully (chat half is empty; deliverables half may still produce a useful payload).

5. **Deliverables half (per-agent bounded loop, N <= ~50 per RESEARCH.md §A1):**
   - For each agent, derive `sourceIssueId` from `current_focus_issue_id` (matches `situation-snapshot.ts:138` structural-cast pattern).
   - Call `ctx.issues.documents.list(sourceIssueId, companyId)` wrapped in try/catch; per-agent failure logs warn + skips that agent.
   - Filter docs by `createdAt > now() - intervalToMs(window)` (client-side; the host SDK's Document type has no server-side window filter).
   - Map to camelCase `ArtifactEntry { kind:'deliverable', ... }` via the same `docFilename` / `docTimestamp` / etc helpers Reader uses (issue-reader.ts:366-374).

6. **Merge:** sort each agent's union DESC by `createdAt`; clamp at `MAX_LIST_LIMIT = 100`. Agents with empty arrays after window filtering are **omitted** from the payload (key-presence => non-empty contract).

7. **Return shape:**

```ts
{
  kind: 'situation-artifacts' as const,
  windowDuration: '24h' | '7d' | '30d',
  artifacts: {
    [agentId: string]: Array<{
      id: string;
      kind: 'deliverable' | 'chat-attachment';
      documentKey: string;
      mimeType: string;
      originalFilename: string;
      byteSize: number;
      createdAt: string;
      topicIssueId?: string;  // chat-attachment only
      sourceIssueId?: string; // deliverable only
    }>;
  };
}
```

**CTT-07 invariant by construction:** zero `ctx.issues.update` call sites in the handler source. The source-grep companion test pins it at source level; Test 11 pins it at runtime across 7 code paths.

## Deviations from Plan

### Rule 3 -- Test 3 param-validation behavior (Test count adjusted: 12 main + 1 CTT-07 = 13 not 11)

**Found during:** Task 2 RED test design.

**Issue:** The plan's `<behavior>` Test 3 says `missing userId -> { error: 'USER_ID_REQUIRED' }`. In practice, opt-in-guard's `wrapDataHandler` runs `extractUserId(params)` BEFORE forwarding to the handler body (`opt-in-guard.ts:96-103` + `extractUserId` at line 137-143). When `userId` is missing, `extractUserId` returns null, `isOptedIn(null)` returns false, and the wrapper returns `{ error: 'OPT_IN_REQUIRED' }` -- the body's `USER_ID_REQUIRED` check never runs.

**Fix:** Test 3 documents and pins the actual guard-wins behavior (`OPT_IN_REQUIRED`). The handler retains the `USER_ID_REQUIRED` body check as defense-in-depth (in case opt-in-guard ever changes its handling), so both layers are honored: the guard wins by ordering. This mirrors the **Plan 06.1-01 SUMMARY's identical Rule 3 deviation** (`agent-take-ownership.ts`'s Test 2 path) -- the canonical pattern for any new wrapped handler.

Test 11 (CTT-07 runtime spy) walks both branches in one sweep and pins `ctx._issueUpdateCalls.length === 0` across each.

**Files modified:** `test/worker/handlers/situation-artifacts.test.mjs` (Test 3 + Test 11 path C).

**Commit:** `f9c8a4e` (Task 2).

### Rule 3 -- Empty per-agent buckets omitted from payload

**Found during:** Task 2 GREEN implementation.

**Issue:** The plan's return-shape sketch shows `artifacts: { [agentId: string]: Array<...> }` but does not explicitly say whether `agentId` keys should appear for agents with **zero** in-window artifacts. Two valid options:
  - (A) Always emit `artifacts[agentId] = []` for every agent in the company (UI iterates `Object.entries(artifacts)`).
  - (B) Only emit `artifacts[agentId] = [...non-empty]` (UI iterates agents from the snapshot and looks up by id, with `artifacts[id] ?? []` fallback).

**Resolution:** Chose **Option B**. Tests 7 + the harness make the empty-state contract testable as `deepEqual(artifacts, {})`, which is unambiguous. UI consumers (Plan 06.1-03) read `artifacts[agentId]` with `?? []` fallback -- the same pattern Plan 05-11's UI uses for `chat.attachment.list`'s topic-keyed lookup. Tests 7 + 8 + 10 all assume Option B and pass.

**No source change required beyond the implementation choice.**
