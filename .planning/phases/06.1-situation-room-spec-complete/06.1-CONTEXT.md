## Implementation Decisions

### Owner resolution at chain leaf (ROOM-09)

- **D-01:** **`clarity_agent_owners` schema = flat upsert.** Migration 0013 plugin-namespace-only additive table: `agent_id UUID PRIMARY KEY, owner_user_id UUID NOT NULL, set_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`. INSERT ... ON CONFLICT (agent_id) DO UPDATE — last write wins. Ownership history (replaced_by + audit table) explicitly deferred to v1.1+ as an additive migration; v1.0 does not need the audit trail and a 3-column table is the cheapest write path. The snapshot resolver's lookup-map build path consults this table FIRST (`SELECT agent_id, owner_user_id FROM clarity_agent_owners`), then falls back to `public.agents.owner_user_id` for any agent not in the side table. Operator chose this in /gsd:discuss-phase 6.1 over Option (ownership history append-only with replaced_by).
- **D-08 (Claude's discretion — derived):** **`agent.takeOwnership` viewer authority pattern.** Reuse Plan 04-04 `chat.promote` T-04-16 pattern: handler re-verifies viewer is opted-in + has authority to claim agents within their company, server-side. No trust on UI-passed ownership; resolve viewer's userId server-side from the request context. Migration 0013 ships with a `companyId` discriminator column (4 columns total: `agent_id, owner_user_id, set_at, company_id`) so multi-company hosts don't cross-contaminate ownership claims — matches every other plugin-namespace table's company-scoping pattern.
- **D-09 (Claude's discretion — derived):** **UI passes viewer userId via Plan 02-09 `useResolvedUserId` hook.** The Take-Ownership button dispatches `agent.takeOwnership` with `{ agentId, ownerUserId: viewerUserId }`; handler verifies `ownerUserId === resolvedViewerUserIdFromSession`. If host-bridge `userId` is null (the Plan 02-09 gap), button renders in disabled state with the same "Sign in to claim ownership" tooltip pattern that Reader uses for its userId-gated affordances.

### Artifact shelf rendering (ROOM-10)

- **D-02:** **Per-agent inline chip rows, REPLACE bottom shelf.** Delete `src/ui/surfaces/situation-room/artifacts-shipped-shelf.tsx` (the existing bottom `<section className="clarity-artifacts-shelf">`) and its mount in `situation-room/index.tsx`. Render an inline horizontal chip row (`<ArtifactChipRow>`) inside each `<AgentCard>` under the agent's "now-doing" line. Single source for ROOM-04 + ROOM-10. Empty windows render NOTHING (no placeholder copy — matches the warm-dark "degraded states always name themselves" principle from sketch-findings without using it for empty states). Operator chose this in /gsd:discuss-phase 6.1 over Option B (augment / keep bottom shelf as global firehose) and Option C (keep bottom shelf only).
- **D-05:** **Artifact window default 24h, configurable via `instanceConfigSchema`** (allowed values: 24h / 7d / 30d). Mirrors the Phase 2 D-03 pattern for the 60s/30s situation refresh cadence. The 24h default matches the mockup's "Artifacts shipped today" verbatim.
- **D-10 (Claude's discretion — derived):** **`situation.artifacts` SQL union shape.** Single bulk query joining `ctx.issues.documents.list` (filtered to `created_after = now() - window`) ∪ `plugin_clarity_pack_*.chat_message_attachments` (joined to `chat_messages` for agent attribution via `chat_messages.author_id`). Sorted newest-first. Default limit 5 per agent (clamp 100); UI renders the first 5 chips with a "+N more" affordance opening a per-agent artifact drawer (reuse the canonical popover pattern from Plan 05-11 `AttachmentChipWithPreview`). PRIM-01 single-round-trip — one bulk query covers all agents on the grid, NOT per-agent N+1.
- **D-11 (Claude's discretion — derived):** **Chip click opens `DeliverablePreview` popover.** Reuse Plan 05-04's deliverable-preview surface (single source of truth across Reader / chat / Situation Room — same primitive everywhere). Preview popover dispatch keyed on `documentKey` (existing contract).

### Create-Task entry points (ROOM-11, surface contract)

- **D-03:** **Both — standalone header `+ Create task` AND per-blocker "Convert to task".** Plan 05-08 D-17 already mounted `ClaritySurfaceHeader` with `+ Create task` on Reader / Bulletin / Chat — Situation Room simply joins the mount list (one-line change in `ClaritySurfaceHeader`'s `SURFACES_WITH_CREATE_TASK` registry or equivalent). The per-blocker affordance is NEW: each Critical Path row gets a "Convert to task →" button next to "Take ownership" (when terminal is `HUMAN_ACTION_ON.__unowned__`) or instead of it (when terminal is `CYCLE` — see D-04). Operator chose this in /gsd:discuss-phase 6.1 over Option B (standalone only) and Option C (per-blocker only).
- **D-12 (Claude's discretion — derived; operator-preference-driven):** **Per-blocker dialog NEVER pre-fills title/description/assignee.** Per project memory `feedback_trust-the-clarification-loop`: "Never propose features that pre-load operator guesses about what an agent will need (task templates, smart-prefill, scaffolding pickers). The agent+chat+comment clarification loop is designed to ASK for what's missing; front-loading scaffolding works against it." The blocker context (chain narration, terminal agent ID, blocker reason) renders BESIDE the dialog as static context — never INSIDE the dialog's editable fields. Operator types the task fresh; the agent's first reply will ask any clarifying questions.

### CYCLE terminal UI (ROOM-11, edge case)

- **D-04:** **Render cycle label as-is in v1.0; defer auto-break to v1.1+.** When `blocker-chain.ts` produces a `CYCLE` terminal (mutually-blocking agents, e.g., A → B → C → A), the Critical Path row displays "Cycle: A → B → C → A" verbatim (humanize-snapshot already rewrites the agent UUIDs to display names via the same lookup map ROOM-09 augments). Operator can break the cycle manually via host UI reassignment. v1.1+ can layer auto-suggest cycle-break logic (heuristic: oldest edge / lowest-priority edge / smallest blast-radius). Matches the Plan 04.1-04 FLAG-1 principle: "never silent, never auto-resolve." Operator chose this in /gsd:discuss-phase 6.1 over Option B (auto-suggest in v1.0).
- **D-13 (Claude's discretion — derived from taxonomy):** **CYCLE rows get `+ Create task` button only — NO Take-Ownership button.** Take-Ownership writes `clarity_agent_owners(agent_id, ...)` which presupposes a single leaf agent; CYCLE terminals don't have a single leaf. Operator's escape hatch from a cycle is to create a task ("break the A↔B blocking cycle") that they own — that's a `+ Create task` action, not a Take-Ownership action.

### Claude's Discretion (planner picks; not load-bearing)

- Migration 0013 column ordering, index choices (PK on `agent_id` is sufficient for v1.0 lookup; secondary index on `company_id` if multi-company scan becomes hot — defer until profiled).
- Whether `agent.takeOwnership` returns the new row, returns a success boolean, or returns the full updated snapshot (planner picks; recommend "returns the new row" for handler-as-data-source consistency with Plan 05-11).
- Whether the Take-Ownership button shows a confirmation modal or fires directly with a toast confirmation (planner picks; recommend fires directly + toast — matches the Plan 05-08 D-16 bulk-unarchive pattern of "no confirmation modal regardless of N"). The action is reversible (just call `agent.takeOwnership` again with a different owner) so a confirmation modal would be operator friction without safety value.
- Exact `<ArtifactChipRow>` layout/spacing within `<AgentCard>` — match the agent card's existing visual rhythm; consult `sketch-findings-clarity-pack` for the warm-dark chip styling pattern.
- Test pinning approach for the new owner-resolution behavior (runtime spy on the snapshot-builder's lookup-map construction + source-grep that `clarity_agent_owners` SELECT precedes the `public.agents` SELECT — matches Plan 04.1-05 CTT-07 pinning shape).
