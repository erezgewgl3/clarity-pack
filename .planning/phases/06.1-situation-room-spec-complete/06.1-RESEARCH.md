### Pattern 1 — Action handler (`agent.takeOwnership`)

**Source files (read in this order):**

1. `src/worker/handlers/chat-topic-archive.ts` — minimal action-handler shape (param validation, opt-in-guard wrap, repo write, structured error envelope). Closest in size and shape to `agent.takeOwnership`.
2. `src/worker/handlers/chat-promote.ts:60-119` — T-04-16 viewer-authority pattern (company-scoped read-back to verify the caller really owns the entity they're acting on). The `ctx.issues.listComments` in chat-promote scopes by `companyId`; analogous read for `agent.takeOwnership` is `ctx.agents.get(agentId, companyId)` (PluginAgentsClient line 1117 — see SDK API §below).
3. `src/worker/opt-in-guard.ts:108-124` — `wrapActionHandler` signature. The new handler MUST use `wrapActionHandler` so opted-out callers get `{ error: 'OPT_IN_REQUIRED' }` before the body runs.

**Canonical skeleton (planner hands to executor verbatim):**

```ts
// src/worker/handlers/agent-take-ownership.ts
import { wrapActionHandler, type OptInGuardActionCtx } from '../opt-in-guard.ts';
import type { PluginAgentsClient, PluginLogger } from '@paperclipai/plugin-sdk';
import { upsertClarityAgentOwner, type ClarityAgentOwnersRepoCtx } from '../db/clarity-agent-owners-repo.ts';

export type AgentTakeOwnershipCtx = OptInGuardActionCtx &
  ClarityAgentOwnersRepoCtx & {
    agents: PluginAgentsClient;
    logger?: PluginLogger;
  };

function reqStr(p: Record<string, unknown> | undefined, k: string): string {
  const v = p?.[k];
  if (typeof v === 'string' && v) return v;
  throw new Error(`agent.takeOwnership: ${k} required`);
}

export function registerAgentTakeOwnership(ctx: AgentTakeOwnershipCtx): void {
  wrapActionHandler(ctx, 'agent.takeOwnership', async (params) => {
    const companyId = reqStr(params, 'companyId');
    const agentId = reqStr(params, 'agentId');
    const ownerUserId = reqStr(params, 'ownerUserId');
    const userId = reqStr(params, 'userId'); // opt-in-guard already verified

    // D-09 — viewer-authority: ownerUserId MUST equal the request's userId.
    // No "claim on behalf of" path in v1.0.
    if (ownerUserId !== userId) {
      return { error: 'OWNER_MISMATCH' as const };
    }

    // T-04-16 — verify the agent exists in the caller's company. ctx.agents.get
    // returns null when the agentId is not in this company (or not at all).
    let agent;
    try {
      agent = await ctx.agents.get(agentId, companyId);
    } catch (e) {
      ctx.logger?.warn?.('agent.takeOwnership: agents.get failed', {
        agentId, err: (e as Error).message,
      });
      return { error: 'OWNERSHIP_FAILED' as const };
    }
    if (!agent) return { error: 'NOT_FOUND' as const };

    try {
      const row = await upsertClarityAgentOwner(ctx, {
        agent_id: agentId,
        owner_user_id: ownerUserId,
        company_id: companyId,
        set_at: new Date().toISOString(),
      });
      return { ok: true as const, agentId, ownerUserId, setAt: row.set_at };
    } catch (e) {
      ctx.logger?.warn?.('agent.takeOwnership: upsert failed', {
        agentId, err: (e as Error).message,
      });
      return { error: 'OWNERSHIP_FAILED' as const };
    }
  });
}
```

**Notes:**
- `OWNER_MISMATCH` error pattern matches Plan 04-04 `OPT_IN_REQUIRED` / `NOT_FOUND` / `PROMOTE_FAILED` envelope style.
- The handler does NOT write `public.agents.owner_user_id` — that's host-owned and additive-only protections forbid it (CTT-07 invariant by construction; CLAUDE.md coexistence #3).
- `userId` is re-extracted from params because opt-in-guard already reads it; we re-read for the authority comparison.

**The chain leaf is where `__unowned__` enters** — at `src/shared/blocker-chain.ts:147-181`. The walk WRITES `__unowned__` when the leaf node's `meta.ownerUserId` is null. The fix is to ensure the leaf node's `nodeMeta[toId].ownerUserId` is populated from the side-table BEFORE `flattenBlockerChain` runs (situation-snapshot.ts:155-160 is where `nodeMeta[toId]` is written from `blocker.assigneeUserId ?? blocker.ownerUserId ?? null`).

**Planner's load-bearing question:** Does Phase 6.1 modify `buildEmployeeRow` to inject the side-table owner into `nodeMeta[toId]` after the relations walk, OR does it pre-populate the resolved owner BEFORE the walk via the lookup map?

**Recommendation (BEFORE the walk is cleaner):** Build a `Map<agentId, ownerUserId>` from `(sideTableOwners ∪ employees.map(e => [e.id, e.owner_user_id]))` (side-table wins). Inside `buildEmployeeRow`, after the relations walk populates `nodeMeta`, do one final pass: `for (const [id, meta] of Object.entries(nodeMeta)) { if (meta.ownerUserId == null) meta.ownerUserId = ownerMap.get(id) ?? null; }`. This way the blocker-chain walk sees populated owners and emits `HUMAN_ACTION_ON(<real_user_id>)` instead of `HUMAN_ACTION_ON(__unowned__)` — no further changes to humanize-snapshot needed (the `__unowned__` rewrite pass becomes a no-op when there is no `__unowned__` terminal).

**File:** `src/worker/jobs/humanize-snapshot.ts:98-103` — modify ONLY if the planner picks "inject AFTER walk" approach. Otherwise this file is unchanged. Recommendation: don't touch it.

        calls.push({ kind: 'execute', sql, params });

        return { rowCount: 1 };
      },
    },

    _calls: calls,

## Gotchas / Landmines

1. **Coexistence check numbering** — CONTEXT.md says "10-take-ownership.mjs" but `10-uninstall-runbook.mjs` already exists. The new check is **11**-take-ownership.mjs, not 10. Confirmed via `Glob` of `scripts/coexistence-checks/*.mjs` (2026-05-26).

2. **Migration 0012 is intentionally skipped** (per STATE.md Plan 05-11 HOTFIX-2 record). The host validator rejects `UPDATE` statements with `Plugin migrations may contain DDL statements only`. Plan 05-11 HOTFIX-2's intended backfill became a manual one-shot SQL block. **Phase 6.1's migration is 0013 — do NOT try to reuse the 0012 slot.** Any `INSERT ... ON CONFLICT DO UPDATE` ships in the worker repo (`upsertClarityAgentOwner`), never in the migration.

3. **Host stamps actor as `system` for plugin-worker mutations** (project memory: rc.8 Phase B 2026-05-26 + Plan 04.1-11). This bites the chat-promote marker comment path. For `agent.takeOwnership` it does NOT matter — the side table is plugin-namespace, written by `ctx.db.execute`, not by `ctx.issues.update`. No host-stamping in the write path. **The side-table `owner_user_id` reflects whoever the worker writes; the host has no opinion.**

4. **`ctx.agents.list` returns rows from `public.agents` with snake_case fields** (situation-snapshot.ts:106-115). The structural cast `anyEmp.user_id ?? anyEmp.id` is required — the Agent type from `@paperclipai/shared` is opaque. **Same pattern for Phase 6.1**; don't try to read typed fields off Agent without a cast.

5. **`ctx.agents.get(agentId, companyId)` is company-scoped** — returns `null` if `agentId` exists but belongs to a different company. This is the natural T-04-16 viewer-authority gate; the handler's only check needs to be "ctx.agents.get returned non-null".

6. **`useResolvedUserId` returns `{ userId: null, loading: false, error: 'no-user-context' }` in the production host-bridge gap** (per Plan 02-09 + STATE.md). The Take-Ownership button MUST render disabled with the `Sign in to claim ownership` tooltip on this state — never throw, never optimistic-dispatch. UI-SPEC.md §Flow 1 Step 2 already locks this.

7. **`__unowned__` is a STRING SENTINEL, not a null** — see `blocker-chain.ts:178`. Tests must compare against the literal string `'__unowned__'`. The humanize-snapshot rewrite pass at line 98 also keys on this literal.

8. **The `NO_OWNER_ASSIGNED_RE` filter at `critical-path-strip.tsx:44`** (rc.8 final commit `1b11649`) suppresses chains whose narration matches "no owner assigned". Phase 6.1's owner-resolution should make this filter rarely fire — but the filter STAYS as a safety net (UI-SPEC.md §Edge Cases). Don't delete the filter.

9. **The `clarity-pack-1.0.0.tgz` superseded chain** — multiple tarballs ship the literal version `1.0.0`. Phase 6.1 ships NO version bump (project memory `plugin-version-bump-two-sources` + CONTEXT.md). Plan 05-10 owns the rc → 1.0.0 transition. The tarball after Phase 6.1 closure stays at `1.0.0` (Plan 05-10's atomic).

10. **No new capabilities required.** Existing manifest capabilities list (`src/manifest.ts:343-440`) already includes `agents.read`, `database.namespace.write`, `database.namespace.read`. Phase 6.1's two new handlers + one new SELECT path compose entirely from these existing grants.

11. **The 60s situation-snapshot job is gated on active-viewer count** (situation-snapshot.ts:217-231). Take-Ownership force-refetch does NOT wait for the next cron tick — the UI dispatches `usePluginData` query-key invalidation, which re-fires the read handler immediately. The snapshot recompute is async — the UI may see the new owner via the read handler in < 100ms but the materialized `situation_snapshots` row in the side table updates on the next cron tick.

12. **`instanceConfigSchema` shape** — Phase 6.1 adds ONE key (`situationArtifactsWindow`). Mirror the Plan 02-04 `situationRefreshIntervalMs` shape (manifest.ts:517-525). Allowed values via `enum: ['24h', '7d', '30d']` — JSON Schema doesn't natively prohibit other strings at runtime, so the handler should validate the value at read time and coerce to default on unknown input.
