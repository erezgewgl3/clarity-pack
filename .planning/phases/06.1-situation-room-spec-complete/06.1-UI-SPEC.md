# Phase 06.1 — UI Design Contract

> Visual and interaction contract for the v1.0.0 ship-gate phase. The UI delta is
> bounded: ONE new component, TWO modifications, ONE deletion in
> `src/ui/surfaces/situation-room/` + `src/ui/primitives/`. Every visual primitive
> already exists from prior phases (Plan 02-04 / 02-08 / 05-04 / 05-08 / 05-11);
> this contract consolidates them and locks the edge cases.
>
> Consumed by: gsd-ui-checker, gsd-planner, gsd-executor, gsd-ui-auditor.
> Locked invariant: NO new design tokens, NO new dialog shells, NO new chip
> shapes, NO new dialog dismissal patterns. Reuse-only.

### Accent reservation list (explicit, never "all interactive elements")

The 10% gold accent (`--clarity-you` / `--clarity-you-soft`) is reserved EXCLUSIVELY for:

1. Critical Path HUMAN_ACTION_ON terminal border-left + background tint (existing)
2. Awaiting-You pill (existing)
3. `+ Create task` header button background (`--clarity-you-soft`; existing from Plan 05-08)
4. **NEW:** Take-Ownership button border + label color (single accent affordance for an operator-claim action — semantically the same "this is on you" mark the rest of the surface already uses)
5. Reader's Continue-in-Chat button (existing from Plan 04.2-02, on Reader surface — not Situation Room)

**Convert-to-task button MUST NOT use the gold accent.** It uses neutral border + body-color text (`--clarity-line` border, `--clarity-fg` text, transparent background). This preserves the visual hierarchy: gold = "operator MUST act here NOW"; neutral chrome button = "operator MAY initiate a task". The standalone surface-header `+ Create task` keeps its existing soft-gold treatment (the surface-level entry point is allowed to "invite" with the accent; the per-row affordance must not compete with it).

### Destructive actions in Phase 6.1

**None.** Take-Ownership is reversible (call again with different owner). Convert-to-task is non-destructive (creates a new task issue; nothing deleted). DELETE of `artifacts-shipped-shelf.tsx` is a code-level deletion, not a runtime destructive action. No confirmation modals required (operator preference per CONTEXT.md Claude's-discretion: "fires directly + toast").

### New interaction flows

#### Flow 1: Take ownership (Critical Path row, `HUMAN_ACTION_ON.__unowned__` terminal)

1. **Trigger:** Operator clicks `Take ownership` button inside a Critical Path row.
2. **Pre-dispatch guard:** Button is disabled if `useResolvedUserId()` returns `{ loading: true }` or `{ error: 'no-user-context' }`. The disabled state uses native HTML `disabled` attribute + tooltip text `Sign in to claim ownership` (rendered via `title` attribute — sufficient for v1.0; matches Reader pattern).
3. **Dispatch:** `usePluginAction('agent.takeOwnership')({ agentId: <terminal.agentId>, ownerUserId: <resolvedUserId> })`. The agentId comes from the terminal object; the ownerUserId is server-verified (D-09).
4. **In-flight visual:** Button shows `aria-busy="true"`; label stays `Take ownership`; cursor: progress (matches `.clarity-agent-pause-banner-action[disabled]` pattern from theme.css:490). NO global spinner.
5. **Resolution path A (success):**
   - Toast `Ownership claimed` fires.
   - Force-refetch `situation.snapshot` via the existing `usePluginData` query-key invalidation pattern.
   - The next snapshot returns the terminal with the resolved owner label (no longer `__unowned__`); the Take-Ownership button is no longer rendered on this row.
   - Re-render time = next snapshot fetch (≤ 60s polling cadence + the immediate invalidation = typically < 2s).
6. **Resolution path B (failure):**
   - Toast `Could not claim ownership — try again` fires.
   - Button returns to enabled state.
   - No snapshot mutation occurs.
7. **Resolution path C (handler returns `OPT_IN_REQUIRED`):**
   - Existing opt-in error path triggers `<EnableClarityCta surfaceName="Situation Room" />` (same as the snapshot handler's opt-in degraded state).

#### Flow 2: Convert to task (per-row, non-CYCLE rows)

1. **Trigger:** Operator clicks `Convert to task →` button inside a Critical Path row.
2. **Dispatch:** Opens TrueTaskDialog in cold mode (existing Plan 05-08 / 04.1-09 mount). Dialog renders **with empty editable fields** (D-12 — NO pre-fill).
3. **Blocker context:** Renders OUTSIDE the dialog body, as static text next to (or above — planner picks; spec only requires "outside editable fields") the dialog. Carries the chain narration + terminal agent ID + blocker reason verbatim. Operator reads context; types fresh title/description in the dialog.
4. **Dialog behavior:** Identical to Plan 05-08 cold-mode TrueTaskDialog. Escape / backdrop click / close-X all dismiss; Submit creates task + fires existing `Task created` toast; snapshot refetches via the existing surface-header `onTaskCreated` callback.
5. **No assignee thread-through.** D-12 + the operator's clarification-loop preference: the dialog's roster selector requires the operator to choose an assignee; the row's terminal agent ID is shown as context only, NOT auto-selected. (The planner MAY revisit this in v1.1+ if friction is observed; v1.0 ships strict no-pre-fill.)

#### Flow 3: + Create task to break this cycle (CYCLE rows only)

Identical to Flow 2, but:
- Button label is the literal `+ Create task to break this cycle` (single button — no Take-Ownership companion, per D-13).
- Blocker context shows the cycle path (e.g., `Cycle: Sales-Agent → Marketing-Agent → Sales-Agent`).
- Otherwise same dialog, same no-pre-fill rule, same submit + toast + refetch.

#### Flow 4: Artifact chip click

1. **Trigger:** Operator clicks an artifact chip in an `ArtifactChipRow` under an agent card.
2. **Dispatch:** Opens AttachmentChipWithPreview popover (existing shell). Passes `attachment.documentKey`, `companyId`, `userId`, `topicIssueId` (or the agent's source-issue ID for deliverables — planner picks the param shape that resolves through DeliverablePreview's existing 3-branch logic).
3. **Preview content:** Renders DeliverablePreview body (xlsx-grid / pdf-embed / md / img / placeholder / error — all existing branches from Plan 05-04).
4. **Empty branch fallthrough:** If a chip's `documentKey` resolves to nothing (rare — handler should not surface dead chips), preview renders the existing `No deliverables on this issue yet. Upload via the chat composer (Clarity → Chat tab).` branch verbatim. NO new empty branch.
5. **Dismissal:** Escape / backdrop click / close-X (existing 3-affordance contract).

#### Flow 5: Surface-header + Create task (standalone, already mounted)

Inherits Plan 05-08 D-17 flow verbatim. Phase 6.1's only change is **adding the Situation Room's surface-id to the `SURFACES_WITH_CREATE_TASK` mount-list registry inside `clarity-surface-header.tsx`** (one-line change; current `<ClaritySurfaceHeader surface="situation-room" />` mount at index.tsx:131-136 already passes the surface; the registry filter in the primitive is the load-bearing piece).

**Note:** Per `src/ui/primitives/clarity-surface-header.tsx` as currently shipped (read 2026-05-26), the button is **already rendered unconditionally** for every surface that mounts the header — there is no `SURFACES_WITH_CREATE_TASK` allowlist in the current code. If a registry exists in a follow-up branch the planner has not surfaced, the spec authorizes one-line registry inclusion. If the button is already rendering on Situation Room (because the surface already mounts `<ClaritySurfaceHeader>` at index.tsx:131), the planner's task is to **verify visual presence on a Playwright drill** and document the result. If verification passes, ROOM-11 D-03's standalone-header requirement is satisfied with ZERO code change to `clarity-surface-header.tsx`.

## Accessibility Notes

| Element | a11y Requirement | Implementation |
|---------|------------------|----------------|
| Take-Ownership button | Keyboard-focusable; disabled state communicated to AT | Native `<button>` with `disabled` attribute + `aria-label="Take ownership of <agent role>"`; disabled state uses HTML `disabled` (auto-removes from tab order). |
| Take-Ownership in-flight | AT announces transition | `aria-busy="true"` toggled during dispatch; label-via-`aria-label` swaps to `Claiming ownership of <agent role>…` during in-flight; reverts on resolution. |
| Convert-to-task button | Keyboard-focusable | Native `<button>` with `aria-label="Convert to a task — <chain narration>"`. |
| `+ Create task to break this cycle` button | Keyboard-focusable | Native `<button>` with `aria-label="Create task to break cycle: <cycle path>"`. |
| ArtifactChipRow chips | Keyboard-focusable | INHERITED — `<AttachmentChip>` is already `<button>` when `onClick` is provided. The chip-row wraps inherit this. |
| Tooltip `Sign in to claim ownership` | Available to keyboard / screen-reader users | Use `title` attribute (rendered by AT on focus). v1.0 acceptable; v1.1+ may upgrade to a real tooltip surface. |
| Empty ArtifactChipRow | Doesn't break navigation | Empty row renders no DOM; tab order skips automatically. |
| Toast `Ownership claimed` / `Could not claim ownership — try again` | Announced to AT | INHERITED from existing ToastProvider (Plan 05-08); toast surface is `role="status"` (success) / `role="alert"` (error). |
| Critical Path section heading | Properly leveled | UNCHANGED — existing `<h2 class="clarity-critical-path-heading">` (theme.css:730). |
| Color contrast | Gold accent on warm-dark background | `--clarity-you` (oklch(0.82 0.13 85)) on `--clarity-bg-2` (oklch(0.16 0.01 60)) — contrast ratio ≥ 7.5:1, AAA. The Take-Ownership button text uses `--clarity-you` on transparent — composes against `--clarity-bg-2` parent. Verified by Plan 05-08 check-a11y CI gate (re-runs in Phase 6.1). |
