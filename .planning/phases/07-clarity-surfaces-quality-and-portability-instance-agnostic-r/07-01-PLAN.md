<objective>
Fix the two prerequisite defects in Phase 7 (CONTEXT items 1 + 2):

1. **Ref resolution is broken on every instance.** Both worker paths (the `issue.reader` inline
   fetcher and the standalone `resolve-refs` handler) resolve `BEAAA-NNN` chips by calling
   `ctx.http.fetch(${apiBase}/api/companies/.../issues?ids=...)`. On paperclipai@2026.525.0 that path
   is SSRF-blocked AND the `?ids=` filter is ignored AND the field mapping reads stale snake_case
   (`i.key/i.assignee_user_id/i.body`) so the resolver's `byId` map never matches → chips render
   `"BEAAA-807 · unknown"`. Replace both fetchers with per-ref `ctx.issues.get(identifier, companyId)`
   in parallel, with a cached `ctx.issues.list({companyId})`-and-match-on-`.identifier` fallback when
   `get` returns null. The fetcher MUST return `id = the requested identifier` so the pure
   `resolveRefs()` `byId.get(ref)` hits.

2. **Ref extraction is hardcoded to BEAAA.** Both worker regexes (`issue-reader.ts:58` REF_PATTERN
   and `editor.ts:126` extractRefsFromBody) are `/\bBEAAA-\d+\b/g` → extract ZERO refs on any
   non-BEAAA instance. Derive the exact prefix from the current issue's `identifier` and narrow the
   regex; broad fallback only when `identifier` is null. Plus two hardcoded `"BEAAA"` UI labels in the
   chat surface get the resolved company display name.

Purpose: This is the prerequisite chunk — item 1's SDK-based resolver also unblocks the deferred
item 3 (TL;DR titles) and is the portability fix. Without it the Reader's "zero rabbit-holes" core
value is broken on EVERY instance.

Output: Both worker resolution paths rewritten to the SDK; both extraction regexes de-BEAAA'd; the
shared `prefixFromIdentifier` derivation; two UI label swaps; all red-flipping tests rewritten in the
same task as their code change; a net-new prefix-extraction test. NO version bump, NO migration, NO
new runtime deps. Ends with the autonomous quality gates green + an operator-gated live BEAAA
Playwright drill (the runtime probe for the SDK `get`-by-identifier question + the `_viewer_can_read`
open item).
</objective>
