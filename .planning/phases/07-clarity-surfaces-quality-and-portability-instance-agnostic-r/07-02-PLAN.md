## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host issue body/title → worker → TL;DR body string → UI SafeMarkdown | Untrusted issue prose (the TL;DR is an LLM summary of an untrusted issue body; the excerpt is a verbatim untrusted issue body) flows into a markdown renderer rendered same-origin in the host page. |
| LLM compile output → TL;DR body | The Editor-Agent's compiled body is model output (effectively untrusted text) that the renderer parses. |
| ref token → resolveRefsViaSdk → title → rewritten TL;DR body | A regex-extracted `<PREFIX>-NNN` token becomes a `ctx.issues.get` argument; the resolved title is spliced back into the body string. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-02-XSS | Tampering / Elevation | SafeMarkdown rendering of untrusted markdown in the same-origin host page | mitigate | The renderer emits React element nodes ONLY — NEVER `dangerouslySetInnerHTML` (so literal `<...>` is escaped to inert text by React) — AND applies an href allowlist (`http`/`https`/`mailto`/relative) so a `javascript:`/`data:`/`vbscript:` URL is never emitted as a live link. Pinned by the Task-1 XSS-guard tests (both the no-script-element and no-javascript-href cases) + a source-scan that the module has no dangerouslySetInnerHTML. This is the load-bearing threat: same-origin trusted UI means an HTML-injection here would run in the host's origin. |
| T-07-02-IL | Information Disclosure | TL;DR excerpt / refs→title rewrite | accept | The refs→title rewrite only inlines a title the 07-01 resolver already returns (and already renders in the Anchored-to ref-card). No new data crosses a boundary that 07-01 didn't already cross; the PRIM-02 / `_viewer_can_read` posture is inherited unchanged from 07-01 (T-07-01 there). |
| T-07-02-DoS | Denial of Service | markdown parse over a long TL;DR body + per-render parse | accept | The TL;DR body is length-capped by the tightened prompt (D-I3-03 output cap) + MAX_TOKENS input cap; the parser is linear (block split on blank lines, then bounded inline passes, no nested backtracking quantifiers). The refs→title token extraction reuses the 07-01 linear prefix-narrowed regex. |
| T-07-02-RW | Tampering | refs→title rewrite corrupting the body | mitigate | The rewrite is idempotent (a token already followed by ` — title` is not re-rewritten), degrades unresolved tokens to the bare ID, never throws, and is wrapped in a try/catch in the reader handler so a failure leaves the un-rewritten body. Pinned by the Task-2 idempotency + degrade-safe tests. |
| T-07-02-SC | Tampering | npm/pip/cargo installs | mitigate | NO new runtime deps in this plan (CONTEXT constraint — the markdown renderer is hand-rolled, plugin-local). slopcheck N/A — no package installs. If any install is attempted, STOP: this plan forbids new deps and Task 4's acceptance criteria assert no new `dependencies` entry. |
