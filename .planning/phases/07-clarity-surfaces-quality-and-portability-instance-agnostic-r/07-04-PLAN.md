<objective>
Phase 7 ITEM 3.1 ‚Äî Unified ref-aware markdown rendering (operator feedback on item 3).

After item 3 (07-02) shipped, the operator reviewed BEAAA-828 live and reported: "The TLDR looks
perfect. The rest of the reader looks half rendered‚Ä¶ still asterisks, and still BEAAA-704 etc. that
do not show the title. The way things are rendered in the TLDR should be rendered in the rest of the
reader as well." Asked how in-prose refs should look: "a clickable chip with a title, but I also want
the same behavior in TLDR."

Root cause: the main Reader prose body is rendered by `prose-with-ref-chips.tsx`, which (a) renders
text segments as PLAIN text (literal `## BLUF`, `**bold**`, `-` bullets) and (b) renders refs as a
`RefChip` showing `ID ¬∑ status` (never the title). Item 3 only applied SafeMarkdown to the TL;DR strip
and the Anchored-to excerpt ‚Äî the main prose body was in NO item's render scope, and the chip never
showed a title anywhere.

This plan delivers ONE unified ref-aware markdown renderer used everywhere:

1. **D-I31-01 ‚Äî RefChip shows `ID ‚Äî title`, clickable, status as a small badge.** `ref-chip.tsx`
   already resolves `{id, title, status}` via the `resolve-refs` handler; change its render from
   `ID ¬∑ status` to `ID ‚Äî <title>` (keeping the clickable anchor to `/<prefix>/issues/<id>` and the
   hover-peek), with status as a small badge/dot. Degrade to the bare ID when the title is unresolved.
2. **D-I31-02 ‚Äî SafeMarkdown becomes opt-in ref-aware.** Add a `linkRefs` + `companyPrefix` mode: during
   inline parse, a `PREFIX-NNN` token becomes a `ref` span the `.tsx` renders as `<RefChip refId=‚Ä¶ />`
   instead of a plain-text node. Instance-agnostic prefix; keyed nodes; XSS guards UNCHANGED. Without
   the prop ‚Üí no chips (back-compat).
3. **D-I31-03 ‚Äî BOTH the TL;DR strip AND the main prose body AND the excerpt render via ref-aware
   SafeMarkdown.** `prose-with-ref-chips.tsx` is REWRITTEN to delegate to ref-aware SafeMarkdown;
   `tldr-strip.tsx` and `ref-card.tsx` enable ref-awareness on their SafeMarkdown.
4. **D-I31-04 ‚Äî SUPERSEDE the item-3 worker TL;DR text rewrite.** Client-side titled chips make the
   worker `tldr-ref-titles.ts` rewrite (body ‚Üí "ID ‚Äî title" TEXT) redundant ‚Äî and double-rendering
   ("ID ‚Äî title" chip + trailing " ‚Äî title" text). REMOVE the wire-in from `issue-reader.ts`, DELETE
   the `tldr-ref-titles.ts` module + its test, and revert the item-3 integration assertions in
   `issue-reader.test.mjs` (the TL;DR body is raw again; the chip supplies the title). READER-10 stays
   Implemented (mechanism ‚Üí client-side, clickable ‚Äî strictly better).

Purpose: close the operator-perceived "half-rendered reader" gap. The prose body and every in-prose ref
look identical to the TL;DR ‚Äî formatted markdown + clickable titled chips ‚Äî everywhere.

Output: a ref-aware SafeMarkdown (pure `.ts` token splitter + `.tsx` RefChip mapping, with the XSS
allowlist intact); a retitled RefChip (`ID ‚Äî title`); a rewritten `prose-with-ref-chips.tsx`
delegating to it; tldr-strip + ref-card enabling ref-awareness; the worker rewrite REMOVED with its
tests deleted/updated (no double-render); all red-flipping tests rewritten in the same task as their
code change. NO version bump, NO migration, NO new runtime deps. Ends with the autonomous quality gates
green + the tarball packed; the orchestrator runs the live BEAAA drill on BEAAA-828 after the build.
