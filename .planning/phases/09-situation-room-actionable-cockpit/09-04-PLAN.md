<root_cause_recap>
`situation-assign-owner.ts:107` calls `ctx.issues.update(leafIssueId, patch, companyId, actor)` where `leafIssueId` is the HUMAN identifier (`BEAAA-43`). The host's `issues.update` expects the issue UUID (cf. the proven reference `editor.ts:663` Ôćĺ `ctx.issues.update(operationIssueId, { status: 'done' }, companyId)` where `operationIssueId` is a UUID). `leafIssueId` is human-readable BY DESIGN (`build-employees-rollup.ts:311-326`, M2 / NO_UUID_LEAK ÔÇö resolves to `leaf.identifier ?? focusIssue.identifier`). Display-id and mutation-id were conflated. Unit tests never caught it because the fakes accept any id string. v1.3.0 is the first live core-issue mutation.

The UUID is ALREADY in hand at the build site: `leafNodeId = picked.pathIds[picked.pathIds.length - 1]` (`build-employees-rollup.ts:314`) and `leaf.id` from the `ctx.issues.get(leafNodeId)` fetch at ~line 317; fall back to `focusIssue.id`. We add it as a sibling field, never derive it from `.identifier`.
</root_cause_recap>
