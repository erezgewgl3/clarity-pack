Close the single blocking R3 gap from the live v1.3.0 BEAAA drill: `situation.assignOwner` passed the **human** issue key (`BEAAA-43`) to `ctx.issues.update`, which needs the issue **UUID** → host rejected → `ASSIGN_FAILED`. The fix carries the UUID separately from the display key end-to-end: add `leafIssueUuid` to the worker rollup's `blockerChain` + `NeedsYou.topAction` (sourced from the leaf node UUID already in hand), add an OPTIONAL `leafIssueUuid?` prop to the **shared** `owner-picker-popover.tsx` so it ALWAYS dispatches `leafIssueUuid: leafIssueUuid ?? leafIssueId` alongside the human `leafIssueId`, thread the UUID through the two surfaces that mount the picker (`employee-row.tsx`, `needs-you-banner.tsx`), and have `situation.assignOwner` read `reqStr(params,'leafIssueUuid')` and call `ctx.issues.update` with that UUID while still logging the human key. **Why the popover (not just the row) is in scope:** the shared popover is the SINGLE dispatch site — it builds the `assignOwner({...})` object. The popover must dispatch the new key, with a `?? leafIssueId` fallback so the proven org-backlog mount (which passes only `leafIssueId={row.issueId}`, already a UUID) keeps working unchanged.

<root_cause_recap>
`situation-assign-owner.ts:107` calls `ctx.issues.update(leafIssueId, patch, companyId, actor)` where `leafIssueId` is the HUMAN identifier (`BEAAA-43`). The host's `issues.update` expects the issue UUID (cf. the proven reference `editor.ts:663` → `ctx.issues.update(operationIssueId, { status: 'done' }, companyId)` where `operationIssueId` is a UUID). `leafIssueId` is human-readable BY DESIGN (`build-employees-rollup.ts:311-326`, M2 / NO_UUID_LEAK — resolves to `leaf.identifier ?? focusIssue.identifier`). Display-id and mutation-id were conflated. Unit tests never caught it because the fakes accept any id string AND because they drive the handler directly (bypassing the popover dispatch). v1.3.0 is the first live core-issue mutation.

The UUID is ALREADY in hand at the build site: `leafNodeId = picked.pathIds[picked.pathIds.length - 1]` (`build-employees-rollup.ts:314`) and `leaf.id` from the `ctx.issues.get(leafNodeId)` fetch at ~line 317; fall back to `focusIssue.id`. We add it as a sibling field, never derive it from `.identifier`.

**The integration gap the checker caught:** the shared `owner-picker-popover.tsx` dispatches `assignOwner({ companyId, leafIssueId, userId, ...params })`. If the handler reads a NEW key `leafIssueUuid` but no task makes the popover DISPATCH it, the handler gets `undefined` → `reqStr` throws → `ASSIGN_FAILED`. The popover MUST dispatch the new key. The fix: add an optional `leafIssueUuid?` prop and always send `leafIssueUuid: leafIssueUuid ?? leafIssueId`; the `?? leafIssueId` fallback covers the backlog mount (which already passes a UUID via `leafIssueId`).
</root_cause_recap>
