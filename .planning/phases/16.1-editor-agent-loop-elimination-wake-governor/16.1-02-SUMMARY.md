# Phase 16.1 Plan 02: Loop-Break + Throughput Governor Summary

Severed the wake from Clarity's delivery path and added the throughput cap. Two structural changes land the loop fix: (1) `agent-task-delivery.ts` now writes durable own-operation provenance the moment it creates OR reuses an op-issue and the fire-and-forget `requestWakeup` block is gone entirely; (2) new `wake-governor.ts` bounds wakes by trailing-60s throughput against an env ceiling (default 6/min) and engages the restart-durable kill-switch on overflow. After this plan no Clarity write can wake an agent from the delivery path, and any legitimate wake (Plan 04's heartbeat/cron pull) is rate-bounded with an operator-clear-only kill-switch.

## What Was Built

- **`wake-governor.ts`** (D-06/D-07/D-08) ‚Äî `checkAndRecordWake(ctx, companyId)`: reads `CLARITY_WAKE_CEILING_PER_MIN` (Number-coerced, safe default 6, malformed ‚Üí default); short-circuits `false` when `isEngaged` (durable restart backstop); else `appendWake` + `pruneOldWakes(120)`, then `countTrailingWakes(60)` ‚Äî if `rate > ceiling` it `engage()`s the kill-switch, logs warn, returns `false`; otherwise returns `true`. Mirrors `circuit-breaker.ts` governance parity: engage-once, never auto-clear, narrow Pick ctx, fail-open durable read (inside the repo). **Gates only ‚Äî never touches `ctx.agents.*`** (comment-stripped source has 0 `ctx.agents` call sites).
- **`agent-task-delivery.ts`** (D-03/D-04/D-05) ‚Äî imported `recordOwnOperationIssue`; added `await recordOwnOperationIssue(ctx, opts.companyId, issue.id)` beside the in-memory `rememberOwnOperationIssue` (after `issue` is assigned, so it covers BOTH the create branch and the reuse branch); the in-memory call stays as a non-authoritative fast-path cache. **Deleted the entire fire-and-forget `requestWakeup` block** and dropped `requestWakeup` from `AgentTaskDeliveryCtx`; added `db: PluginDatabaseClient` to that ctx for the durable write.

**2. [Rule 2 - Critical] Malformed-override fallback on the ceiling read**
- **Found during:** Task 1
- **Issue:** The plan specifies `Number(env) || default 6`, but `Number('abc')` is `NaN` (falsy ‚Üí default, OK) while a hostile/typo override like `0` or `-1` would, with a naive `>` compare, effectively disable the cap (every rate `> 0` or `> -1` ‚Ä¶ actually `0` makes everything trip, `-1` worse) ‚Äî either way a non-positive ceiling is a foot-gun against the very DoS this plan exists to stop.
- **Fix:** `readCeiling()` guards `!Number.isFinite(n) || n <= 0` ‚Üí `DEFAULT_WAKE_CEILING_PER_MIN`. A malformed or non-positive override can never silently change the cap behavior.
- **Files modified:** src/worker/agents/wake-governor.ts
- **Commit:** 009d9af
