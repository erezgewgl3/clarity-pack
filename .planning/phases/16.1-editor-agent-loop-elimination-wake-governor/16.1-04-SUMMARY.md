# Phase 16.1 Plan 16.1-04: Bounded Warm-on-Heartbeat + Scoped/Governed Bulletin Cron Summary

Scheduled proactivity is preserved under the new pull model — now PULL-based, SCOPED, SWR-served, and GOVERNED. The heartbeat (the compile-bulletin scheduled job, the sole native pull path after 16.1-03 disposition (b)) now warms at most N=5 SWR-stale awaiting-you TL;DRs per opted-in company, with staleness read against the correct per-row `tldr-cache` (A2) and each warm gated by the wake-governor; and the daily bulletin cron passes through the SAME opt-in scope gate + wake-governor (at its genuine wake origin) without changing its cadence. As a side effect of wiring the governor's caller, the 5 `wake-governor.ts` TS2345 errors carried forward from 16.1-03 are resolved (typecheck is now 0 errors).

## What Was Built

- **`awaiting-you-selector.ts`** (D-10) — `selectAwaitingYouIssueIds(rows): string[]`, a PURE filter (0 `ctx.` references) that returns the issue UUIDs of rollup rows whose `blockerChain.needsYou === true`, resolving `targetIssueUuid ?? leafIssueUuid`, de-duped, stable order. No DB/ctx/host call — the heartbeat supplies the rows and does the staleness/governor work.
- **`editor.ts` bounded warm** (D-10/D-11/LOOP-03/LOOP-04) —
  - `runBoundedWarm(ctx, companyId, rows, warmCompile, now?)`: selects awaiting-you ids, reads the **per-row `tldr-cache`** (`getTldrByScope`, A2) to skip FRESH rows (`isTldrFresh` against `WARM_FRESHNESS_WINDOW_MS`), takes at most `readWarmMaxRows()` STALE rows (`CLARITY_WARM_MAX_ROWS`, safe default 5), and calls `checkAndRecordWake(ctx, companyId)` BEFORE each warm (a false return — over-ceiling / kill-switch — skips that warm). The `warmCompile` seam creates the op-issue the agent pulls; **no `requestWakeup`**.
  - `runHeartbeatBoundedWarm(ctx, companyId, warmCompile?)`: the heartbeat entry — `ensureSeeded` + `isCompanyOptedIn` scope gate first (D-13), then builds the rollup (viewer-agnostic) and runs the bounded warm; best-effort (never throws). The production warm-compile seam (`defaultWarmCompile`) drives one `driveTldrCompileStep`.
- **`compile-bulletin.ts` cron gating** (LOOP-03/LOOP-04/D-13) — the per-company loop now `ensureSeeded`s once, **skips companies where `isCompanyOptedIn` is false**, passes a `wakeGovernor` option into `compileBulletinForCompany` (gated at the START dispatch right before `startAgentTask` — the genuine wake origin), and runs `runHeartbeatBoundedWarm` per opted-in company. The bulletin **cadence (cron `0 6 30 * * *`) is unchanged**; a throttled tick returns `{kind:'skipped'}` leaving the schedule pointer for retry.
- **`wake-governor.ts`** — `WakeGovernorCtx.db` widened to `PluginDatabaseClient` and `logger` to the `PluginLogger` `meta?: Record<string,unknown>` shape (the latter surfaced once the full ctx was required). This is the carried-forward typecheck fix.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Governor placement: gate the START dispatch, not the whole per-company call**
- **Found during:** Task 2
- **Issue:** The first wiring called `checkAndRecordWake` unconditionally before `compileBulletinForCompany` every cron tick. Because `appendWake` writes an `INSERT INTO wake_ledger` and the every-minute cron mostly returns not-due/pending/bootstrap (no op-issue created), this recorded a phantom wake every minute — it broke `compile-bulletin-noop.test.mjs` (no-op-when-not-due expected 0 INSERTs, got 1; bootstrap expected 1 INSERT, got 2) and is semantically wrong (a wake recorded when nothing woke would falsely consume the throughput ceiling).
- **Fix:** Inject `wakeGovernor` as an option into `compileBulletinForCompany`; call it ONCE right before the START dispatch (`startAgentTask`) — the genuine wake origin. A false return returns `{kind:'skipped'}` and leaves the schedule pointer untouched for retry. force/on-demand callers omit the option → ungated (unchanged behavior).
- **Files modified:** src/worker/jobs/compile-bulletin.ts
- **Commit:** 176e1ea

**2. [Rule 1 - Bug] Bulletin cron tests skipped by the new opt-in scope gate**
- **Found during:** Task 2
- **Issue:** Adding `isCompanyOptedIn` to the cron loop made every `registerCompileBulletinJob`-driving test (e2e, host-faithful, content-defects, noop) skip its company — their fake `db.query` returned `[]` for the seed queries, so no company was ever opted in (same class of break Plan 16.1-03 hit with chat-stream-bridge).
- **Fix:** Seeded each ctx (and the shared `host-faithful-ctx.mjs` helper) with scripted `clarity_user_prefs` + `clarity_agent_owners` rows mapping the test companies to opted-in, and added `test.beforeEach(invalidateOptedInCache)` so the module-level set re-seeds per test rather than serving the 60s TTL stale.
- **Files modified:** test/helpers/host-faithful-ctx.mjs, test/worker/bulletin/{compile-bulletin-end-to-end,compile-bulletin-host-faithful,compile-bulletin-noop,bulletin-content-defects}.test.mjs
- **Commit:** 176e1ea

**3. [Rule 3 - Blocking / carried-forward] wake-governor TS2345 (logger contravariance surfaced after the db widen)**
- **Found during:** Task 2 (wiring the governor caller, per the carry-forward instruction)
- **Issue:** Widening `WakeGovernorCtx.db` to `PluginDatabaseClient` (the documented carry-forward fix for the 5 TS2345) exposed a second incompatibility: `WakeGovernorCtx.logger` used `meta?: unknown`, contravariantly incompatible with the host `PluginLogger`'s `meta?: Record<string, unknown>` when `CompileBulletinCtx` was passed in.
- **Fix:** Widened the governor logger to the `PluginLogger` `meta?: Record<string, unknown>` shape. `npx tsc --noEmit` now reports 0 errors total.
- **Files modified:** src/worker/agents/wake-governor.ts
- **Commit:** 176e1ea
