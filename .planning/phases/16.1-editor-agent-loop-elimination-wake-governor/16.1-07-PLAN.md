<objective>
Close LOOP-07. The 16.1 storm fix (Plan 16.1-02 / D-05) DELETED the `requestWakeup` call from the delivery path on the assumption the Editor-Agent's native heartbeat would pull op-issues. It does not — op-issue assignments sit undispatched and fall to Paperclip's periodic **recovery sweep**, which dispatches them under `recoveryAssigneeAdapterOverrides("status_only")` (`modelProfile:cheap`, `allowDocumentUpdates:false`, `resumeRequiresNormalModel:true`). The host's `routes/issues.js` (`assertDeliverableMutationAllowedByRunContext` / `isStatusOnlyCheapRecoveryContext`) then hard-rejects (403/422) any document/deliverable write from such runs. Result: the Editor-Agent computes CORRECT TL;DRs but can NEVER persist them — the entire plain-English layer is non-functional live and surfaces fall back to raw task numbers.

Re-introduce a SINGLE GOVERNED `requestWakeup` at op-issue CREATION in `startAgentTask` (right after `recordOwnOperationIssue` / `ctx.issues.create`), gated through `checkAndRecordWake` (the wake-governor 16.1 already shipped). This restores prompt **normal_model** (write-capable) dispatch via `issue-assignment-wakeup` so TL;DRs persist.

Purpose: Make the plain-English layer functional live (the phase goal) WITHOUT reopening the storm.
Output: Governed wake at op-issue creation; governed-wake + kill-switch-degrade tests; storm-safety regression confirmed; two-source version bump 1.5.0 -> 1.5.1.

D-05 over-corrected — it deleted the wake entirely instead of governing it. The governor (16.1-02) was built precisely to make a bounded wake safe; this plan uses it.

<storm_severance_invariant>
The storm mechanism is: event-ingress -> wake -> agent write -> re-enters ingress -> wake (recursion).
