<domain>
## Phase Boundary

Worker-tier re-architecture to make Clarity Pack safe to run on the live Paperclip instance. Convert host-event ingress to observe-only (no agent wakeups from event handlers), break the self-feedback loop by construction with restart-durable provenance, add a durable env-configurable throughput wake-governor + kill-switch, and gate event ingress (and the bulletin/situation crons) on opt-in / active-company scope — while leaving the read-time "zero rabbit-holes" guarantee (resolve-refs / blocker-chain-flatten / deliverable-preview) fully untouched.

No new user-facing surfaces. Action-cards stays gated OFF (its re-enable remains Phase 19). Scope is fixed by 16.1-SPEC.md; this discussion locks only HOW.

<decisions>
## Implementation Decisions

### Dispatch architecture & loop-break (LOOP-01, LOOP-02)
- **D-01:** Op-issues remain the durable work queue. The agent's own heartbeat/cron picks them up via **native pull** — there is no `requestWakeup` and no event-reactive op-issue creation. (Op-issues already function this way: the next heartbeat picks the operation issue up regardless — see `agent-task-delivery.ts:451` comment.)
- **D-02:** Event handlers become **observe-only**: they may write a durable dirty-marker and nothing else. No `ctx.agents.*` wake / `requestWakeup` / dispatch is reachable from any `ctx.events.on` handler (statically enforced per LOOP-01 acceptance).
- **D-03:** Recursion is broken by a **durable own-operation provenance table** in the plugin namespace (working name `own_operation_issues`, columns ~ `company_id`, `issue_id`, `created_at`, with `UNIQUE (company_id, issue_id)` as the idempotency/lookup key — same shape pattern as `reply_resume_dedup`). Every op-issue id Clarity creates is recorded here.
- **D-04:** The provenance table is checked at **ingress, before any host call** (replacing the in-memory `OwnOperationIssueSet` as the authoritative guard). The in-memory set may remain as a pure fast-path cache on top, but is NEVER the sole guard — the durable table is the restart backstop (fixes the empty-on-restart onset condition in `op-issue-set.ts:19-23`).
- **D-05:** The existing fire-and-forget `requestWakeup` call in the delivery path is **removed entirely** — native heartbeat pull is the only dispatch mechanism (it is already documented as unreliable on this host and redundant).

### Throughput governor (LOOP-03)
- **D-06:** Rate limiting is a **sliding-window wake ledger**: append one row per wake attempt (`ts`, `company_id`) to a plugin-namespace table; the count of trailing-60s rows is the rate. Excess-over-ceiling wakes are suppressed.
- **D-07:** Ceiling is read from env var **`CLARITY_WAKE_CEILING_PER_MIN`**, conservative default **6/min** — safe out of the box with no config required.
- **D-08:** Kill-switch is a **durable row** that engages when the rate exceeds the ceiling and **stays engaged until an operator manually clears it** — governance parity with the failure circuit-breaker's manual-Resume (coexistence #4). Fail-safe is cheap here: a tripped kill-switch degrades only proactive TL;DR freshness (stale prose), because the read-time path is untouched (LOOP-06). Engaged state persists across worker restart (per LOOP-03 acceptance).
- **D-09:** Both the durable provenance table and the governor ledger/kill-switch ship in a single **additive plugin-namespace migration** (next number `0017_*`), following the `0016_reply_resume_dedup.sql` host-validator rules (fully-qualified `plugin_clarity_pack_cdd6bda4bd.` names, `CREATE TABLE IF NOT EXISTS`, inline `UNIQUE` not standalone index, apostrophe-free `COMMENT ON`). No `public.*` DDL.

### Warm-on-heartbeat scope (bounded proactive compile)
- **D-10:** The bounded warm compiles **awaiting-you rows only**, per opted-in company.
- **D-11:** Capped at **N=5 rows per heartbeat** (env-tunable), and only entries that are **SWR-stale** are warmed — fresh cache hits are skipped. This is the only push→pull-surviving proactive compile besides the daily bulletin cron.

### Opt-in / active-company ingress gate (LOOP-04)
- **D-12:** Ingress scope is decided against a **boot-seeded in-memory opted-in-company set**, seeded once at worker boot from `clarity_user_prefs` (plugin namespace, migration `0003`), refreshed on opt-in-change events plus a periodic TTL re-read. Zero host/DB call per event.
- **D-13:** Every event handler short-circuits when `event.companyId` is not in the opted-in set, **before** any reconcile/get/enqueue (per LOOP-04 acceptance). The same scope + governor gating wraps the bulletin and `*/1` situation cron paths — not only event ingress — so "default OFF for existing users" actually throttles the worker.

### Claude's Discretion
- Exact column types / indexes / TTL-cleanup mechanism for the provenance and ledger tables (follow the `reply_resume_dedup` / `editor_agent_failures` precedents).
- Ledger pruning strategy (e.g. delete rows older than the window) so the table self-drains.
- Precise refresh TTL for the opted-in-company set.
- The storm-safety CI test harness shape — its required behavior is already locked in LOOP-05 (≥100-event burst incl. own op-issue + agent result writes, in-memory state cleared mid-burst, assert wakes ≤ ceiling/min AND zero self-trigger recursion).
- Whether the in-memory `OwnOperationIssueSet` is retained as a fast-path cache or removed (D-04 makes it non-authoritative either way).
</decisions>

<specifics>
## Specific Ideas

- Strategy is **eliminate-the-loop-by-construction**, not mitigate-with-filters — this storm class recurred across v0.6.5, v1.4.1/BEAAA-2092, the bulletin 2-min runaway, and v1.4.4 despite repeated filter hardening. Durable provenance + observe-only ingress + throughput governor are the structural mechanisms, not another filter layer.
- "Clarity is the trigger; the employee agents are the fuel" — the Clarity worker itself is ~0% CPU; the cost is each wake spinning a downstream LLM agent. Bounding wakes is bounding the fuel.
- Fail-safe posture is acceptable because the read path is independent: even fully tripped, reads still resolve inline. The kill-switch protects the box; it never breaks "zero rabbit-holes."
</specifics>

## Existing Code Insights

### Reusable Assets
- **Op-issues as a durable queue**: `ctx.issues.create({ originKind, originId, surfaceVisibility:'plugin_operation' })` already produces a durable, agent-pulled work item — keep it as the queue (D-01).
- **Additive durable-table pattern**: `reply_resume_dedup` / `editor_agent_failures` give a proven, host-validator-legal template (fully-qualified namespace, inline UNIQUE, ON CONFLICT idempotency) for the provenance + governor tables.
- **Manual-Resume breaker**: `circuit-breaker.ts` is the governance-parity precedent for the durable kill-switch (engage once, operator clears).
- **Phase 16 SWR cache**: `snapshot-cache.ts` serve-last-good is the latency mitigation for the push→pull shift.
- **Heartbeat debounce/batch**: `heartbeat-dispatcher.ts` already collapses N events → one heartbeat; the dirty-marker write slots into its enqueue path.

### Established Patterns
- **Plugin-namespace literal `plugin_clarity_pack_cdd6bda4bd`** is required verbatim in migrations (no template substitution) — host validator constraint.
- **Two-source version bump** (`package.json` + `src/manifest.ts`) — host reads `dist/manifest.js`.
- **Node strip-only TS** for worker modules imported by `.mjs` tests — no TS parameter-property shorthand (explicit field declare+assign), as in `op-issue-set.ts` / `heartbeat-dispatcher.ts`.

### Integration Points
- Event subscription loop in `worker.ts:~509` → becomes observe-only enqueue (dirty-marker + scope/provenance short-circuit).
- Bulletin cron + `*/1` situation cron → wrapped by the same opt-in scope gate + governor (D-13).
- New `0017_*` migration → durable provenance table + governor ledger/kill-switch.
- BEAAA reinstall (LOOP-07) → bookended by a recorded DO snapshot per the deploy runbook.
