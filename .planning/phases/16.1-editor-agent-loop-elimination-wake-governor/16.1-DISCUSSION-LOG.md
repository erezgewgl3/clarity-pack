## Loop-break + work queue (LOOP-01/02)

| Option | Description | Selected |
|--------|-------------|----------|
| Op-issues stay queue + provenance table | Heartbeat creates op-issues (pull); native heartbeat picks them up; no requestWakeup/event-reactive creation; durable own_operation_issues table checked at ingress before any host call; events write durable dirty-markers only. Least new surface. | ✓ |
| Dedicated compile_queue table | New durable compile_queue is the single pull-work source; events enqueue intents; op-issues become pure downstream delivery artifacts. More explicit decoupling, more tables. | |

**User's choice:** Op-issues stay queue + provenance table (Recommended)
**Notes:** Reuses the proven native-heartbeat-pull + additive-table patterns already in the codebase; directly satisfies "checked at ingress before any host call." Follow-on: requestWakeup removed entirely (D-05).

## Throughput governor design (LOOP-03)

| Option | Description | Selected |
|--------|-------------|----------|
| Sliding-window ledger + manual-clear | Per-wake rows in a plugin-namespace table; trailing-60s count is the rate; env ceiling CLARITY_WAKE_CEILING_PER_MIN default 6; durable kill-switch stays engaged until operator clears (breaker parity). | ✓ |
| Sliding-window ledger + auto-cooldown | Same ledger/ceiling; kill-switch re-arms after N idle min; self-recovers but loses "human must look" property. | |
| Token-bucket (durable state) | 1-row bucket tokens+refill; smoothest, self-healing; auto-refill = implicit auto-reset, weakest safety posture. | |

**User's choice:** Sliding-window ledger + manual-clear (Recommended)
**Notes:** Governance parity with the failure circuit-breaker's manual Resume. Fail-safe is cheap — a tripped kill-switch only degrades proactive TL;DR freshness; the read path is untouched (LOOP-06).

## Warm-on-heartbeat scope

| Option | Description | Selected |
|--------|-------------|----------|
| Awaiting-you rows, capped, stale-only | Per opted-in company; awaiting-you rows only; cap N=5/heartbeat (env-tunable); only SWR-stale entries. Tightest scope. | ✓ |
| Awaiting-you + open Situation-Room rows | Also warm currently-open room rows (active-viewer); broader warmth, more compile, viewer-state dependency. | |
| Awaiting-you rows, uncapped | Compile every stale awaiting-you row each heartbeat; simpler but weaker bound. | |

**User's choice:** Awaiting-you rows, capped, stale-only (Recommended)
**Notes:** Bounds compile cost at the source; pairs with the governor as a second bound. SWR-stale-only skips fresh cache hits.

## Opt-in / active-company ingress gate (LOOP-04)

| Option | Description | Selected |
|--------|-------------|----------|
| Boot-seeded in-memory set + refresh | Seed opted-in-company set at boot from clarity_user_prefs; refresh on opt-in-change + TTL; ingress short-circuits if companyId not in set; zero host call per event. | ✓ |
| Per-company cached boolean (lazy TTL) | First event per company per TTL does one lookup, then cached; small recurring ingress cost. | |
| Active-viewer signal gate | Gate on open Situation Room/Reader; tighter but misses bulletin cron and couples to UI viewer state. | |

**User's choice:** Boot-seeded in-memory set + refresh (Recommended)
**Notes:** Deterministic, zero per-event cost; set is tiny (single-tenant). Same scope + governor gating also wraps the bulletin and `*/1` situation crons, not just event ingress (D-13).
