## User Constraints (from CONTEXT.md)

### Locked Decisions (D-01..D-13 — verbatim from CONTEXT.md `<decisions>`)

**Dispatch architecture & loop-break (LOOP-01, LOOP-02)**
- **D-01:** Op-issues remain the durable work queue. The agent's own heartbeat/cron picks them up via **native pull** — no `requestWakeup`, no event-reactive op-issue creation.
- **D-02:** Event handlers become **observe-only**: they may write a durable dirty-marker and nothing else. No `ctx.agents.*` wake / `requestWakeup` / dispatch reachable from any `ctx.events.on` handler (statically enforced).
- **D-03:** Recursion broken by a **durable own-operation provenance table** in the plugin namespace (working name `own_operation_issues`, cols ~ `company_id`, `issue_id`, `created_at`, `UNIQUE (company_id, issue_id)` — `reply_resume_dedup` shape). Every op-issue id Clarity creates is recorded.
- **D-04:** Provenance checked at **ingress, before any host call** (replaces the in-memory `OwnOperationIssueSet` as the authoritative guard). The in-memory set MAY remain as a pure fast-path cache but is NEVER the sole guard.
- **D-05:** The fire-and-forget `requestWakeup` call in the delivery path is **removed entirely** — native heartbeat pull is the only dispatch.

**Throughput governor (LOOP-03)**
- **D-06:** Rate limiting is a **sliding-window wake ledger**: append one row per wake attempt (`ts`, `company_id`); count of trailing-60s rows is the rate; excess-over-ceiling suppressed.
- **D-07:** Ceiling from env `CLARITY_WAKE_CEILING_PER_MIN`, default **6/min**.
- **D-08:** Kill-switch is a **durable row** that engages above ceiling and **stays engaged until an operator manually clears it** — governance parity with the failure breaker's manual-Resume. Engaged state persists across restart.
- **D-09:** Provenance + governor ledger/kill-switch ship in a single additive plugin-namespace migration **`0017_*`**, following the `0016` host-validator rules. No `public.*` DDL.

**Warm-on-heartbeat scope**
- **D-10:** Bounded warm compiles **awaiting-you rows only**, per opted-in company.
- **D-11:** Capped at **N=5 rows per heartbeat** (env-tunable); only **SWR-stale** entries warmed.

**Opt-in / active-company ingress gate (LOOP-04)**
- **D-12:** Ingress scope decided against a **boot-seeded in-memory opted-in-company set**, seeded from `clarity_user_prefs` (migration `0003`), refreshed on opt-in-change events + periodic TTL re-read. Zero host/DB call per event.
- **D-13:** Every event handler short-circuits when `event.companyId` not in the set, **before** any reconcile/get/enqueue. Same gate wraps the bulletin and `*/1` situation cron paths.

## (1) Current Code-State Ground Truth

### `src/worker.ts` — the instance-wide subscription (verified at :509-529)

The loop is exactly as CONTEXT.md describes. `worker.ts:509`:

```ts
for (const evt of ['issue.created', 'issue.updated', 'issue.comment.created'] as const) {
  ctx.events.on(evt, async (event) => {
    try {
      heartbeatDispatcher.enqueue({
        entityId: event.entityId,
        entityType: event.entityType,
        companyId: event.companyId,
        actorId: event.actorId,
        actorType: event.actorType,
      });
    } catch (err) { /* logged */ }
  });
}
```

- The handler's ONLY call is `heartbeatDispatcher.enqueue(...)`. There is **no `companyId`/opt-in scope check** — every event for the whole instance is enqueued.
- A second `ctx.events.on('company.created', ...)` at `worker.ts:452` calls `reconcileEditorAgent` directly (a host round-trip in an event handler — also in-scope for LOOP-01/LOOP-04 review, not mentioned in CONTEXT.md; **flag for planner**).
- A third subscription: `registerChatStreamBridge(ctx)` at `worker.ts:536` subscribes to `issue.comment.created` and re-emits onto an SSE channel. It does NOT wake an agent (it `ctx.streams`-publishes). LOOP-01 must confirm the static gate does not false-positive on it, and LOOP-04 must decide whether the chat bridge also needs the company-scope gate (likely yes for cost-parity, but it does no LLM work — **flag for planner**).
- The wake path from ingress is: `enqueue` → debounce flush → `deps.runHeartbeat` → `handleEditorHeartbeat` (`worker.ts:500-505`) → `startAgentTask` → `requestWakeup`. **D-01/D-05 sever this by removing the `requestWakeup` block and making the heartbeat itself never originate from an event handler.**

### `src/worker/agents/circuit-breaker.ts` — failure-only breaker + manual-Resume (verified :100-112, full file)

## (2) Durable-Table / Migration Recipe (D-03, D-06, D-08, D-09)

### Host-validator-safe DDL conventions (verified verbatim from `0016` + `0003` + the validator port in `ddl-prefix-validator.test.mjs`)

Every rule below is enforced by `test/migrations/ddl-prefix-validator.test.mjs` and `test/migrations/no-procedural-blocks.test.mjs` — a violating `0017` fails `pnpm test` before any install:

1. **Fully-qualified namespace literal**, no template substitution: `plugin_clarity_pack_cdd6bda4bd.` on every table. `[VERIFIED: migrations/0016_reply_resume_dedup.sql:47]`
2. **`CREATE TABLE IF NOT EXISTS`** for idempotent re-install. `[VERIFIED: 0016:47]`
3. **Inline `UNIQUE (...)` inside `CREATE TABLE`** — NOT a standalone `CREATE INDEX`. The host's `extractQualifiedRefs` has no pattern for `CREATE INDEX ... ON schema.table`, so a standalone index is rejected with "must use fully qualified schema names." The inline UNIQUE produces the btree index your lookup needs. `[VERIFIED: 0016:55 + ddl-prefix-validator.test.mjs:163-177]`
4. **Apostrophe-free `COMMENT ON` body** (and apostrophe-free `--` comments). The host's greedy `/'([^']|'')*'/g` strip pairs a lone apostrophe across statements and swallows the leading keyword. `[VERIFIED: 0016:24-26, ddl-prefix-validator.test.mjs:106-114]`
5. **Statements must begin with `create` / `alter` / `comment`** only — no procedural blocks, no `DO $$`, no standalone `CREATE INDEX`. `[VERIFIED: ddl-prefix-validator.test.mjs:137, no-procedural-blocks.test.mjs]`
6. **Column type patterns** (from `0016`/`0003`): `id bigserial PRIMARY KEY`, `company_id text NOT NULL`, `<key> text NOT NULL`, `created_at timestamptz NOT NULL DEFAULT now()`, `<flag> boolean NOT NULL DEFAULT false`, `payload jsonb NOT NULL`.
7. **ON CONFLICT idempotency** is applied at the repo (INSERT) layer, not the migration. `0016` documents `ON CONFLICT (company_id, message_uuid) DO NOTHING`.

### Proposed `0017_loop_governor.sql` (three structures — planner to finalize exact names)

```sql
-- own_operation_issues — D-03 durable provenance (replaces the in-memory OwnOperationIssueSet
-- as the authoritative restart-durable loop-break guard).
CREATE TABLE IF NOT EXISTS plugin_clarity_pack_cdd6bda4bd.own_operation_issues (
  id          bigserial PRIMARY KEY,
  company_id  text NOT NULL,
  issue_id    text NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (company_id, issue_id)
);
COMMENT ON TABLE plugin_clarity_pack_cdd6bda4bd.own_operation_issues IS
  'D-03 durable provenance of Clarity-authored operation issues. Checked at event ingress before any host call so a Clarity write can never re-trigger Clarity compilation. Survives worker restart (the empty-on-restart in-memory set was the incident onset condition). Additive plugin-namespace table -- disable preserves data.';

-- wake_ledger — D-06 sliding-window throughput ledger. One row per wake attempt;
-- count of trailing-60s rows is the rate. Self-drains via repo-side prune.
CREATE TABLE IF NOT EXISTS plugin_clarity_pack_cdd6bda4bd.wake_ledger (
  id          bigserial PRIMARY KEY,
  company_id  text NOT NULL,
  woke_at     timestamptz NOT NULL DEFAULT now()
);
COMMENT ON TABLE plugin_clarity_pack_cdd6bda4bd.wake_ledger IS
  'D-06 sliding-window wake-rate ledger. One row per wake attempt; the count of rows in the trailing 60 seconds is the current wake rate measured against CLARITY_WAKE_CEILING_PER_MIN. Repo prunes rows older than the window. Additive plugin-namespace table.';

-- wake_kill_switch — D-08 durable kill-switch. Engages above the ceiling, stays
-- engaged until an operator manually clears (governance parity with the failure breaker).
CREATE TABLE IF NOT EXISTS plugin_clarity_pack_cdd6bda4bd.wake_kill_switch (
  id             bigserial PRIMARY KEY,
  company_id     text NOT NULL,
  engaged        boolean NOT NULL DEFAULT false,
  engaged_at     timestamptz,
  reason         text,
  plugin_version text,
  UNIQUE (company_id)
);
COMMENT ON TABLE plugin_clarity_pack_cdd6bda4bd.wake_kill_switch IS
  'D-08 durable wake kill-switch. Engages when the trailing-60s wake rate exceeds the ceiling and stays engaged until an operator clears it (no auto-clear in worker code -- governance parity with the failure breakers manual Resume). Engaged state survives worker restart. A tripped switch degrades only proactive TL;DR freshness; the read-time path is untouched.';
```

Notes for the planner:
- **No `CREATE INDEX`.** The `UNIQUE (company_id, issue_id)` on `own_operation_issues` covers the ingress lookup `WHERE company_id=$1 AND issue_id=$2`. `wake_ledger`'s trailing-window count (`WHERE company_id=$1 AND woke_at > now() - interval '60 seconds'`) is small (ceiling is ~6/min) so a seq scan on a self-pruned table is fine; a btree on `woke_at` would need a `CREATE INDEX` (rejected) — instead prune aggressively in the repo so the table stays tiny.
- **Kill-switch `UNIQUE (company_id)`** enables `INSERT ... ON CONFLICT (company_id) DO UPDATE SET engaged=true, engaged_at=now(), reason=$2` for an atomic engage.
- Apostrophes removed from all comment bodies above (e.g. "breakers manual Resume", "switch survives") per rule 4.

### Durable-table repo API pattern (verified from `reply-resume-repo.ts`)

Mirror `reply-resume-repo.ts` exactly. The host `ctx.db` contract (verified in that file's header + `snapshot-cache.ts`):
- **`ctx.db.query<T>(sql, params)`** — SELECT-only, single statement, returns rows. Used for provenance reads, trailing-window counts, kill-switch reads.
- **`ctx.db.execute(sql, params)`** — namespace-local DML (INSERT/UPDATE/DELETE), **returns rowCount only, NEVER rows** (no `RETURNING`). Used for provenance writes, ledger appends, ledger prune, kill-switch engage.

ON CONFLICT (company_id, issue_id) DO NOTHING`, called from `agent-task-delivery.ts:419`/`:441`).

### Ledger pruning (Discretion)
`wake_ledger` self-drains: on each `appendWake`, also call `pruneOldWakes` (delete `WHERE woke_at < now() - interval '<window+slack> seconds'`). ## (3) Storm-Safety CI Test Harness Recipe (LOOP-05)

### Runner + mocking style (verified)
- **Runner:** `node --test "test/**/*.test.mjs"` (from `package.json:15`). Tests import `.ts` source directly under Node's type-stripping loader — the Node strip-only TS constraint exists precisely so these imports work.
- **Mocking style:** plain hand-rolled fake ctx objects (NOT a mock library). See `editor-heartbeat-recursion.test.mjs:48-88` (`makeCtx`) and `self-loop-filter.test.mjs`. The fake ctx provides `logger`, `db` (`query` returns `[]`, `execute` returns `{rowCount:0}`), `issues` (`get`/`list`/`create`/`listComments`/`requestWakeup`), and `agents`. Calls are captured into arrays (`createCalls.push(input)`) and asserted with `node:assert/strict`.
- **The existing recursion test already proves the pattern**: it asserts `createCalls.length === 0` to prove no op-issue was spawned. The storm test extends this to wake-count + restart.

### Concrete storm-harness sketch

```js
import { strict as assert } from 'node:assert';
import test from 'node:test';
// import the converted observe-only ingress entrypoint + the governor + the
// in-memory provenance fast-path. The durable provenance is mocked via the
// fake db below (so the test can clear it mid-burst to simulate restart).

function makeStormCtx() {
  const wakeCalls = [];          // every agent wake / requestWakeup the system attempts
  const opIssueCreates = [];     // every plugin op-issue create
  const provenance = new Set();  // simulates the durable own_operation_issues table
  const wakeLedger = [];         // simulates wake_ledger rows (ts per wake)
  let killSwitchEngaged = false;

  const ctx = {
    logger: { info() {}, warn() {}, error() {} },
    db: {
      namespace: 'plugin_clarity_pack_cdd6bda4bd',
      async query(sql, params) {
        if (/own_operation_issues/.test(sql)) {
          // ingress provenance check (company_id, issue_id)
          return provenance.has(params[1]) ? [{ exists: 1 }] : [];
        }
        if (/count.*wake_ledger/i.test(sql)) {
          const cutoff = Date.now() - 60_000;
          return [{ n: wakeLedger.filter((t) => t > cutoff).length }];
        }
        if (/wake_kill_switch/.test(sql)) return [{ engaged: killSwitchEngaged }];
        return [];
      },
      async execute(sql, params) {
        if (/insert into.*own_operation_issues/i.test(sql)) provenance.add(params[1]);
        if (/insert into.*wake_ledger/i.test(sql)) wakeLedger.push(Date.now());
        if (/wake_kill_switch.*engaged\s*=\s*true/i.test(sql)) killSwitchEngaged = true;
        return { rowCount: 1 };
      },
    },
    issues: {
      async create(input) { opIssueCreates.push(input); return { id: `op-${opIssueCreates.length}` }; },
      async requestWakeup(id) { wakeCalls.push(id); return { queued: true }; },
      async get() { return null; },
      async list() { return []; },
      async listComments() { return []; },
    },
    agents: {
      managed: { async reconcile() { return { agentId: 'editor-uuid' }; } },
      async pause() {}, async resume() {},
    },
  };
  return { ctx, wakeCalls, opIssueCreates, provenance, wakeLedger, get killSwitchEngaged() { return killSwitchEngaged; } };
}

test('LOOP-05 storm: 100+ events incl. own op-issues + agent result writes, restart mid-burst, wakes bounded + zero recursion', async () => {
  const h = makeStormCtx();
  // 1. Feed 100 mixed events: real issues, the plugin's OWN op-issue creates
  //    (companyId in provenance), and agent result writes (actorType:'agent').
  for (let i = 0; i < 60; i++) await ingest(h.ctx, { entityId: `BEAAA-${i}`, companyId: 'c1', actorType: 'user' });
  for (let i = 0; i < 30; i++) await ingest(h.ctx, { entityId: `op-${i}`, companyId: 'c1', actorType: 'plugin' }); // own ops (in provenance)
  // 2. Simulate restart: clear the in-memory fast-path set mid-burst.
  __clearInMemoryProvenanceFastPath();        // durable Set in fake db survives — that is the point
  for (let i = 0; i < 30; i++) await ingest(h.ctx, { entityId: `op-${i}`, companyId: 'c1', actorType: 'agent' }); // agent result writes re-entering
  // 3a. Zero synchronous wakes emitted FROM ingress (LOOP-01).
  assert.equal(h.wakeCalls.length, 0, 'event ingress emitted zero agent wakes');
  // 3b. Zero self-trigger recursion: no op-issue create was triggered by an own op-issue event (LOOP-02).
  //     (every event whose entityId is in provenance must have produced no downstream create)
  // 3c. Now drive the pull-based heartbeat through the governor; assert total wakes <= ceiling/min (LOOP-03).
  for (let i = 0; i < 20; i++) await heartbeatPullAndMaybeWake(h.ctx, 'c1'); // tries to wake 20x in <60s
  assert.ok(h.wakeCalls.length <= 6, `wakes capped at ceiling (got ${h.wakeCalls.length})`);
  assert.equal(h.killSwitchEngaged, true, 'kill-switch engaged after exceeding ceiling');
});
```

Key assertions the planner must lock into the test:
- `wakeCalls.length === 0` after pure ingress (LOOP-01 — the decisive observe-only proof).
- Restart simulated by clearing the **in-memory** fast-path while the **durable** Set (fake `db`) persists — proving D-04's restart backstop (LOOP-02).
- Re-entrant own op-issue / agent-result events produce zero new op-issue creates (zero recursion).
- Driving > ceiling wakes/min through the pull path caps actual wakes ≤ 6 and engages the durable kill-switch (LOOP-03).
- The test must **fail if `requestWakeup` is re-introduced into an event handler** — pair it with the §4 static gate so re-introduction fails at the source level too.

## (6) Landmines & Pitfalls

- **L-1 — SDK version drift in docs.** CLAUDE.md repeats `@paperclipai/plugin-sdk@2026.512.0` throughout, but `package.json` pins `2026.525.0` (peer + dev) and worker code reasons about 525.0-specific PR #6547 invocation-scope behavior. **Plan and verify against 525.0.** `[VERIFIED: package.json:32,37 + node_modules/.../types.d.ts present]`

- **L-2 — `clarity_user_prefs` has no company column.** D-12's "opted-in-company set" cannot be built from `clarity_user_prefs` directly — it's `user_id`-keyed only (`opt-in-guard.ts:71`). To build a per-**company** opted-in set the worker must map opted-in users → their companies (via a host call or an existing user→company resolver) OR the planner redefines the gate as "any opted-in user exists" + per-event company membership via that mapping. **This is the single biggest unspecified detail in D-12** — the planner must resolve the user→company mapping at boot-seed time. Candidate: the boot seed runs `SELECT user_id FROM clarity_user_prefs WHERE opted_in_at IS NOT NULL` then resolves each user's company set. Note boot-time host calls are dead on 525.0 (PR #6547) — so a pure-SQL seed (if a user→company table is reachable in the plugin namespace) or a lazy first-event seed inside valid invocation scope is required. **Flag prominently for planner.**

- **L-3 — Boot-time host calls are rejected on 525.0.** `worker.ts:429-448` documents that `ctx.companies.list()` and any worker→host call at boot fails with "missing, expired, or unknown invocation scope" (PR #6547). The opted-in-company set's **seed** therefore cannot make host calls at boot — it must seed lazily inside the first valid event/job invocation, or read only from the plugin namespace (SQL `ctx.db.query` is invocation-scoped too — verify it works at boot; the breaker's `isCircuitOpenDurable` runs inside handler scope, not boot). Safest: seed the set on the first event of each invocation and refresh on a TTL, exactly as D-12 hints ("refreshed on opt-in-change events plus a periodic TTL re-read").

- **L-4 — `requestWakeup` removal must not break the type/tests.** `AgentTaskDeliveryCtx` (`agent-task-delivery.ts:170`) lists `requestWakeup` in its `Pick<>`. Removing the call (D-05) lets you drop it from the type, but `editor-heartbeat-recursion.test.mjs:82` and other fakes provide a `requestWakeup` stub — removing the type member won't break the fakes (extra members are fine) but check no test asserts it WAS called.

- **L-5 — `worker.ts:452` `company.created` handler and `worker.ts:536` chat-stream bridge are also `ctx.events.on` subscriptions.** The `company.created` handler calls `reconcileEditorAgent` (a host round-trip — arguably a LOOP-04 ingress violation; reconcile is not a wake but it IS unscoped host work). The static gate (§4) and the opt-in scope gate (D-13) must explicitly decide their disposition — likely: company.created reconcile is fine (rare, idempotent) but should still be company-scoped; chat bridge should gain the opt-in scope short-circuit for cost parity. **Flag for planner.**

- **L-6 — `0017` is the next number despite `0012` being absent on disk.** Migrations on disk: `0001`-`0011`, `0013`-`0016` (no `0012`). The host runs files in lexical order, so the gap is harmless; the new file is `0017_*` (highest+1). Do NOT reuse `0012`.

The host reads `dist/manifest.js` (built from `src/manifest.ts`), and `circuit-breaker.ts:38` reads `manifest.version` for durable-row version-scoping — a mismatch silently version-scopes failures/kill-switch wrong. `[VERIFIED: package.json:3, src/manifest.ts:626, MEMORY plugin-version-bump-two-sources]`
