# Phase 16.1: Editor-Agent loop elimination & wake governor — Specification

**Created:** 2026-06-04
**Ambiguity score:** 0.19 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

Make Clarity Pack safe to run on the live Paperclip instance: installing Clarity and then creating or changing any issue, agent, or task MUST NOT trigger a self-sustaining storm of agent wakeups — eliminate the event-amplification feedback loop by construction, bound agent wakeups by a durable throughput governor, and gate event ingress on opt-in/scope, while leaving the read-time "zero rabbit-holes" guarantee fully intact.

## Background

**The incident (2026-06-04):** Clarity Pack was uninstalled because it made the live BEAAA instance unusable — every issue/agent/task change drove host CPU up and woke 4–5 agents.

**Root cause (three independent code-review passes converged):** a closed positive-feedback loop on the host event bus.
- The worker subscribes to instance-wide `issue.created` / `issue.updated` / `issue.comment.created` events with no opt-in/company/board scope (`src/worker.ts:509`).
- It reacts by creating an operation-issue (`agent-task-delivery.ts:419`) AND actively waking the Editor-Agent via `requestWakeup` (`agent-task-delivery.ts:451`).
- The Editor-Agent's own result writes (a comment + status→done, `actorType:'agent'`) re-enter the same subscription → loop. Measured baseline: **~3.8 self-triggered heartbeats/sec ≈ 228/min** (`heartbeat-dispatcher.ts:15`).

**Why the existing guards failed at the edges:**
1. The fast-path recursion guard is **in-memory and empty on worker restart** — i.e. exactly at install/upgrade, the incident's onset condition (`op-issue-set.ts:19-23`).
2. The self-loop **tag** filters are inert: host `PluginEvent`s carry no `tags` field, so only `author_id` matching is live (`self-loop-filter.ts` vs `heartbeat-dispatcher.ts:56-62`).
3. The "circuit breaker" gates on consecutive **failures**, not throughput, so a fully-successful storm never trips it (`circuit-breaker.ts:100-112`).
4. Opt-in gates only UI RPC handlers, NOT event ingress or the `*/1` bulletin cron (`opt-in-guard.ts`) — "default OFF for existing users" does not throttle the worker.

The Clarity worker process itself shows ~0% CPU; the cost is downstream — each wake spins an LLM agent. **Clarity is the trigger; the employee agents are the fuel.** This storm class recurred across v0.6.5, v1.4.1/BEAAA-2092, the bulletin 2-min runaway, and v1.4.4 despite repeated filter hardening — so the strategy shifts from *mitigate-with-filters* to *eliminate-the-loop-by-construction*.

**Value-impact analysis (locked boundary — why this fix does not erode the product):**
The "zero rabbit-holes" core value is a **read-time** promise delivered by synchronous, local work that never enters the event loop:

| Value pillar | Delivered by | Touched by this fix? |
|---|---|---|
| Inline reference resolution | `resolve-refs` at Reader open | No |
| Blocker chain flattened to one human action | Situation Room snapshot at read time | No |
| Deliverable previewed in place | `deliverable-preview` at read time | No |
| Plain-English TL;DR | pre-compiled Editor-Agent LLM output, cached | **Yes — the only one** |

The storm lives entirely in the **proactive compilation** machinery (TL;DR / bulletin / action-cards). The principle this phase locks: **proactive compilation must be PULL-based + SCOPED (only what the user will actually read — their awaiting-you/Situation-Room rows, the issue they open, today's bulletin) and SWR-served (Phase 16), never PUSH-based + instance-wide.** Scheduled proactivity is preserved (daily bulletin cron + a *bounded* warm-on-heartbeat for awaiting-you rows). User-visible cost: an occasional first-open "compiling…" on a never-seen TL;DR, mitigated to near-zero by the Phase-16 SWR serve-last-good cache; AI prose may be minutes-stale rather than instantaneously refreshed (arguably more honest — timestamped). What the user gains is the dominant term: a usable instance, the precondition for any of the value.

## Requirements

1. **LOOP-01 — Observe-only ingress (zero wakeups from event handlers)**: No host-event handler performs an agent wakeup or dispatch.
   - Current: `issue.*`/`comment.*` handlers reach `startAgentTask` → `requestWakeup` (`agent-task-delivery.ts:451`) inside the event-driven heartbeat flush.
   - Target: every agent run originates from the agent's own heartbeat/cron pulling a durable work queue; event handlers only observe (enqueue/record), never wake.
   - Acceptance: a static check proves no `ctx.agents.*` wake / `requestWakeup` / dispatch call is reachable from any `ctx.events.on` handler; a unit test feeding issue/comment events asserts zero wake calls emitted synchronously from ingress.

2. **LOOP-02 — Loop broken by construction with restart-durable provenance**: A Clarity-authored "operation" can never re-enter and re-trigger Clarity's own event-reactive compilation, and the suppression survives a worker restart. *(Mechanism — off-bus queue vs. durable op-issue provenance — is intentionally deferred to discuss-phase; this requirement locks the outcome only.)*
   - Current: own-operation suppression depends on an in-memory set that is empty after restart (`op-issue-set.ts:19-23`) and on inert tag filters (`self-loop-filter.ts`).
   - Target: own-operation provenance is durable (survives restart) and is checked at ingress before any host call; no Clarity write can cause Clarity to wake an agent.
   - Acceptance: the storm-safety test (LOOP-05) shows zero self-trigger recursion even when the in-memory state is cleared mid-burst (simulated restart); a Clarity-authored op write injected as an event produces zero downstream wakes.

3. **LOOP-03 — Durable, env-configurable throughput wake-governor + kill-switch**: Agent wakeups are bounded by a global rate ceiling that trips on volume (not just failures) and whose state survives restart.
   - Current: the only governor is a consecutive-failure circuit breaker (`circuit-breaker.ts:100`); a successful storm is unbounded; no rate limiter exists.
   - Target: a global wake-rate governor caps wakes at an env-configurable ceiling (safe default ~6/min) with a durable kill-switch that engages above the ceiling and persists across restart; tripping is observable in logs.
   - Acceptance: feeding > ceiling wake requests/min causes excess wakes to be suppressed and the kill-switch to engage; the engaged state persists across a simulated restart; the ceiling is read from an env var with a documented safe default.

4. **LOOP-04 — Opt-in / active-company scope gate at ingress**: Event ingress does no host call or work for events outside opted-in / active-company scope.
   - Current: event subscriptions and the bulletin cron fire unconditionally regardless of opt-in (`opt-in-guard.ts` wraps only RPC handlers).
   - Target: each event handler short-circuits (before any `reconcile`/`get`/enqueue) when no user has opted in for the relevant company / it is out of active scope.
   - Acceptance: with opt-in OFF, a burst of issue/comment events produces zero host calls and zero wakes (asserted by a test with a spy on host calls).

5. **LOOP-05 — Falsifiable storm-safety test (CI)**: An automated test reproduces the incident conditions and proves boundedness.
   - Current: no test exercises the self-write re-entry / restart / throughput path; the storms were only ever observed live.
   - Target: a CI test feeds a burst (≥100 issue/comment events including the plugin's own op-issue creates + agent result writes), clears in-memory state mid-burst (simulated restart), and asserts (a) total agent-wakes ≤ the governor ceiling per minute and (b) zero self-trigger recursion.
   - Acceptance: the test exists, runs in CI, fails if `requestWakeup` is re-introduced into an event handler or if wakes exceed the ceiling, and passes on the corrected build.

6. **LOOP-06 — Read-time "zero rabbit-holes" guarantee regression-proofed**: The synchronous read-path resolution is untouched and still works.
   - Current: `resolve-refs`, blocker-chain flatten, and `deliverable-preview` run at read time, independent of the event loop.
   - Target: these code paths are unchanged by the loop fix; their behavior is unaffected.
   - Acceptance: existing resolve-refs / blocker-chain / deliverable-preview tests stay green; a diff review confirms no change to their logic; opening a Reader and the Situation Room still resolves references inline and flattens the blocker chain.

7. **LOOP-07 — Live make-safe verification on BEAAA**: The corrected build is reinstalled on BEAAA and a live drill proves no storm.
   - Current: Clarity is uninstalled from BEAAA because the installed build stormed.
   - Target: the new build is reinstalled (bookended by a verified DO snapshot); creating/changing an issue and dispatching an agent task produces no CPU storm and no multi-agent wake cascade.
   - Acceptance: after reinstall, a live drill shows the Clarity worker stays near-idle (≈0% CPU), agent wakes stay within the governor ceiling (observed in logs), no 4–5-agent spawn cascade follows a single change, and the Situation Room loads with no 502; bookend DO snapshot id recorded.

## Boundaries

**In scope:**
- Convert host-event ingress to observe-only (no agent wakeups from event handlers).
- Pull/queue-based compile dispatch driven by the agent's own heartbeat/cron (durable work queue).
- Restart-durable own-operation provenance that breaks the loop by construction (mechanism chosen in discuss-phase).
- A durable, env-configurable, throughput-based global wake-rate governor + kill-switch.
- An opt-in / active-company scope gate at the event-subscription layer (before any host call).
- An automated CI storm-safety test (burst + simulated restart + throughput assertion).
- A read-time regression guard proving resolve-refs / blocker-flatten / deliverable-preview are unaffected.
- Version bump (both `package.json` + `src/manifest.ts`), rebuild, and a LIVE reinstall on BEAAA bookended by a DO snapshot, with a no-storm drill.
- Keeping the daily bulletin cron and a *bounded* warm-on-heartbeat for awaiting-you rows working under the new pull model.

**Out of scope:**
- Re-enabling `ACTION_CARDS_ENABLED` / the action-cards async re-architecture — remains **Phase 19** (action-cards stays gated OFF here; only its loop-safety is ensured).
- Any new user-facing surfaces or features — the Phase 17–20 feature work is unaffected and deferred.
- Changing the read-time resolution logic (resolve-refs / blocker-flatten / deliverable-preview) — untouched by design (that's the value we're protecting).
- Redesigning the daily bulletin cadence — kept as-is; only its loop-safety is in scope.
- Tuning the exact governor number beyond shipping a safe env-configurable default — operators can tune via env.
- Choosing the loop-break mechanism (off-bus queue vs. durable op-issue provenance) — deferred to discuss-phase (LOOP-02 locks the outcome, not the how).

## Constraints

- **Coexistence / additive-only**: any durable state needed (provenance, kill-switch, wake-rate ledger) must live in the plugin namespace via an additive migration; disable/uninstall must preserve data (coexistence guarantees #3/#6). No `public.*` DDL.
- **Reuse Phase 16 SWR**: the shift from push-reactive to pull/lazy compilation must serve reads via the existing serve-last-good snapshot/TLDR cache so the user-visible latency cost is near-zero.
- **Instance-agnostic**: no company-prefix literals; the fix works on any instance (BEAAA values substituted at deploy).
- **Bookended-by-snapshots rule**: the LIVE reinstall (LOOP-07) MUST be bookended by a verified DO snapshot with a rehearsed rollback (uninstall→reinstall-prior), per CLAUDE.md.
- **Two-source version bump**: bump version in both `package.json` and `src/manifest.ts` (host reads `dist/manifest.js`).
- **Governor default**: ship a conservative env-configurable default (~6/min) — must not require config to be safe out of the box.
