## GAP (LOOP-07 not fully met) — Editor-Agent write dispatch broken by the loop fix

**Symptom:** Reader TL;DRs show the agent's status text ("cheap/recovery profile —
harness blocks document writes, resumeRequiresNormalModel: true") instead of a
summary. The Editor-Agent computes CORRECT plain-English TL;DRs (verified: e.g.
"The PII-scrubbing scanner activation is fully approved — you do not need to act")
but can never persist them. The whole plain-English layer (TL;DRs + narratives) is
non-functional live → surfaces fall back to raw task numbers.

**Root cause (hard-verified, host source + live log):**
- Paperclip's normal, write-capable dispatch path is `issue-assignment-wakeup`,
  triggered by `requestWakeup`. 16.1 Plan 16.1-02 (D-05) DELETED the `requestWakeup`
  call from `agent-task-delivery.ts` (the storm source) and assumed the Editor-Agent's
  native heartbeat would pull op-issues.
- It does not. Op-issue assignments sit undispatched and fall to Paperclip's
  **periodic recovery sweep**, which dispatches them via
  `recoveryAssigneeAdapterOverrides("status_only")` → `modelProfile:cheap`,
  `allowDocumentUpdates:false`, `resumeRequiresNormalModel:true`.
- `routes/issues.js` (`assertDeliverableMutationAllowedByRunContext` /
  `isStatusOnlyCheapRecoveryContext`) then **hard-rejects** any document/deliverable
  write from such runs (403/422). → write-block.
- Evidence: 8 `assignmentDispatched:1, continuationRequeued:0` recovery events across
  04:52–10:43, each a different op-issue UUID; latest = the f584657c TL;DR compile
  op-issue `e2d549b7` at 10:43:48. TL;DRs wrote fine on 2026-06-04 under the old build
  that still had `requestWakeup` (`result DOCUMENT received`). Nothing dispatched
  normally since the 1.5.0 reinstall — only recovery (status_only).
- NOT the model (gpt-5.4/mini both fine), NOT the cheap-model toggle (flipping it off
  did not stop the block — 53 blocks after), NOT clarity's compile logic.

## Fix direction (for gap-closure planning)

Re-introduce a **single GOVERNED `requestWakeup`** at op-issue CREATION in
`agent-task-delivery.ts` (right after `recordOwnOperationIssue`, ctx.issues.create) —
**not** from event ingress. Gate it through `checkAndRecordWake` (the wake-governor
16.1 already shipped).

- Restores prompt **normal_model** (write-capable) dispatch → TL;DRs persist.
- Storm-safe by construction: the storm mechanism (event-ingress → wake → re-entry)
  stays severed — ingress is observe-only + provenance-gated (LOOP-01/02), so the
  agent's resulting writes re-enter ingress and are dropped at the provenance gate.
  The wake now fires only at deliberate, bounded op-issue creation (bounded warm
  ≤5/company + on-demand), governed by the trailing-60s ceiling + kill-switch.
- Degrade-safe: if the kill-switch is engaged (wake suppressed), still create the
  op-issue — the recovery sweep eventually runs it (status_only), i.e. no worse than today.

**Tests:** existing `test/loop/storm-safety.test.mjs` must still pass (burst exercises
the governor). Add: op-issue creation triggers exactly one governed wake (not zero,
not unbounded); kill-switch-engaged path still creates the op-issue without a wake.

**16.1 design note:** D-05 over-corrected — it deleted the wake entirely instead of
governing it. The governor (16.1-02) was built precisely to make a bounded wake safe;
the gap plan uses it.
