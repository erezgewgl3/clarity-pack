The "third db.query" in snapshot-prefetch (D-05) is NOT a public.* read and NOT the
waitMap prefetch. It is the Phase-19 action-cards-flag namespace read:

  SELECT enabled FROM plugin_clarity_pack_cdd6bda4bd.action_cards_flag WHERE company_id = $1 LIMIT 1

from src/worker/db/action-cards-flag-repo.ts (isActionCardsEnabled). It is called at
TWO mutually-exclusive sites in src/worker/handlers/situation-room.ts (line 603 = sync
recompute path; line 709 = SWR serve-last-good path) — exactly ONE fires per snapshot, so
in the test's cache-miss scenario it is a SINGLE read, NOT an N+1. It is the SAME posture
as clarity_user_prefs / situation_snapshots / clarity_human_waits, all of which the test
already excludes from `dbQuerySql` via dedicated handling.

The current makeCtx db.query (snapshot-prefetch.test.mjs ~line 99-124) already special-cases
clarity_user_prefs, situation_snapshots, and clarity_human_waits BEFORE pushing to
spies.dbQuerySql. The action_cards_flag read currently falls through into dbQuerySql, making
the count 3 instead of 2.

U7 watchdog mechanism (chat-messages.test.mjs ~line 451-487 + src/worker/handlers/chat-messages.ts):
  - line 216: `void ensureTopicWakeable(...)` — the watchdog get is fire-and-forget (NOT awaited).
  - line 249: `await ctx.issues.get(...)` — the stuck-read get IS awaited.
The flake is the `assert.ok(elapsed < 85)` wall-clock threshold (line 480). Under CI load the
single awaited 50ms stuck-read get + listComments can cross 85ms even when the watchdog get is
correctly not awaited → false failure.

The makeCtx fake's `ctx.issues.get` (chat-messages.test.mjs ~line 73-83) uses `getDelayMs` via
`await new Promise(r => setTimeout(r, getDelayMs))`. The condition-based redesign replaces the
delay+threshold with a controllable BARRIER (a deferred promise the watchdog get blocks on) that
the test releases AFTER the response resolves — proving the response did not await the watchdog
get. The U6/U6b tests (~line 412-449) already use the `await new Promise(r => setImmediate(r))`
tick-drain pattern as the in-suite analog for fire-and-forget settling.

safety-CLI deps (D-06): `tar` and `@electric-sql/pglite` are NOT declared anywhere and absent
from node_modules. scripts/safety/lib/snapshot.mjs line 24 does a TOP-LEVEL `import * as tar from
'tar'`, so ANY test importing the lib (gate, restore, restore-tar-cve, snapshot, snapshot-pglite,
snapshot-postgres-mock transitively, verify) fails at module-load with "Cannot find package 'tar'".
The tests are designed with injection points (`_spawn`, `_pglite`, `startStubServer`, `_paperclipCli`)
so they NEVER spawn real pg_dump or hit a real DB — snapshot-postgres-mock uses a stub spawn, verify
uses an in-process stub HTTP server. Therefore providing `tar` (+ `@electric-sql/pglite` for the
pglite dynamic import path, snapshot.mjs line 261) should make them genuinely pass.
