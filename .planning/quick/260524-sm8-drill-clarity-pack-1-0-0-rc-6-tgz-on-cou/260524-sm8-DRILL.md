## Step 3 — Confirm `paperclip_restoring` DB exists (idempotent)

The verify step in this runbook (Step 6) restores into a staging DB literally named `paperclip_restoring`. The runtime role lacks CREATEDB, so the DB must pre-exist. This is a one-time setup per box and is likely already present from prior drills; the block is idempotent and safe to re-run (per `countermoves-verify-install-drill` drawer #3).

```bash
export DB_URL="$(sudo grep DATABASE_URL /etc/paperclip/db.env | cut -d= -f2-)"
unset PAPERCLIP_API_URL           # gotcha #1 (paperclip-auth-pattern drawer): exporting PAPERCLIP_API_URL breaks auth.json apiBase keying — paperclipai child procs 403 on plugin list
export PAPERCLIP_API_KEY="$(jq -r '.. | objects | select(.token? // empty | type == "string") | .token' ~/.paperclip/auth.json | head -1)"
export PAPERCLIP_COMPANY_ID=COU
DB_USER=$(printf '%s' "$DB_URL" | sed -E 's#^[a-z]+://([^:@/]+).*#\1#')
sudo -u postgres psql -tAc "SELECT 1 FROM pg_database WHERE datname='paperclip_restoring';" | grep -q 1 \
  || sudo -u postgres psql -c "CREATE DATABASE paperclip_restoring OWNER \"$DB_USER\";"
echo "paperclip_restoring: $(sudo -u postgres psql -tAc "SELECT datname FROM pg_database WHERE datname='paperclip_restoring';")"
```

Expected output (final echo):
```
paperclip_restoring: paperclip_restoring
```

If the SELECT returns empty even after the CREATE, the DB_USER derivation from `$DB_URL` is wrong — inspect with `echo "$DB_URL"` and re-derive `DB_USER` by hand.
